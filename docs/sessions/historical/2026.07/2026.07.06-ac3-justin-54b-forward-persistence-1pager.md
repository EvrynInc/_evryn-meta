## 1. The problem (verified against `main`)

- **Forwarded emails are stored ONLY in `emailmgr_items`, which has a 6-month TTL.** `processForward` (`evryn-backend/src/email/process.ts`) creates an `emailmgr_items` row and reifies the original sender ("Eva") as a `users` record — but it writes **no `messages` row** for the forwarded content. (Only `processDirect`, for direct messages, writes to `messages`.)
- **So once an `emailmgr_items` row ages out, the contact's original email is gone from our DB** (it still lives in Mark's Gmail). That **breaks the v0.3 reach-back plan** — going back to contact everyone Mark forwarded, while they still remember emailing him (BUILD "Phase I backlog").
- **54a rides on this:** Evryn's "have I dealt with this person before?" triage check keys off `emailmgr_items.original_from_user_id` (ADR-036) — the table that self-deletes. Past 6 months she'd look someone up and find nothing. The fix points her at a *permanent* store — but that store has to exist first, which is 54b.

## 2. The premise — the reasoning error to avoid (verified)

The instinct is "just copy every forward into `messages`." The brief warns against that, because Evryn loads a user's recent `messages` whenever she works with them — and copying ~200 forwards/day into `messages` sounds like it would **flood Mark's history**.

**I verified this is NOT actually a pollution risk.** `getRecentMessages(userId)` matches `sender_id = userId OR recipient_id = userId` (`evryn-backend/src/db/messages.ts`). A forward recorded as a **contact↔Evryn** message has the **contact** as sender and **Evryn** as recipient — **Mark is neither**, so it never enters *Mark's* window. **So the gap is PURELY "preserve the contact's email permanently," NOT "fix a pollution problem."** (Designing it as a pollution fix would be solving a problem that doesn't exist.)

## 3. Your steer (captured — and it's right)

Record each forward as a **contact↔Evryn message, tagged "intended-for-Mark"** in metadata. Preserves the contact's email for the v0.3 reach-back; doesn't touch Mark's history.

## 4. The design (recommendation)

On `processForward`, in addition to the existing `emailmgr_items` row, write a `messages` row:

- `sender_id` = the reified contact (`original_from_user_id`, "Eva")
- `recipient_id` = `EVRYN_USER_ID`
- `source` = `"email"`, `content_raw` = the forwarded body
- `metadata` = `{ origin: "forward", intended_for: <gatekeeper user_id>, emailmgr_item_id: <id> }`

What this buys:
- **Permanent** — survives the 6-month `emailmgr_items` TTL. The contact's email is preserved for v0.3 reach-back, queryable by sender-as-user.
- **No pollution of Mark's history** — Mark is neither sender nor recipient, so it never enters his `getRecentMessages` window (§2).
- **Correctly IN the contact's own history** — the contact *is* the sender, so when Evryn later works *with that contact* (v0.3 reach-back, or the near-term "why doesn't Mark want to talk to me?" reply via `processDirect`), she has the original email as real context. That's desirable, not bloat.
- **Unblocks 54a** — triage.md's "dealt with them before?" check can then point at the permanent `messages` (sender = this contact) + the pass-stamp on the contact's record, instead of the TTL'd `emailmgr_items`.

## 5. Decisions (ratified by Justin, 2026-07-06)

**5.1 — Write model: (a) eager write at forward-time.** Record the contact↔Evryn message the moment the forward arrives — so the never-conversed contacts the v0.3 reach-back targets are all preserved (lazy-backfill would miss exactly them). Cost ~200 rows/day: trivial storage, and it's the contact's-own-history, not bloat.

**5.2 — Scope: persist for every real person; skip only `ignore` (Justin's sharpening).** The exclusion isn't the raw null-FK case (a technical extraction hiccup) — it's the *judgment* that there's no human. The clean rule:
- **Persist** the contact↔Evryn message for every forward where a real person gets a user record — **gold, edge, pass, AND bad_actor.** (Deliberately **NOT** pass-only: the pass-*stamp* is the pass-specific deterministic record; *this* persistence preserves the *email itself* for anyone real, so the v0.3 reach-back can reach them.)
- **Skip** only when Evryn classifies **`ignore`** — no discernible human → no user record → nothing to preserve. (This is exactly Justin's intuition: "no discernible human → it's an `ignore`.")
- **Anchor the write to "a real contact record exists,"** satisfied EITHER by the forward-time reification (`original_from_user_id` set) OR by Evryn's triage `create_user`. This catches the edge where the FK is null for a *technical* reason (a forward header that didn't auto-parse) but there IS a real human Evryn identifies during triage — so a real person is never dropped over a parse hiccup. It also means the write can happen at forward-time (runtime defense-in-depth) for the common reified case, with the triage path as the backstop.

**5.3 — Build now: (a) this hardening wave.** Small, unblocks 54a, near-term. Owner: AC3 design (this doc) → DC+QC build.

*One thing to note for the v0.3 legal work (not a v0.2 decision):* a gatekeeper forwarding a contact's email is the gatekeeper *consenting* to share it with Evryn — a different posture from Evryn reading two users' private messages to each other. That distinction, plus "does the time delay change the posture of a v0.3 reach-back 'reply'?", is flagged in BUILD's Phase-I backlog — worth carrying to the v0.3 reach-back design, not a blocker here.

## 6. Lifecycle refinement — the message converts contact↔Evryn → contact↔Mark **at MATCH** (Justin, 2026-07-06)

Justin's catch, folded in. The contact↔Evryn shape is right while the person is *just a lead* (gold / edge / pass) — Mark is neither sender nor recipient, so it stays out of his history. But when a gold **becomes a match** (the gatekeeper confirms the fit — the `emailmgr_items` `delivered → matched` transition), the contact and Mark are genuinely connected, and the original email *belongs* in the contact↔Mark relationship. So:

- **At the `matched` transition, re-attribute the preserved forward message's recipient** from `EVRYN_USER_ID` → the gatekeeper (Mark). The target is already stashed — the `metadata.intended_for` written at forward-time — so the conversion is just `recipient_id = metadata.intended_for`.
- **Atomic with the status flip.** The `delivered → matched` transition and the message re-attribution are ONE operation (an RPC / single transactional step), so there's never a window where the match exists but the message is still mis-attributed (or vice versa). This is the atomic gold→match process Justin described.
- **Only `matched`.** A pass/edge that never matches stays contact↔Evryn forever (preserved for the v0.3 reach-back, never in Mark's history). Only a *confirmed connection* enters Mark's history — which is exactly when Mark should have that original email in his conversation record with this now-matched person.

**Why this is elegant (it fully closes the pollution question):** the message is contact↔Evryn *until* match, contact↔Mark *at* match. Out of Mark's `getRecentMessages` window while it's just a lead; *in* it precisely when the two connect. "When does it belong in Mark's history?" now has a clean answer — **at match, not before.**

**Coherence with the existing outbound:** at match, Mark's history already carries Evryn's gold-notification (recorded Evryn→Mark by `executeApproval`); this adds the contact's *original* email (contact→Mark). Both fit — Mark sees the email he forwarded *and* Evryn's notification about it.

**Scope:** rides the 54b build (extends the design; doesn't change §5's decisions). Where/how the atomic `matched`-transition conversion lives is a DC design point for the 54b build brief.
