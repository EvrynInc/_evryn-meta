## 1. The problem (in one sentence)

**Environment selection is implicit and shares a namespace, so a "dev" integration test that forgets to override its credentials silently runs against PROD — the worst possible default (fail-open-to-prod).**

## 2. The concrete mechanism (verified against the runtime, 2026-07-01)

- The runtime AND every `@supabase/supabase-js` integration test read the **same two env vars**: `SUPABASE_URL` + `SUPABASE_SERVICE_KEY`. Verified: `evryn-backend/src/config.ts` declares both via `required(...)` (single-valued, no dev/prod variant), and `evryn-backend/src/db/client.ts` builds the one Supabase client from `config.supabaseUrl` + `config.supabaseServiceKey`. **There is no dev-vs-prod selector in the runtime** — it's whatever those two vars point at, and in `evryn-backend/.env` they point at **prod**.
- The psql/pg_dump admin tooling uses a *different*, correctly-qualified set: `SUPABASE_DB_URL_PROD` / `_DEV` / `_EAST` (per ADR-037). So there are **two different credential schemes for "the same" environment** — the qualified psql scheme, and the single unqualified supabase-js scheme.
- Consequence: pointing a supabase-js integration test at DEV means *overriding* `SUPABASE_URL` + `SUPABASE_SERVICE_KEY` for that run. **Forget the override, and a "`-dev`" test hits prod** — creating users/items, calling `record_pass`, etc., against live data.
- **This bit us this session (2026-07-01):** AC2 applied the Haiku-screen migration to DEV via the psql string `SUPABASE_DB_URL_DEV` (correct), but the DEV *integration test* (`evryn-backend/tests/test-haiku-screen-dev.ts`) reads `SUPABASE_URL` / `SUPABASE_SERVICE_KEY` = **prod** — so the "dev" test would have run against prod. Caught by inspection, not by structure.

**Today's only protection is vigilance:** test-file headers say "never prod," and one test (the Step-32 `executeApproval` transport test, `evryn-backend/tests/test-execute-approval-transport.ts`) carries a hand-rolled "refuses prod" DEV-only guard. Justin's point: **relying on remembering to check every run means we WILL eventually miss it.** This is a security-by-construction gap that contradicts Evryn's own bulkhead mindset.

## 3. Why it's like this now (do NOT naively reverse it)

Two `SUPABASE_SERVICE_KEY=` lines in one `.env` **collide** — dotenv is last-one-wins — so the dev *API* creds were moved **out of `.env` into Bitwarden** to avoid the collision (per ADR-037 notes / Justin's recollection). **Adding a second same-named key back into `.env` recreates the exact collision.** So the fix is **structural (rename the namespace), not a config add.**

## 4. Design directions (options — a decision for the assigned owner, with AC2's recommendation)

These are the four Justin laid out, lightly sharpened:

1. **Explicit env selection, fail-SAFE-to-dev (the root fix).** Qualified creds — `SUPABASE_URL_PROD` / `SUPABASE_URL_DEV` + `SUPABASE_SERVICE_KEY_PROD` / `_DEV` (killing the same-name collision) — plus an `EVRYN_ENV` / target selector. The **runtime declares prod explicitly**; **tests default to DEV and must opt into prod deliberately.** This inverts fail-open → fail-safe.
2. **Structural test-DB guard (the belt).** Every integration test asserts the connected project ref ≠ prod and hard-aborts otherwise — generalizing the one-off Step-32 guard into a shared helper. Security-by-construction, matching the `STAGING_RECIPIENT_ALLOWLIST` fail-closed send-gate already in `evryn-backend/src/email/client.ts`.
3. **Separate test config / repo (Justin's idea — heaviest).** Keep test fixtures + runner config where prod creds can't sit one variable away. #1 + #2 likely buy the same safety without a repo split — weigh cost vs. benefit.
4. **Fold into the in-flight staging work.** ADR-045 + SPRINT Steps 78–82 already stand up a dedicated 4th Supabase project + staging runtime + send-allowlist — **the same environment-separation family.** There is likely a shared "fail-safe env-selection" primitive to design once, across both.

**AC2's recommendation:** do **#1 + #2 together** — #1 as the root-cause structural fix (fail-safe env selection), #2 as the belt-and-suspenders guard — and **design them in the same family as the staging work (#4)** so there's one env-selection primitive, not two. This is architectural → it deserves its own **tracked SPRINT step + likely an ADR** (env selection is an architectural decision). **Not a band-aid** (Justin was explicit on this).
