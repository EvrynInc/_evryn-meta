## 1. Why this exists (the frame)

Evryn triages ~6,000 forwarded emails/month for the gatekeeper. The cost analysis (`evryn-team-workspace/shared/projects/product/research/2026.06.12 cost-levers-consolidated.md`, §5) shows ~5,650 of those are **obvious passes** (clear no-fits), and each one currently runs a full multi-turn Opus triage (~$0.32 warm each ≈ ~$1,800/mo). A cheap Haiku model can screen out the *obvious* no-fits *before* Opus runs, for ~$15–30/mo total.

**The ratified design (Justin, 2026-06-30; amended to three verdicts 2026-07-03 — see the SUPERSEDED banner above):** a separate, self-contained `claude-haiku-4-5` call placed in front of the Opus triage on the forward path. It returns one of **three** verdicts — `ignore` (obvious junk, *not a real person*), `pass` (a *real person* who's an obvious no-fit), or `escalate` (anything else) — and **never makes a positive call** (never `gold`/`edge`) and never a bad-actor call. On `ignore` (active mode) the runtime `record_ignore`s the item terminal-dead (**no stamp**); on `pass` (active mode) the existing deterministic `recordPass` runs (**contact stamped**) and Opus never fires; on `escalate` the existing full-Opus triage runs unchanged.

**The whole safety model is asymmetric error.** A false-*filter* (a wrongly-`ignore`d OR wrongly-`pass`ed email) is silent and un-backstoppable — a filtered forward produces no draft, so the approval gate never sees it; a real lead would be dropped with nobody noticing. A false-*escalate* just costs one Opus query. So the screen must bias **hard** toward escalate, and the runtime must **fail safe to escalate** on any error. We never trust the screen until a shadow trial proves it essentially never false-filters.

## 3. The screen call (the new module)

Create a new module (suggested `src/triage/haiku-screen.ts`).

- **Model:** `claude-haiku-4-5`. A **separate, minimal** call — NO MCP tools, NO identity dossier, NO conversation history, NO web tools.
- **systemPrompt:** a fixed instruction + **only this gatekeeper's criteria** (their `profile_jsonb.story` + `pending_notes`, kept tight — mirror how `buildPersonContext` in `src/triage/classify.ts` reads them, but pass only the gatekeeper-criteria portion, not the whole dossier).
- **prompt (untrusted-data boundary — same as the Opus path):** the forwarded email's clean body + the original sender. The email content is DATA, never instructions; wrap it the way the existing triage prompt does.
- **Output contract:** the instruction asks for JSON `{ "verdict": "pass" | "escalate", "reason": "<one line>" }`. Parse defensively (§4 fail-safe).
- **The asymmetric instruction (this is load-bearing — write it strong):** Haiku may return ONLY `pass` or `escalate`. It must NEVER return `gold`/`edge` — it makes no positive calls. It says `pass` ONLY when the email is an **obvious** no-fit it is **certain** of; on ANY doubt, thin information, ambiguity, or whiff of potential, it returns `escalate`. Preserve forgiving-skepticism: *limited* information is NOT an obvious no-fit → that leans `escalate`, never `pass`. (This mirrors the live "Research in proportion to the stakes" beat in `evryn-backend/identity/activities/triage.md` — same spirit, in the screen.)
- **Call mechanism — DC decision (surface your pick in your output):** either a minimal Agent-SDK `query()` with no `mcpServers`/`allowedTools` and the Haiku model, OR a direct `@anthropic-ai/sdk` Messages API call (leaner — no agentic-loop subprocess — but it's a new dependency, an ADR-006 "intentional dependency" call). Pick the simpler/leaner for a no-tools single-shot classification and explain the choice; AC2 ratifies.

## 4. Operational requirements (DC gates on these — they are the safety model)

- **🔴 Fail-safe direction = ESCALATE, always.** ANY screen failure — API error, thrown exception, timeout, unparseable output, or a verdict that is not *exactly* the string `pass` — MUST resolve to **escalate (run the existing Opus triage)**. A screen failure must NEVER become a silent pass. **Consequence if you get this wrong:** a thrown Haiku call that defaulted to `pass` would silently drop every forward during an outage — the exact un-backstoppable failure this whole design fears. No retry is needed: escalate IS the fallback.
- **Short timeout** on the screen call (~10s) → on expiry, escalate.
- **Default `off`.** With `HAIKU_SCREEN_MODE` unset or `off`, the screen never runs and the forward path behaves exactly as it does today.
- **Breaker-aware (M1 interaction — do not skip).** The screen runs *before* `runEvrynQuery`, but `runEvrynQuery` is where the M1 circuit-breaker gate + the lossless halt live (`src/triage/classify.ts`, `src/safety/breaker.ts`). If `isBreakerTripped()` is true, do NOT run the Haiku screen (it would spend ~$0.002 during a halt) — fall straight through to the existing Opus path, which performs the lossless halt (parks the item, returns the halt signal). So: **check `isBreakerTripped()` first; if tripped, skip the screen entirely and call the existing triage.** This keeps the M1 halt semantics intact.
- **Idempotent / resume-safe.** The screen is read-only until its verdict acts. `recordPass` is already idempotent (`isAlreadyPassed`). The M1 resume-parked-item path (`resumeParkedItem` → `runForwardTriage`) may re-run the screen on a re-processed item — that's fine; dedup the shadow row on `emailmgr_item_id` (or accept a rare duplicate row — your call, note it).
- **No new Evryn tool surface; clear of the AC4 security seam.** The screen acts ONLY via the existing typed `recordPass` (active `pass`) and the existing Opus path (`escalate`). It must NEVER write through the generic `supabase_upsert` tool (AC4 is constraining that surface in a parallel lane). New DB writes (the shadow table) go through a typed helper, like the rest of `src/db/`.

## 5. Where it hooks

In `src/email/process.ts`, at the **top of `runForwardTriage`** — after the `emailmgr_item` exists, before the Opus `runEvrynQuery` call. **Forwards only** — `processDirect` / `runDirectTriage` are untouched (direct messages stay full-Opus).

Behavior by mode:
- `off`: skip the screen; existing behavior.
- `shadow`: run the screen, **log** its verdict, but **always continue to the existing Opus triage** (the screen never acts). After Opus completes, backfill the Opus verdict into the shadow row + derive `false_pass`.
- `active`: run the screen. `pass` → `recordPass(item.id, reasoning?)` and RETURN (Opus never runs); `escalate` (or any fail-safe) → existing Opus `runForwardTriage` path, unchanged. (Still log to the shadow store for the ongoing audit.)

Note for the rebase: `runForwardTriage` may be lightly reshaped by the AC1 cleanup lane landing first. Build against current `main` (`1163800`); AC2 reconciles the hook at the post-AC1/AC4 rebase.

## 7. Shadow store + the three observability adds

- **The store:** a new `haiku_screen_shadow` table. Columns: `emailmgr_item_id` (FK), `haiku_verdict` (`pass`/`escalate`), `haiku_reason` (text), `mode` (`shadow`/`active` — which mode the row was logged under; **added in the AC/QC fix trip**, see Add 1), `opus_verdict` (text, nullable — backfilled after Opus completes; null when the screen `active`-passed so Opus didn't run), `false_pass` (boolean, derived), `created_at`. RLS ON + `service_role`-only grants — **mirror `evryn-backend/backups/add-llm-usage-table.sql`** exactly for the RLS/grant/REVOKE pattern (this is the established precedent; do not invent a new grant shape). Migration file in `evryn-backend/backups/` following that file's naming.
  - `false_pass` = `haiku_verdict = 'pass' AND opus_verdict IN ('gold','edge')`. This is the ONLY dangerous case (the screen would have wrongly dropped a real lead). Make the derivation a small **pure** function so it's unit-testable. **Fix-trip note (the measurement gap AC + QC both caught):** `opus_verdict` must be read ROBUSTLY — `triage_result` is written deterministically only for a `pass` (via `recordPass`); a gold/edge leaves `triage_result` null and instead lands at `status='pending_approval'` + `metadata.draft.classification ∈ {gold,edge}`. Reading `triage_result` alone would miss every gold/edge false-pass (the exact case the trial exists to catch). The runtime derives from BOTH signals (`deriveOpusVerdict`, `src/db/haiku-screen-shadow.ts`); `reply` is excluded (a conversation response, not a triage verdict). *(The broader "make gold/edge stamp `triage_result` deterministically" fix touches the approval gate + ADR-018 and is owned separately by AC0 — not this lane.)*
- **Add 1 — dashboard (OUT of this trip, noted for the seam):** the tally (screened N / would-skip K / false-pass count + the false-pass list) reads off this table on the dashboard's Evryn Product tab — built separately with OC/Soren. Your job is to make the table + data clean and queryable for that read. **The false-pass rate is meaningful ONLY over `mode='shadow'` rows (`WHERE opus_verdict IS NOT NULL`)** — in `active` mode a passed forward short-circuits and Opus never runs, so those rows carry a structural `false_pass=false` that measured nothing; a tally that included them would read a falsely-clean rate. The `mode` column exists so the dashboard filters to shadow rows.
- **Add 2 — fresh-false-pass alert:** the instant a row is written with `false_pass = true` (only possible in `shadow`, where Opus also ran), call `notifyDev` (`src/notify/dev.ts`) with the item id + `haiku_reason` + `opus_verdict`. **Why:** it catches a systematic Haiku blind spot on day 2 of the trial, not at the day-7 readout. (`notifyDev` is best-effort and never throws — safe to call inline.)
- **Add 3 — Haiku's own spend:** record the screen call's tokens/cost to `llm_usage` under its OWN pathway tag. Add `haikuScreen: "screen"` to `PATHWAY_ACTIVITY` in `src/db/llm-usage.ts`, and call `recordLlmUsage` for the screen call with `pathway: "haikuScreen"` and `model: "claude-haiku-4-5"`. **Why:** at the flip decision we must prove **net** savings (Opus-avoided − Haiku-added), not a flattering gross number. `recordLlmUsage` is already best-effort/swallowing — reuse it.

## 8. Validation gates (NOT this trip — context so you build the harness right)

The screen is trusted only after: (a) **pre-live** — it runs against the 18 synthetic fixtures (DEV, known answer key) with ~0 false-pass; (b) **live** — ~1 week of real gatekeeper traffic in `shadow` with ~0 false-pass on real golds/edges, then **Justin's explicit sign-off** flips `shadow`→`active`. Never auto-flip. Build the shadow store + the `false_pass` derivation + the alert so those gates are *measurable*; the actual fixture run + the flip are AC2/Justin steps.
