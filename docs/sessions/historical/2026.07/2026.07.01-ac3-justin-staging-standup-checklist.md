## 1. How the DB env-fix (just built) changed this picture

The env-fix (ADR-048) makes staging **safe by construction** instead of by vigilance:

- **Before:** staging would have shared the same unqualified database credentials as prod and dev, so the only thing stopping staging from accidentally pointing at the *production* database was remembering to check — a manual checklist you'd run every standup.
- **Now:** each environment declares itself explicitly. Staging gets its **own named credentials** (an `EVRYN_ENV=staging` selector + `SUPABASE_URL_STAGING` / `SUPABASE_SERVICE_KEY_STAGING`), and a structural tripwire refuses to run against the production database ref, period. So when you create the 4th Supabase project below, its credentials simply become "the staging pair," and the runtime can't confuse them for prod.

**Net effect on your list:** the *shape* of your four actions is unchanged — but the wiring afterward is cleaner and the "did I accidentally cross-wire staging to prod?" risk is gone. You create the accounts; the env-selector handles keeping them straight.

## 2. The two email addresses — your question, resolved

You're reading it exactly right. They're two **different functions**:

| Address | What it is | Who "owns" it | Needs to be… |
|---|---|---|---|
| **`systemtest@evryn.ai`** | **Fake Mark's inbox.** Where *you*, playing Mark, receive Evryn's notifications and forward emails from. | You (as Mark) | **Stays an alias on `justin@`** — no change, no new seat. It only *receives* + you *forward* from it. (One requirement: it must be on staging's send-allowlist so staging-Evryn can actually email it — I handle that.) |
| **`evryn-staging@…`** | **Evryn's staging inbox.** The staging runtime *polls* this address (Gmail API) exactly like prod-Evryn polls `evryn@evryn.ai`. | The staging runtime | **A standalone, pollable account with its OWN login + OAuth**, separate from prod's `evryn@`. |

**Why `evryn-staging@` must be standalone (not an alias):** the runtime logs into Gmail with a refresh token tied to a *specific account* and uses the Gmail history API to track "what's new since last poll." An alias shares its parent account's mailbox — it can't have its own OAuth or its own poll cursor. And if staging borrowed prod-Evryn's `evryn@` login, you'd have two runtimes fighting over the *same real inbox* (the inbox version of two programs writing the same file). A dedicated account gives staging its own isolated inbox + cursor.

**💰 The money-saver (you don't need a paid seat):** `evryn-staging@` does **NOT** need to be a paid `@evryn.ai` Google Workspace seat. Staging only ever holds *fake* data and is never seen by a real user — so a **free `@gmail.com` account** (e.g. `evryn.staging.test@gmail.com`) polls + sends via the Gmail API + OAuth *identically*, at zero cost. Use an `@evryn.ai` seat only if you want the tidiness of a matching domain — it buys nothing functional for staging. **My recommendation: a free Gmail.** (If you'd rather it look "real," a Workspace seat works too — your call, but don't pay for it on my account.)

## 4. How staging gets *used* (so the pieces make sense)

Staging is a **full-dress rehearsal** you run *before* any change ships to prod: deploy the exact candidate build to staging, exercise the whole system end-to-end against staging's DB (emailing only allowlisted test addresses), confirm it all works, then promote the **identical** build to prod. Pre-Mark a bad deploy is recoverable; once Mark's forwarding it isn't — staging is how every future change gets proven as a *whole system*, not just "the code looks right."

**The concrete loop:**
1. You (as **fake Mark**, at `systemtest@`) forward an email to **`evryn-staging@`**.
2. Staging-Evryn polls it, triages, drafts a notification, and posts it to **`#evryn-approvals-staging`**.
3. You approve in the staging Slack channel.
4. Staging-Evryn emails **fake Mark** (`systemtest@`) — allowed *only* because `systemtest@` is on the send-allowlist.
5. You-as-Mark see the result. **All of it against staging's DB — zero risk to prod or real Mark.**

The **send-allowlist** (already built + shipped) is the safety net: staging runs real email send, but the runtime only ever delivers to allowlisted test addresses (`systemtest@` + `review@evryn.ai`). A real person can *never* be emailed from staging, even if a draft's recipient is wrong.
