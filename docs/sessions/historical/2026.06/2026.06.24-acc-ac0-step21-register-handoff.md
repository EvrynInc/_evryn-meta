## Section A — Step 21: durable email cursor / cold-start catch-up

**Branch:** `r2/lane-a-step21-durable-cursor` (worktree `evryn-backend-step21`). **Commits:** `a6a96c3` (build), `35506f4` (pagination test fast-follow). **QC-GO ×2** (original + re-verify of the test/refactor delta).

**What it fixes (convergence-plan GAP 1 = SPRINT Step 21):** the Gmail poll cursor (`latestHistoryId`, `poll.ts`) is in-memory only; on a process restart the old cold-start / historyId-expiry fallback (`fetchLatestBatch`, capped at `maxResults: 10`) silently dropped mid-window mail when >~10 arrived during downtime — breaking M1's lossless-restart guarantee on exactly the path M1 exists for.

**The design (Justin's "bookmark" framing):** the durable bookmark is the newest email we already durably stored — `getNewestItemCreatedAt()` (newest `emailmgr_items.created_at`). On the fallback path (and only there), re-scan the inbox for everything since that bookmark **minus a generous margin** (`DEFAULT_CATCHUP_MARGIN_MS = 1h`) via a **paginated** Gmail date-search (`in:inbox after:<unix-sec>`, follows `nextPageToken` to completion — no one-page cap). The existing durable `external_id` dedup (Step 19's `resolveExternalId` + `findItemByExternalId`) absorbs the deliberate overlap. **Invariant:** the window always errs OLD (margin + floor-to-seconds both move earlier; clamped ≥0) → never starts newer than the last stored item, so it can't drop. Normal incremental polling is unchanged. True cold start (no stored items, e.g. post-wipe) degrades to from-now, never errors.

**Files (diff vs Lane A base):**
- `src/db/items.ts` — `getNewestItemCreatedAt()` (the durable bookmark; null on empty table, **throws** on Supabase error so the caller holds-and-retries rather than mis-recovers; keyed on `created_at` — insert time, immune to the Step-13 `updated_at` trigger).
- `src/email/client.ts` — `deriveCatchUpPlan()` (PURE, unit-tested window math), `DEFAULT_CATCHUP_MARGIN_MS`, the `NewestStoredItemProvider` lazy-provider type, `fetchColdStartCatchUp`, `fetchCatchUpBatch`, and (fast-follow `35506f4`) `gatherCatchUpMessageIds(query, list)` with an injectable `CatchUpLister` (default = real Gmail). **`client.ts` stays DB-free** — the bookmark read is the injected closure, invoked only on the fallback. Wired into both fallback entry points (cold start + 404 historyId expiry).
- `src/email/poll.ts` — `pollOnce` passes `getNewestItemCreatedAt` as the lazy provider (one line).
- `tests/test-catchup-cursor.ts` (pure window-math: err-old, margin, cold-start, dedup-overlap, clamp, defensive) + `tests/test-catchup-pagination.ts` (multi-page `nextPageToken` to completion — the heart of the fix).

**M1 interlock — SATISFIED on this branch:** M1's lossless resume needs BOTH Step 21 (this) AND Step 19 (empty-Message-ID → `email.id` fallback). Both are present on this branch's base (Step 19 = Lane A's `resolveExternalId`), so the dedup that absorbs the catch-up's overlap is non-null-keyed. QC re-confirmed.

**Tests:** `tsc --noEmit` clean; `test-catchup-cursor`, `test-catchup-pagination`, `test-poll-helpers-r2`, `test-poll-retry`, `test-catchup-missed`, `test-send-transport` all pass.

**No migration.** The bookmark is derived from existing `emailmgr_items` data — no new table by design.

**One deferred follow-up (QC NB, NOT in this bundle — track as a new SPRINT step):** in the catch-up path the next-resume `historyId` is read from `getProfile` AFTER the `messages.list`, so a message arriving in the sub-second gap between the two calls could be skipped by the next incremental poll. It's **pre-existing** (the old `fetchLatestBatch` had the same shape) and tiny, but since Step 21 *is* "lossless restart," the clean fix is to read the historyId BEFORE the list across all three fetchers (dedup absorbs the deliberate overlap, err-old). QC and ACc both judged it out-of-scope for Step 21 (it's a cross-fetcher cursor-contract change touching EVR-71 logic, deserves its own pass). **Recommend AC0 add it as a small follow-up step.**
