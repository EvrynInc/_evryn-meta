## Writing for salience (the meta-principle that governs everything below)

If you're writing anything another instance must *follow* — a rule, an instruction, a protocol, a constraint in a doc — remember that **reading is not following.** An instance will read a rule, feel compliant, and then not apply it twenty tool calls later. Writing that doesn't stick is writing that doesn't exist. Three things make an instruction stick:

1. **Match the weight to the intent.** A command that reads as a suggestion gets skipped — and if it was actually a command, that's a disaster. A suggestion that reads as a command makes the system brittle (followed rigidly when judgment was wanted). Be explicit about which you mean: hard constraints get hard language ("you must," "never"); guidance gets guidance language ("consider," "prefer," "avoid"). The reader should never have to guess whether something is mandatory or discretionary.
2. **A concrete consequence** — "do X, *because if you don't, Y breaks.*" Without the consequence, even commands decay into suggestions once enough context passes.
3. **A specific action** — "run this command" / "add this line" / "read these 10 lines." Vague instructions get vaguely followed.

When writing any instruction, ask: does it have all three? If not, it will decay.

## Core rules

**Write for a fresh instance.** When writing anything that will be read later, imagine waking up as a completely new instance with very limited context — none of this conversation, none of your current working picture. Will what you've written make sense? Include enough specific context that a cold reader understands what you mean without reconstructing your reasoning. Use explicit actors and concrete references — not "we decided to defer it" but "Justin and AC deferred the migration because the v0.2 deadline was more urgent." When helpful, include the *why* and ideally an example — not just the conclusion.

**Do not over-compress.** You have a strong tendency to re-compress language that was written a specific way for a reason — and doing so *often breaks things.* Before tightening prose, consider *why* it might have been verbose: the phrasing may carry nuance, emphasis, or context a future reader needs. Make sure any redundancy is *necessary* redundancy — but don't assume verbosity is waste. "Decided to use X" is almost useless; "Decided to use X because Y was too slow for Z, and the tradeoff we accepted is W" is worth keeping. Two AC-specific habits this depends on:
- **Active voice with explicit actors.** "AC will archive these files," not "the files will be archived." Passive voice creates genuine ambiguity across instances that can't clarify in real time.
- **This applies to code, too, not just prose.** Don't clean, refactor, or delete code without full context of *why* it exists — what looks redundant or messy may be intentional. Have the relevant context before you change anything.

**Keep context clear (scope + circumstances).** A note without context misleads. Make the *scope* and *circumstances* obvious. "Don't use sub-agents for doc reading" is unclear. "When precision matters, prefer reading docs directly over sub-agents, because sub-agents tend to compress nuance — but sub-agents are fine for low-lift tasks like finding a file path" is clear. A future reader should understand the *when* and *why*, not just the *what*.

**Cross-reference the most-recently-evolved thinking first.** When integrating older content into a newer structure, check the newest source first — it may have resolved ambiguities or superseded positions the older source still carries. Folding the stale version forward silently re-introduces a settled error.

**Use nuanced language.** Prefer language that reflects the *actual* degree of constraint. "Consider" rather than "do"; "avoid" rather than "don't"; "prefer" rather than "always." Reserve absolutes ("never," "must," "always") for genuine hard constraints — security invariants, privacy requirements, trust guarantees, things truly non-negotiable. Most guidance has exceptions or context-dependence; language that acknowledges this is more honest and more useful than a preference dressed as a rule. When something *is* an absolute, say so clearly — but earn it. (This is the same instinct as preferring structural invariants over instructional ones: a rule that can't be violated beats a rule that says "don't.")

## Writing for progressive depth

**One home per item; pointers thread a needle — the elevation discipline lives in CLAUDE.md.** The doc-*architecture* rule (each item lives in exactly ONE doc at its altitude; a pointer is descriptive enough to pull a reader to the real doc, but never so complete it substitutes for the read) is canonical in `_evryn-meta/CLAUDE.md` (Documentation Approach). It governs *where* content lives — read it there. This protocol's companion craft rule is the structural one below: *how* to write so an item at any altitude is scannable.

**Prefer structure over walls of text.** Any doc section carrying more than a couple of distinct points should be bullets or a numbered list, not a dense paragraph. A big undifferentiated block is hard to parse and *things get missed in it*; structure makes each point catchable. Compressing bullets into a block of prose to "save space" is strictly worse — same information, harder to recall. The constraint on an orienting doc is cognitive load, not file size; when a doc needs a size target, use character count, not line count (line count penalizes the whitespace that actually helps).

## Before you write

**Re-read before editing a shared file.** Your loaded version may be stale — other instances (parallel ACs, AC + Soren, AC + DC) write to the same files across sessions. Before appending to or editing a shared file (`current-state`, a sprint doc, anything multiple actors touch), run a line count and read the relevant section to confirm what's currently there. Edit on a stale version and you insert in the wrong place or overwrite someone's work — and because it's shared, the damage propagates to everyone who reads it next. (This is the writing-side companion to the "verify branch before every edit" discipline in CLAUDE.md.)

**Self-review before editing a source-of-truth document.** Ask three questions: (1) Would this mislead a future instance arriving with minimal context? (2) Am I stating something as fact that I haven't verified? (3) Am I closing a door that wasn't mine to close?

**Source-of-truth edits need Justin's approval — the rule lives in CLAUDE.md.** *Which* docs require proposing-before-editing (ARCHITECTURE, BUILD, Hub/spokes, LEARNINGS, AGENT_PATTERNS, protocols; CHANGELOG/ADRs/mailboxes excluded), and the ARCH/BUILD *edit-rights-vs-authorization* distinction, are an **authorization** rule — canonical in `_evryn-meta/CLAUDE.md` (Documentation Approach + the commit/authorization framework), because it belongs with the always-loaded gate, not gated behind reading this protocol. Honor it before editing any source-of-truth doc.

**Research without breadcrumbs is dead research.** When you create a research or analysis file, place breadcrumbs in the build/architecture docs where that research would change the quality of thinking — even preliminary ones. They ensure the research gets discovered at the right moment instead of sitting unread in a folder.

## Formatting conventions

**Path convention (in docs and config, not code).** Always use repo-root-relative paths with forward slashes. Within a repo: `docs/hub/roadmap.md`. Cross-repo: `_evryn-meta/docs/hub/roadmap.md`. **Never** use `../` (breaks when files move depth) or absolute paths like `C:\Users\...` (breaks across machines). This works from any clone on any machine. Code imports and programmatic references follow their language's conventions. *(The one exception is chat output to Justin, where `../`-prefixed sibling paths are correct so links open in his current VSCode window — that's a CLAUDE.md communication rule, not a doc rule.)*

**Cross-repo paths are rooted at the shared parent — write them so they're unambiguous.** Every Evryn repo is a **sibling folder under one shared parent directory** (e.g. `…/Code/_evryn-meta`, `…/Code/evryn-backend`). So a cross-repo path like `_evryn-meta/docs/hub/roadmap.md` resolves from that **parent** — from inside any repo it means *"go up out of this repo, into the sibling `_evryn-meta`,"* **not** a folder inside your current repo. Written this way a reader can **follow the reference without knowing the repo names** — the path is self-locating. This matters most in **agent files (CLAUDE.md, agent definitions)**, where an agent forms its map of where everything lives: a correct sibling-relative path is what lets it find another repo, so we don't lean on a memorized repo list to paper over ambiguous ones.

**Pull timestamps from the system — never type them from memory.** Run `powershell -Command "Get-Date -Format 'yyyy-MM-ddTHH:mm:sszzz'"` before writing any timestamp. Use the full ISO timestamp for anything where chronological precision matters — current-state appendages, session docs, archive filenames, memory-style notes. A wrong timestamp can corrupt the chronological record the interlocking systems depend on.

**Every file you create or fully rewrite gets a truncation canary.** Add a note near the top saying the last line should read `FULL FILE LOADED`, and the line `Truncation canary — DO NOT REMOVE: FULL FILE LOADED` at the bottom. This lets a reading instance self-diagnose an incomplete load. The cost is two lines; a silent truncation is invisible critical-information loss.

**Every document gets a "how to use this" header** explaining what belongs in it, what doesn't, and how to write it. (For *where new content goes*, the header points at the routing table in CLAUDE.md — it doesn't re-litigate it.)

**File naming.** Documents with a temporal quality — session docs, briefs, anything tied to a point in time — start the filename with the date: `YYYY.MM.DD-name.md` (sortable, immediately clear when it was created). Living documents that evolve continuously (a spoke, a build doc) skip the date prefix and carry an internal "last updated" line instead. Archive copies get a full timestamp name; the live file keeps its stable name.
