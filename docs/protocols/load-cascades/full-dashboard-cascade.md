## The nature of THIS target — read this before the load list

The dashboard is a **non-agentic** static SPA (`public/index.html`, vanilla JS + Chart.js) plus four Vercel Edge Functions (`api/data.ts`, `api/usage.ts`, `api/product.ts`, `api/team.ts`). **It has NO identity half of its own** — that's exactly what makes it a clean `#cascade-override` target under `ac-orchestration-protocol.md` ("When the target is a SEPARATE codebase"): a dashboard build swaps in *this* load and does NOT pull the Evryn-runtime cascade, because the dashboard doesn't compose wakes or run agent judgment.

**BUT its whole job is to read (and, for the Team tab, WRITE) the state of TWO agentic runtimes plus one legacy DB.** So its dependencies are **contracts owned elsewhere, in systems that evolve independently:**

| The dashboard's… | reads/writes | owned by |
|---|---|---|
| **Evryn Product tab** (`api/product.ts`) | `llm_usage` · `messages` · `emailmgr_items` + Railway `/health` | the **Evryn product runtime** (`evryn-backend`) |
| **Team Runtime tab** (`api/team.ts`) — a **control panel that WRITES** | `agent_state` · `agent_capabilities` · `capability_catalog` · `capability_tiers` · `wake_manifests` | the **team runtime** (`evryn-team-runtime`) |
| **Agents tab** (`api/data.ts`, `api/usage.ts`) | `agent_daily_spend` · `agent_api_calls` · `agent_messages` | the **legacy Agents DB** (prior-era team monitoring) — see §3's PROVISIONAL note |

**This is a FLUID-CONTRACT problem, not a stable-runtime one** — and it is the sharpest form of the "identity-as-runtime dividend." A `dashboard/`-only read tells you what the dashboard *thinks* the contracts are; it CANNOT tell you the runtime has since renamed a column, added a status, changed the `/health` shape, or added a state column (as the team runtime did with `state_reason` — see §3). When that drifts, the dashboard doesn't error — it shows a **dropped light, a stale zero, a mis-labelled status, or (on the Team tab) writes a row the runtime's gate no longer reads the way the dashboard assumed.** The Team tab is the highest-stakes: `api/team.ts` explicitly says *"do not invent — this must match the process exactly, since both write the same rows"* (the ~5s gate-cache TTL means a mismatched write bites within seconds).

**⇒ How to hold this cascade (the "stable spine + drifting contract-register, re-verified live" shape):**

- **§1–§2 (the dashboard's own code + its framing docs) are the STABLE SPINE — must-load, in full, never trim.** They change only when *you* change them.
- **§3 (the CONTRACT REGISTER) is the FLUID part.** Each entry names *what* the contract is and *where its authoritative source lives in the runtime*, and tells you to **read that live source to confirm the contract is still what this doc says.** This doc is a **map to the current territory, never a substitute for it** — the runtime owns the shape, so you re-derive the shape from the runtime at build time.
- **The `Last contract reconcile:` marker (below) makes drift visible.** A stale date is a signal the register may have moved under you.
- **The humility is scoped to §3 ONLY — and it is *additive*, never subtraction.** "Re-verify against the live source" is a step you *add*; it is **never** license to skip loading a named file. You cannot competently touch a dashboard read without knowing the columns it reads, and you cannot know those without loading the source file that defines them. Under-loading here breaks things exactly the way under-loading anywhere does.

## §1 — The dashboard's OWN codebase (STABLE SPINE; load in full)

All in `_evryn-meta/dashboard/`. **Enumerate live** (`git -C _evryn-meta ls-files dashboard`).

- **`public/index.html`** (~2,000 lines) — the entire frontend: CSS, the three-tab SPA, the render + control logic, Basic-auth login, Chart.js usage chart. The load-bearing UI file — read it in full for any dashboard work. Note the three tabs load on **independent tracks** (`loadData` / `loadProductData` / `loadTeamData`, each on its own 30s interval) so one DB's failure never blanks another tab.
- **`api/product.ts`** — the Evryn Product tab's edge function. The single richest read-contract surface: it holds the `llm_usage` / `messages` / `emailmgr_items` column allowlists, the `/health` fetch + `parseHealthChecks`, and — critically — the **PII posture**. ⚠️ **That aggregates-only / PII-firewall posture is a HARD CONSTRAINT** — the dashboard is web-hosted, so any new read you add must not surface raw user data. The live allowlist that leaves the server is only aggregates/statuses/costs/timestamps + `subject` truncated to 60 chars; it NEVER selects `content_raw` / `original_from` / `metadata` / `user_id` / `scope_user_id`.
- **`api/team.ts`** — the Team Runtime control panel (GET reads state, POST writes `pause` / hard-`pause` / `kill` / `resume` / `set_capability` / `apply_tier` / `set_reason`). **Read the header comment block AND the `buildReasonText` block:** they enumerate the runtime semantics this file MIRRORS and MUST match — and they document the deliberate reason-column divergence covered in §3.
- **`api/data.ts`** + **`api/usage.ts`** — the legacy Agents tab (spend + activity + usage chart). Reads the legacy Agents DB via the base `SUPABASE_*` env pair.
- **`package.json`** (only dep: `@supabase/supabase-js`), **`vercel.json`** (`outputDirectory: public`, no build step, `Cache-Control: no-cache`), **`.gitignore`**.
- **Deploy model:** the dashboard ships from `_evryn-meta` — **the `_evryn-meta` push IS the deploy** to Vercel (confirmed in `current-state.md` for the Step-78 Half-B deploy). Auth = HTTP Basic, user `evryn`, `DASHBOARD_PASSWORD` (fail-closed if unset — no hardcoded fallback, in every one of the four functions). Env-var groups (each a SEPARATE Supabase project, never cross them): `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` (Agents) · `PRODUCT_SUPABASE_URL`/`PRODUCT_SUPABASE_SERVICE_ROLE_KEY` (Product) · `TEAM_SUPABASE_URL`/`TEAM_SUPABASE_SERVICE_ROLE_KEY` (Team) · `RAILWAY_HEALTH_URL` (product `/health`). Every endpoint returns `{ configured: false }` (HTTP 200) when its env is unset — the tab renders an inert "not connected" state, never a 500. The service-role key is used only inside the handler and never appears in a response body.

## §3 — The CONTRACT REGISTER (FLUID; RE-READ THE LIVE SOURCE to confirm each is still current)

**Each entry = what the contract is · where its source-of-truth lives · why it drifts.** Load the source file for the contract your task touches, and confirm the shape below still matches — the runtime owns it, this doc only points.

### PRODUCT-runtime contracts (`evryn-backend` — confirmed live 2026-07-21)

- **`llm_usage` columns** (Product tab cost) → source: **`evryn-backend/src/db/llm-usage.ts`** (`LlmUsageRow`). `api/product.ts` reads `total_cost_usd`, `activity`, `model`, `created_at` only. Drifts when a cost column is renamed or the `PATHWAY_ACTIVITY` map / model-id set changes (the by-model Opus/Haiku cut keys on the raw `model` string via a case-insensitive substring test, so a new model family shows under its raw id until the frontend learns its label). **PII line:** `scope_user_id` / `emailmgr_item_id` exist on the row but are NEVER selected.
- **`emailmgr_items` status + fields** (Product tab activity / recent-items / error drill-down) → source: **`evryn-backend/src/db/items.ts`** (the `ItemStatus` union — currently `new · queued · processing · pending_approval · delivered · matched · passed · ignored · bad_actor · no_gk_response · escalated · error`; note `queued` came from Clustering — plus `triage_result` [a free string, in practice `gold`/`edge`/`pass`], `subject`, `created_at`, `updated_at`). `api/product.ts` filters on specific statuses (e.g. gold/edge via `triage_result.in(gold,edge)`, "passed" via `triage_result.eq.pass OR status.eq.passed`, errors via `status.in(error,escalated)`); a NEW status the dashboard doesn't know about goes uncounted rather than erroring. **PII line (HARD):** only `created_at` / `status` / `triage_result` / `subject`(≤60 chars) may leave — never `content_raw` / `original_from` / `metadata` / `user_id`.
- **`/health` shape** (Product tab liveness banner + 5 subsystem lights) → source: **`evryn-backend/src/safety/health.ts`** (`HealthResponseBody`) + **`src/safety/liveness.ts`** (`decideHealthStatus` — the load-bearing `halted` > `wedged` precedence: breaker-tripped → 200 `halted`; poll-or-db not-ok → 503 `wedged`; else 200 `ok`) + the handler wiring in **`src/index.ts`**. Body = tri-state `status` (ok/wedged/halted) + `uptime` + `checks{ db{ok}, slack_socket{ok}, poll{ok,last_success_at,age_seconds}, m1{breaker_tripped,reason} }`. `api/product.ts`'s `parseHealthChecks` reads the OLD (no-`checks`) shape as `null` → idle lights, and reads the body **even on a 503** (that's the wedged case it exists to surface) — so it degrades gracefully, but a *renamed* check key would silently blank a light. The frontend deliberately renders a halt-caused stale poll as an orange "consequence" ring, not a second red alarm.
- **`messages.created_at`** (Product tab last-activity) → source: **`evryn-backend/src/db/messages.ts`** (`MessageRecord`). Only `created_at` leaves the table — the row carries `content_raw` / `message_body` / `scope_user_id` / `metadata` (all PII), none of which is selected.

### TEAM-runtime contracts (`evryn-team-runtime` — a SEPARATE agentic repo; the Team tab both READS and WRITES these, so drift = a broken or lying control panel) — confirmed live 2026-07-21

- **Control semantics** (pause / hard-pause / kill / resume, and the reserved `__global__` / `__runtime__` rows) → source: **`evryn-team-runtime/src/safety/kill.ts`**. The live contract:
    - **graceful pause** (`pauseAgent`, hard=false): `active=false`. Blocks NEW wakes; in-flight finish.
    - **hard pause** (`pauseAgent`, hard=true): `active=false`, `hard_paused=true`. Also parks in-flight wakes at their next tool call.
    - **kill** (`killAgent`): `active=false`, `hard_paused=true`, `killed_reason` set.
    - **resume** (`reactivateAgent`): `active=true`, `hard_paused=false`, `killed_reason=null`.
  - `api/team.ts` writes exactly this `active` / `hard_paused` triple — so the **control semantics still MATCH the runtime exactly**, and the dashboard's pause/kill/resume genuinely drive the runtime's PreToolUse gate (⚠️ the ~5s gate-cache means a mismatched write would bite within seconds).
  - **⚠️ ONE REAL DRIFT — the reason column (found this pass; the prior register had this entry as unverified).** Migration 014 added **`agent_state.state_reason`** (plain human reason, no prefix). The LIVE runtime now writes it: `pauseAgent` sets `state_reason=reason` and leaves `killed_reason=null` ("a pause is not a kill"); `killAgent` writes `state_reason` AND `killed_reason` (same text); `reactivateAgent` clears both. The dashboard **predates this migration**: `api/team.ts` still stores its reason in `killed_reason` with a `"PAUSE:"`/`"KILL:"` prefix, and its GET selects `killed_reason` (not `state_reason`). Migration 014's own comment says the two "coexist harmlessly" — **and they do for CONTROL** (nothing reads either column as a gate signal). But there is a live **display-desync landmine**: a **runtime-initiated** pause/kill (a Slack `/pause`·`/kill`, or the velocity `pauseAll`) writes an *unprefixed* `killed_reason` / a null `killed_reason` — and the frontend infers "killed" from a `killed_reason` starting with `"KILL"` (`trStatus` in `index.html`), so a runtime-side kill would render as **"Hard-paused"** and a runtime-side pause would show **no reason**. Dashboard-initiated actions display correctly (they wrote the prefix). **The flagged follow-up (per migration 014) is to repoint the dashboard to read/write `state_reason`.** Until then, record this as a KNOWN divergence, not a control bug.
- **The ~5s gate cache + what the gate actually reads** → source: **`evryn-team-runtime/src/wakes/capabilities.ts`** (`GATE_CACHE_TTL_MS = 5000`; `foldGateState`). The PreToolUse gate reads `active` (via dispatch) and `hard_paused` (via the cached `GateState.hardParked`) + the per-agent `agent_capabilities` grants — it does **NOT** read `killed_reason` or `state_reason` as control signals (those are audit/display only). One cached DB read (per agent, 5s TTL) feeds both the hard-pause check and the capability check, which is why a dashboard revoke or hard-pause lands "within seconds, not next wake."
- **Scheduler / runtime-alive row-shape** (heartbeat + in-flight thread counts + the reason-column write pattern) → source: **`evryn-team-runtime/src/scheduler/index.ts`** + **`src/boot/singleton.ts`**. The `__runtime__` row's `singleton_lock.heartbeat_at` is refreshed at the TOP of every scheduler cycle (`CYCLE_MS = 15_000`, i.e. ~15s, **even while paused** — a paused runtime is still alive). The dashboard's own freshness window is `ALIVE_FRESH_MS = 60s` (its call, independent of the runtime's `STALE_MS = 90s` reclaim threshold in `singleton.ts`). In-flight counts come from an agent row's `singleton_lock.threads` object. The dashboard's `set_reason` mirrors `buildThreadLockUpsert`'s **"deliberately WITHOUT `active`"** pattern — patch only the intended column so a stale read can't resurrect/re-kill an agent in the read→upsert window.
- **The state tables** → sources (migration + hand-maintained type):
    - `agent_state` → **migration 006** (base) + **013** (`hard_paused` rider) + **014** (`state_reason` rider); type: `src/db/types.ts` (`AgentStateRow`). Reserved rows `__runtime__` (process lock + heartbeat) and `__global__` (the global kill switch — the dashboard's `'all'` target).
    - `agent_capabilities` (the ONLY capability table the runtime reads; fail-closed = missing row is not-granted), `capability_catalog` (twirl-down descriptions + `essential` stars — dashboard-only), `capability_tiers` (PRESETS the dashboard applies; the runtime never reads tiers) → **migration 013**, extended by **014** (adds `tool:load_skill` to the catalog + the `conversation-only` tier).
    - `wake_manifests` (last-wake `reason` + `created_at` per agent) → **migration 010** (+ **014** adds `meta_sha`).
  - `api/team.ts` reads all of these **column-allowlisted** (states / capabilities / timestamps / derived thread-counts / wake-reasons only) and NEVER touches **`agent_messages`**, which carries internal team deliberation — a confidentiality line as hard as the product PII line.
  - **Minor divergence worth knowing:** the dashboard defines "the mains" data-drivenly as the distinct non-reserved agents that have `agent_capabilities` rows (`listAgents`), whereas the runtime's `killAllMains` iterates the code constant `ACTIVE_AGENTS` (`src/config/agents.ts`). A main with no capability row would be invisible to the dashboard's "Kill all mains." Currently only `lucas` is seeded (Phase 1 is Lucas-only), so this is latent, not live.
- **You do NOT need the team runtime's IDENTITY half for a dashboard build.** The dashboard reads/writes the runtime's *state tables*, not its composed wakes — so load the specific `src`/migration sources above, not the agent definitions / memory / composer. (This is the one place the dashboard's contract with an agentic runtime is genuinely narrower than a full runtime load — but it is a claim about *this* dashboard's reads, re-check it if a future tab starts reading composed-identity state.)

## The discipline (why the split is drawn where it is)

**§1–§2 are a normal must-load — you don't trim them.** The dashboard is small, but a change to one tab's read still wants `public/index.html` (the render side) AND the matching `api/*.ts` (the data side) in full, or you'll break the pair.

**§3 is where the humility lives — and it is *additive verification*, never subtraction.** The contracts are genuinely the runtimes' to define, and they move (the `state_reason` drift above is the live proof), so a frozen copy in this doc would rot; the honest move is to name the live source and read it at build time. That is a *requirement to re-read the source*, not permission to skip it. The failure this guards against is a *silent* one — a dashboard that renders a stale, wrong, or mis-labelled picture (or, on the Team tab, WRITES one) because a contract drifted and nobody re-checked. When you touch a contract, re-verify it against its live source **and** update this register's entry + the reconcile date in the same pass — so the next builder inherits the current map, not a stale one.
