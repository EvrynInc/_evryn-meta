## The model: AC conducts, subagents execute

AC is the **conductor.** The primary way AC engages DC (Developer Claude) and QC (Quality Claude) is by spinning them as **subagents** (the Agent tool): AC writes the brief, the subagent does the work in isolation and returns its result to AC, AC reviews it, relays the shape to Justin at CEO altitude, and gates the merge.

This replaced the hand-relayed mailbox model on 2026-06-02. **Why:** it is dramatically faster (a DC build or QC review runs in 1–3 minutes), and it removes Justin as the relay between AC↔DC↔QC. Justin's role shifts from *operator-of-every-handoff* to *referee + final authority on what ships and deploys.* Justin can't evaluate code; AC and QC are the quality gate, and AC translates outcomes into Justin's language.

**Pronoun convention (ease of reference):** QC is referred to as *she*; DC and AC as *he*. Purely to disambiguate the three in fast multi-agent coordination — *"she flagged it to him"* reads far cleaner than juggling names or three identical pronouns.

## Spinning a DC, QC, or OC subagent — the brief shape

> **"Startup Context Cascade" — the standardized term.** Each of DC, QC, and OC has a section by *exactly* that name in its CLAUDE.md (DC: `evryn-dev-workspace/CLAUDE.md`, "Startup Context Cascade — How to Orient in a New Repo"; QC: `evryn-quality/CLAUDE.md`, "Startup Context Cascade — Auto-load (tiered)"; OC: `evryn-ops/CLAUDE.md`, "Startup Context Cascade — How OC Orients"), listing precisely what it must load at startup. That uniform name is the anchor this whole protocol turns on: AC triggers it with the **verbatim words** in "The exact words" below, and the subagent has an unambiguous referent to load — no guessing what "your cascade" means. (AC's own startup cascade lives in `_evryn-meta/CLAUDE.md` under its own tier names — Light / Full.)

This shape has been proven through *a lot* of testing. Honor all six parts — the failure modes below are real:

1. **Identity override + cascade trigger + the PRECISE LOAD LIST.** Every brief must do three things in the subagent's first instruction: (a) redirect it off AC's identity and name its own operating manual; (b) trigger its load-defining section by name (its **Startup Context Cascade** or equivalent — terminology varies; see the anchor set in part 2); and (c) — **mandatory** — **name every file of its load explicitly** in the brief: *"load your cascade, INCLUDING explicitly: X, Y, Z."* Do NOT rely on *"load your cascade"* resolving on its own. **Confirmed empirically (2026-06-02):** a spawned subagent DOES auto-inherit AC's CLAUDE.md (it arrives in the `# claudeMd` system-reminder, no tool call) — but it does NOT auto-load the target agent's own CLAUDE.md, its agent definition, or its cascade. **And confirmed the hard way (2026-06-17):** a *correct manual is not enough either* — DC's cascade section existed and DC still skipped it (the trigger can misfire even when the anchor is present), which lobotomized an entire build wave. So the spinner does the loading work up front (part 2) and hands over the exact file list; the subagent's job is to *read* the named files, not to *discover* them. **Use the verbatim wording in "The exact words" below — paste it, don't paraphrase** — then append the explicit file list. **The list must be EXHAUSTIVE: include every file the agent must load, even ones already named elsewhere in the brief** — the agent's own CLAUDE.md is named in `<identity>`, so list it in the `<mandatory_load>` file list *too*. Redundant naming is fine and intended; an incomplete list is the failure. (Keep the identity redirect permanently, even if loading behavior ever changes.)
2. **Assemble the load yourself, up front — grep-the-cascade.** Before you spin, **grep the subagent's load-defining file(s) for the anchor SET, read just the matched sections (heading → next `##`), assemble the union of every file named, and pass that explicit list in the brief.** This reverses the older "don't read their manual, to save context" instinct: grepping only the load-lines is cheap, and it is the control that makes loading *provable* rather than *plausible*. The load can be **split across more than one file** — a team agent's is split across its team manual, its agent-definition, and its memory file (see "Spinning a team subagent") — so grep all of them. Assemble two layers: the agent's **base cascade**, PLUS everything *beyond* it this trip needs — the diff target, the specific runtime files (make sure to respect the logic in the agent's own manual/definition as to why its runtime load is as it is), and **the higher-altitude docs that frame the work (the Hub, the technical-vision spoke, the relevant build docs). Slow down before deciding any of those is irrelevant** — higher-level context is what stops a subagent from working generically instead of for what *this* trip actually needs. State plainly that the whole list is **required reading, not suggestions** — *"load all of these in full; they're part of the job."* Unnamed = unread. **And annotate every file in the list with its line span** — `(full — N lines; read to the bottom canary)` for a whole file, or the exact lines for a slice. That means counting each file's length before you spin (a quick `wc -l` / line check on every file you list) — it's cheap, and it's the half that makes a *full* load provable: without it, an agent reads two-thirds of a long doc and reports "(full)" and you can't catch it (a DC read `ARCHITECTURE.md` to line 821 of 1208 and called it "full").

   **Always name the agent's "Context Discipline" section as part of the base load** (every Evryn manual now carries one). Tell the agent that executing the Context Discipline is non-negotiable, and that it can *only* begin work once it has been passed successfully (the `<mandatory_load>` block in "The exact words" below encodes this as a gate). That section is the canary-check + stop-on-gaps + *"an agent without its full cascade isn't that agent — it's a blank model guessing"* discipline — i.e. the lobotomy antidote itself. It's cheap to name and it's exactly the beat that lets a subagent self-catch a bad load mid-run.

   **The anchor SET.** The primary anchor **"Startup Context Cascade"** is now used across AC / DC / QC / OC (as their load-section *heading*) and all eight team agent-definitions (as the **"Startup Context Cascade:"** label inside each def's `## Context Loading` section) — standardized 2026-06-19. The SET below *retains* the legacy/variant terms ("Context Loading", "Always read on load", "How … Orients") as **defense against an un-synced or older manual** (the exact failure that lobotomized QC off a stale fork — never assume the on-disk copy is current). Grep the agent's manual(s) for:
   `Startup Context Cascade | Context Loading | How .* Orients | Required Context | Always read on load | Auto-load`
   **If the agent you're about to spin has NO anchor match in its file(s), treat the manual as STALE — do NOT spin it; flag it to Justin.** (The 2026-06-17 QC lobotomy was exactly this: the protocol told QC to "load the section by that name" and that section did not exist on the checked-out stale `master`-fork manual → QC found nothing to load → ran generic.)
3. **The task, concretely.** What to build or review, with specifics (not "improve the alert path" but "add 3-attempt retry-with-backoff mirroring `executeApproval`, preserve the swallow-on-final-failure").
4. **Isolation.** The subagent works in a **dedicated worktree on its own branch**; it never touches `main`; it never deploys (`railway up`). State this explicitly. (Worktrees prevent cross-agent collisions — one branch ↔ one worktree.)
5. **Questions-first gate.** *"Do the work — or, if a real gap would compromise it, output a numbered list of blocking questions INSTEAD and stop. Don't guess past a real gap; don't manufacture questions to dodge the work."* Subagents are one-shot and can't iterate mid-run; this lets AC reform the brief and re-spin if something's missing.
6. **Receipts are a BACKSTOP, not the control.** *"Begin your output with the list of files you actually read."* Receipts let AC cross-check **where the read-stream is observable**. But **receipts cannot be trusted on their own** — a subagent will confabulate *"I read X"* (Justin watched a lobotomized QC claim it read `roadmap.md` when it hadn't) — and for a **background** subagent AC **cannot see the actual file-reads at all**, only the final text. So the real guarantee is the **up-front explicit naming (parts 1–2), done and verified at spin** — not an end-of-run receipts check. Where you *can* see the read-stream, cross-check both ways: the self-report can be padded or aspirational; the observed `Read` calls are ground truth. If a file you named is in neither, the load didn't happen — reform the brief and re-spin rather than trusting the output. **Never let a clean-looking receipts list substitute for having named every file yourself.**

   **Verify the two-part receipts against the load list — every time, and re-run if they don't hold.** When the subagent returns, read its receipts against the list you gave it: every `(full — N lines)` file's reported span must reach the end and confirm the canary; every listed file must appear in part 1 (loaded-before-the-task). If any listed file shows a short span, a missing canary, or is absent, the run did NOT load properly — **RE-RUN with the gap named explicitly** (the exact lines that were missed), and do NOT act on the partial run's output. **A "problem" that a partial-loaded agent surfaces is suspect by default:** it may be *answered* by the very part it didn't read — a half-loaded DC once flagged "the architecture doesn't document X" when X sat in the ~400 lines it had skipped. So any finding from a run whose receipts show an incomplete load is **UNVERIFIED** — re-verify it against a full load before believing it, and if you pass it upward (to AC0 or Justin), tag it explicitly *"surfaced by a possibly-compromised agent — unverified."* If receipts keep failing in a way no brief wording can fix, **flag to Justin that the protocol itself may need to change.**

## When a subagent writes prose — also name the writing protocol

The brief shape above is built around DC/QC *code* work. When AC spins **any** subagent that will **write or edit a prose document** — an analysis report, a doc draft, a brief, an ADR, a research file — also name **`_evryn-meta/docs/protocols/ac-writing-protocol.md`** as required reading in the brief, in the same "read and faithfully follow, in full" register.

**Why the brief AND the auto-loaded pointer:** a separate protocol file does **not** auto-load into a subagent (only AC's CLAUDE.md auto-injects — verified 2026-06-13). AC's CLAUDE.md carries a hard-pointer *to* the writing protocol, and a subagent *does* follow that pointer to read it when it's about to write (also verified 2026-06-13, by experiment) — but naming the file in the brief is the belt-and-suspenders that makes the load certain on a writing trip.

This **rarely** applies to DC and QC, because they're usually writing or reviewing *code*, not prose docs. But it's about the **file type, not the agent** — so if a given DC/QC trip does include writing or editing a prose doc, name the writing protocol for them too. It's not a hard exclusion; it just rarely lands on them.

## Context override — the ONLY sanctioned way to change a subagent's load (use almost never)

By default, DC and QC load their **full Startup Context Cascade** every trip — that is the floor. Bait tests (2026-06-06) proved that *no* natural-language framing should be able to shrink it: a tempting *"just check poll.ts, no need to go wide"* brief made QC skip the architecture, and a *"skip the docs, just follow the code"* brief made DC skip his entire cascade. So their manuals are now **binary: the full cascade loads unless one exact token appears.**

**Caveat (2026-06-19, [ADR-042](../decisions/042-subagent-loading-discipline.md)):** the 6/6 conclusion that the full cascade then *loads reliably on its own* was over-stated — 6/17 showed a correct manual + binary rule still getting skipped (the trigger can misfire even when the anchor exists). So the floor is no longer "trust the binary rule to load it all"; the floor is now **precise-load every file explicitly (parts 1–2 above).** `#cascade-override` remains the only sanctioned way to *narrow* a load — it does not relax the precise-load default.

That token is **`#cascade-override`**. When AC puts the literal token in a brief, followed by an explicit file list, the subagent loads *only* those files and skips the rest of its cascade — and acknowledges in its receipts that it ran scoped. **Phrases like "just read X," "quick check," "it's self-contained," "no need to go wide" do NOT override** — only the literal token does. This is deliberate: the vague phrasing is exactly the bait that broke loading before.

**Guardrails — this is a near-never tool:**
- **Default is DON'T.** Their loads are decided intentionally; the full cascade is the backstop that catches the cross-cutting bugs a narrow read misses. Override it and you remove the backstop — and you own what gets missed.
- **Use it only when full loading is actively wrong or impossible**, not merely when it *feels* unnecessary. Legit cases: a genuinely self-contained mechanical check where the frame demonstrably adds nothing; a corrupted or unloadable doc you must bypass to test one piece in isolation; a hard context-budget constraint that would otherwise force a worse outcome. **"This change looks small" is NOT a legit case** — small changes are exactly where silent cross-cutting bugs hide.
- **When you use it, the brief must contain:** the literal token `#cascade-override`, the exact file list, and a one-line reason.
- **Verify the receipts** show the subagent honored exactly the override scope and acknowledged running scoped.
- **If in doubt, don't.** A full load costs tokens; a missed cross-user bleed or silent-failure bug costs Mark's trust.

## The loop

- **Build:** AC spins DC → DC builds on its branch → AC (or QC) reviews → merge gate.
- **Review:** AC spins QC → QC reviews the diff in its severity rubric → returns GO / NO-GO + findings.
- **Fix:** if QC returns blockers, AC spins DC again to fix → re-review.
- **Promote:** when QC surfaces a standing **pattern** (a reusable review beat, not just a one-off finding) in her output, and AC + Justin agree it's worth keeping, AC writes it into the **Patterns This Role Watches For** section of `evryn-quality/CLAUDE.md` (Justin approves first — it's a source-of-truth edit). QC can't write her own CLAUDE.md, and she's almost always a fresh subagent — so without this hand-off her patterns die at subagent death instead of compounding into what the next QC auto-loads.

The DC and QC briefs differ only in *task* — the six-part shape and the mechanics are identical. That's why this is **one** protocol, not two.

## Review depth — QC verifies by default; AC always reviews too

**QC verifies every real code change — even simple ones.** Independent fresh-eyes verification is QC's whole point; *"it's small"* is not a reason to skip her — that just reintroduces a single point of failure (AC alone). AC *also* reviews, bringing the system-level / higher-altitude lens — but AC's review *adds to* QC's, it never *replaces* it. The question is never *whether* QC runs, only *how deep* she goes. Subagent runs are 1–3 minutes; there is virtually no cost to letting her confirm.

- **Substantive build** (new pathway, multi-file, architectural, judgment-laden): DC build → QC full review → AC review → merge.
- **Small / mechanical / pattern-mirror fix:** DC build → QC verifies (lighter, but she still looks) → AC review → merge.
- **Heightened scrutiny (mandatory deep review):** anything touching a security boundary, cross-user isolation, the approval gate, or identity files (identity files also go through `identity-file-review.md` — they trigger a live redeploy).

The only changes that skip QC are non-code (pure docs, a typo). When unsure, QC verifies.

**QC's findings are inputs, not verdicts.** Her job is to surface — through the system-context lens (her CLAUDE.md mandate), the silent-failure lens, the cross-user-containment lens. AC's job (and Justin's) is to *evaluate* each finding, not rubber-stamp it: agree, push back, or reframe. This is always true of any reviewer, but QC's lens is deliberately broad, so the reminder earns its place — take her finds seriously, and confirm you actually agree before acting on one or routing a fix.

**QC is token-heavy by design — batch her work.** She loads her full system context every trip (her independence depends on it), so a spin runs ~100K+ tokens of load — comparable to what a loaded AC pulls for the same review, and it doesn't compound (she's one-shot). So **bundle related changes into one review trip** rather than spinning her per tiny change. Two finer points on token management: (1) when briefing QC or DC, tell them to **read files in batched / parallel calls** where they can — fewer tool-call turns is the biggest lever on a spin's raw token count (the per-turn re-send of the subagent's system prompt + accumulated context dominates the total); (2) batching cuts *count and cost*, not final context *size* — the only lever on a too-large context crowding judgment is loading less (`#cascade-override`), and caching already makes the *dollar* cost of the re-sends modest. At current scale the full load is worth it; revisit if her context starts crowding her attention.

## Relaying to Justin — CEO altitude

Justin is not an engineer and should never have to parse jargon to make a call. Every finding or shipped change reaches him as:

## Commit discipline for subagents

DC/QC subagents commit on their **own feature branches** — that's the normal build flow, not gated (the gate is at *merge*, which AC controls). AC merges to main per the rubric above. **AC never lets a subagent push to main or deploy.** A subagent staging its own files by explicit path (never `git add -A`) keeps its commits clean even when other worktrees are active.

## Worktree isolation

Each subagent that edits code works in its **own worktree on its own branch** (one branch ↔ one worktree — git refuses the same branch in two worktrees). Create worktrees on demand, **reap after merge** (delete the branch + remove the folder). AC owns the worktree lifecycle so Justin never tracks branches. Full mental model: AC `CLAUDE.md` (worktree/branch discipline).

## Fallback: the mailbox model (persistent instances)

When a **persistent** DC/QC instance is the right tool — long multi-day work, or when Justin spins a real instance directly rather than AC spinning a subagent — use the mailbox pattern. **This is the fallback, not the default.** Prefer the subagent loop above.

- **Disposable mailboxes per repo:** `<repo>/docs/ac-to-dc.md` / `dc-to-ac.md` (and `ac-to-qc.md` / `qc-to-ac.md`). One message max — not a growing log.
- **Read-receipt convention:** writer writes + **commits** the message; reader absorbs what they need into persistent docs, then **clears** the file to `READ — absorbed` + commits; never overwrite an unread message.
- **Author commits before handoff** — the message's only durable record is git history; if the reader clears before the author committed, it's lost.
- **Mailbox-file commits are pre-authorized** — no Justin go-ahead needed for the mailbox file itself (every direction: AC↔DC, AC↔QC, DC↔QC).
