## 2026-06-30 (AC — Windows permission-prompt storm fixed via bare-form user-level settings; ⚠️ TEMP, revert owed)

- **⚠️ REMINDER — THIS IS A TEMPORARY WORKAROUND; REVERT WHEN THE CC BUG IS FIXED. PROMPT JUSTIN TO Re-check 2026-07-06.** The fix below grants filesystem-wide tool access (bare `Read`/`Edit`/`Write`, no path scoping) — deliberately broad to dodge a Windows bug, which means more leeway in the system than we want to keep. Once Claude Code fixes Windows path-scoped permission matching (GitHub [#67849](https://github.com/anthropics/claude-code/issues/67849), OPEN as of this writing), trim the desktop's `~/.claude/settings.json` `permissions` block back down and let the git-synced project `.claude/settings.json` files take over again. - **Symptom: every tool call (Read/Edit/Write/Bash) started popping approval prompts on the desktop**, despite months of smooth operation and unchanged rules. Began within the last ~2 weeks (Justin was on the laptop, so exact onset unknown); hard-blocked work 2026-06-30. Quitting + restarting VS Code and the extension did nothing. The **laptop is NOT affected.** — [Justin]
- **Root cause: path-scoped file-permission rules are broken on Windows in current CC (GitHub #67849, OPEN).** Any `Read()/Edit()/Write()/Glob()/Grep()` rule with a path inside the parens (`Read(~/Evryn/Code/**)`) silently fails to match on Windows — the tool absolutizes the file path before the permission check, then fails to normalize it back to POSIX form for matching. **Only BARE tool names (no parens, no path) match reliably.** Same family as the 2026-06-29 `Bash(*)` regression, but broader than first understood: not just the `(*)` wildcard — *all* path-scoped file rules. (Corrects that entry's "path-pattern rules were unaffected" note — on Windows they ARE.) — [AC, research-confirmed]
- **Fix (confirmed working in a clean-session test): consolidated bare-form rules into the desktop USER-level `~/.claude/settings.json`.** Allow-list expanded from just `Bash` → `Read, Edit, Write, Glob, Grep, Bash, WebFetch, WebSearch` + Linear/Gmail MCP wildcards; the three Bash deny-rules (force-push, hard-reset, `rm -rf`) carried up so guardrails stay active machine-wide. User-level = applies to **every repo at once** → one fix clears the whole machine. Deliberately **not** in git → laptop untouched, and one clean revert point. Verified: full VS Code quit + reopen → zero prompts across repos. — [AC + Justin]
- **Closes the 2026-06-29 open item:** bare `WebFetch`/`WebSearch` were "still failing / unverified" at that session's end; today's clean-session test confirms the bare forms work — that was the deferred clean-session check. — [AC]
- **Also reverted:** the `additionalDirectories: ["~/Evryn/Code"]` Justin added to `evryn-team-workspace/.claude/settings.json` this morning (a fix attempt that did nothing) — removed before it ever landed in git, so it never reaches the laptop. - **Left intentionally (will matter at revert):** the project `.claude/settings.json` files in `_evryn-meta`, `evryn-ops`, `evryn-team-workspace` still carry the broken path-scoped / `(*)` forms. ## 2026-06-29 (AC — Claude Code `Bash(*)` permission regression diagnosed + fixed)

- **Root cause: the `Tool(*)` wildcard permission form silently broke in Claude Code 2.1.195.** `Bash(*)` and `WebFetch(*)` in `.claude/settings.json` stopped auto-allowing — every bash/webfetch command popped an approval prompt, despite the rules being present and unchanged for months. **Bare `Bash`/`WebFetch` (tool-name only) is the working form**, and the canonical one per the official docs (which say `Bash(*)` *should* be equivalent — so this is a regression, not a config error). Path-pattern rules (`Glob(~/Evryn/Code/**)` etc.) were unaffected; only the bare-`(*)` form broke. — [AC + Justin]
- **The detour (recorded because it cost a full session):** first misdiagnosed as the *new default-on Bash sandbox* (also shipped in 2.1.195) blocking network, then as a settings *trust-suspension* — both red herrings. **Justin's A/B cracked it:** `_evryn-meta` (on `Bash(*)`) prompted for everything; `evryn-dev-workspace` (on bare `Bash`) never did — same machine, one syntax difference. The `claude-code-guide` agent then confirmed bare `Bash` is canonical and the `(*)` break is a regression (not formally in the public changelog; GitHub issue #462 matches the symptom). — [Justin + AC]
- **Fix (commit `ebf74b4`, pushed from the desktop):** `.claude/settings.json` → `Bash(*)`→`Bash`, `WebFetch(*)`→`WebFetch`; the sandbox block (added while chasing the wrong theory) removed; the obsolete sandbox handoff doc deleted. Syncs via git → the laptop picks it up on next `git pull`. A user-level bare `Bash` was also added to the desktop's `~/.claude/settings.json` as a machine-local hedge (not in git). — [AC]
- **⚠️ Verification status (session end, 2026-06-29 evening) — the `WebFetch`/`WebSearch` fixes are NOT yet confirmed.** Bare `Bash` is **confirmed working**; `WebFetch` and `WebSearch` were **still failing at session end**. Most likely *this* session's project rules remained trust-suspended (only `Bash`, which also sits at the user level, was honored), making it an unreliable test bed — but that's unverified. Justin deferred the real check to a fresh session next day; today's environment was too churned to trust. The `(*)`→bare reasoning still holds for `Bash`; whether it fully fixes `WebFetch`/`WebSearch` is open until a clean-session test. — [Justin + AC]
- **Two gotchas worth keeping:** (1) **Editing a repo's `.claude/settings.json` re-trips that repo's trust-suspension** — ALL its rules (including the `Edit` rule) go inert until you re-approve in the CC UI, so fixing permissions inside a *live* session progressively degrades it; verify the fix in a *fresh* instance, not the one you edited in.
