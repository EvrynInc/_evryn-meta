### F1 — [Correctness / silent-failure] The 54b persisted-forward lookup is not origin-scoped
`verdict.ts persist54bForward` and `outcome.ts reattributeForwardOnMatch` both select the persisted forward by **`metadata->>emailmgr_item_id` ALONE** with `.limit(1)` and no `origin` filter. But **multiple** `messages` rows carry `metadata.emailmgr_item_id` for the same item: the submit_draft Slack notification (`origin:"submit_draft_notification"`), the `executeApproval` outbound audit, and the approval confirmation (`slack.ts`).
- **`reattributeForwardOnMatch` (record_outcome `matched`) runs POST-delivery**, when those rows already exist. `.limit(1)` (no `.order`) can return a non-forward row → it has no `metadata.intended_for` → the function returns `{reattributed:false}` and **silently skips the re-attribution**. The 54b forward then never enters the gatekeeper's history at match — the exact thing 54b exists to do — with no alert. **CONFIRMED silent defect.**
- **`persist54bForward`** is *currently* saved by call-ordering (the gold/edge verdict hook runs before the Slack notification in `submitDraftForApproval`; a pass has no competing rows at verdict-time) — but it's fragile (a re-ordering breaks it) and should be hardened for correctness-by-construction.
- **Fix:** add `.eq("metadata->>origin", "forward_persistence")` to BOTH lookups (the row `build54bForwardMessage` writes with `origin:"forward_persistence"`). Consider a `notifyDev` on the `matched` no-match branch so a genuinely-missing forward is visible, not silent.

### F2 — [Rebase regression] 54c real-name capture is lost at verdict-time reification
③ moves sender reification to verdict-time. `verdict.ts resolveContact` find-or-creates the contact with **`display_name: email.split("@")[0]`** (the email local-part) — it has no access to the forwarded `From:` header's display name.
