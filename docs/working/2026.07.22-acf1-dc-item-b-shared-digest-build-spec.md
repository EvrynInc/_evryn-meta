## Context (3 lines)

Today each agent stores its **own private copy** of every thread's one-line digest (`digest_lines(agent, thread_id, …)`, migration 008), regenerated **only when that agent itself replies** (`regenerateDigestLine`, `wakes/index.ts:482`). That's an N-way duplication, a stale-when-you're-silent bug, and a self-regen storm waiting to happen in a busy thread. Item (b) re-keys it to **one shared blurb per thread**, refreshed by a clock + on-read check, composed for **the rooms you're NOT currently in**.

## The build — eight pieces, ship as ONE change (see Constraint A)

### 1. New migration `016_thread_digests.sql` — the shared blurb + the thread registry

Create `public.thread_digests` (RLS-enabled, service-role-only, richly commented, idempotent — match the house style of 001–015):

- `thread_id text primary key`
- `blurb text` (nullable — a registry row can exist before a blurb is generated; that nullability is the point, see below)
- `updated_at timestamptz not null default now()` — the blurb's **last regeneration** time
- `last_activity_at timestamptz` — the thread's **newest message time**, maintained by trigger (below). This is consider (c)'s column and the load-bearing add.
- `state text not null default 'live' check (state in ('live','closing','dormant'))` — **three-value, NOT a boolean.** §4.4's atomicity invariant (item (e)) needs `closing`; a boolean cannot express it. Composition filters `state != 'dormant'`, which makes "a closing thread's digest stays visible" **structural** (the filter is the invariant), not a thing someone must remember.

**The insert-trigger:** an `after insert on agent_messages` trigger upserts `thread_digests(thread_id, last_activity_at) = (new.thread_id, new.created_at)` (on conflict, bump `last_activity_at` to the greater of stored/new). This makes `thread_digests` a **first-class thread registry** — the runtime currently has *no row anywhere that represents a thread* (a `thread_id` is a bare text column on `agent_messages`), and nearly every item-(a) defect traced to that gap. The registry collapses three needs into one indexed comparison: the on-read staleness check, the sweep's new-posts scan, and activity-keyed dormancy.

- **Do NOT parse thread-id prefixes in the trigger** (that reintroduces the Pattern-B magic-string registry the `ThreadKind` mechanism killed). Let the trigger create a row for any `thread_id`. Pseudo-threads (`wakeup:`/`checkin:`/`system:parked:`) will get registry rows, but they are **structurally excluded from all *digest* behavior** because that behavior is gated on `thread_subscriptions` existence (piece 6, the L2 rule) — and pseudo-threads have no subscription rows. Registry row = harmless; digest = only for real rooms.
- **Keep `digest_lines`** — do not drop it in this migration (soak, per §10 risk 7). Optional backfill of `thread_digests.last_activity_at` from existing data is fine but not required on DEV.
- **DEV DB first, Justin-gated.** Idempotent-additive.

### 2. L9 rider — add `thread_digests` to the boot-schema guard (SAME trip)

In `src/boot/schema.ts`, add `{ kind: 'table', name: 'thread_digests' }` to `EXPECTED_SCHEMA`. This is the standing rule from the L9 build: *"when item (b) adds `thread_digests`, add its query + the const together"* — so the boot guard refuses to boot until migration 016 is applied, enforcing migration-before-code order for the new table by construction.

### 3. Composer layer 9 rewrite — other rooms only, shared blurb + your headline

Rewrite the layer-9 read in `composer/index.ts` (currently `:212-221`, reads `digest_lines eq agent, eq dormant false`) and its render in `layers.ts` (`:317-321`):

- Join **the agent's `thread_subscriptions`** (its member/ever-member threads) → the shared `thread_digests` blurbs.
- **EVER-member visibility:** include subscriptions whose `dismissed_at` is **set** (a leaver keeps the digest until age-out). So do **not** filter `dismissed_at is null` here — that filter is for wake-gating, not digest visibility.
- Filter `thread_digests.state != 'dormant'` **AND `thread_id != wake.threadId`** — **never compose the current thread's own digest** (the structural loop-kill; the wake holds the current thread in full at layer 10, so its digest is redundant, and excluding it makes an in-thread wake structurally unable to trigger its own regen).
- Pair each other-room blurb with **the agent's own scratchpad headline** for that thread — wire in `scratchpadHeadline()` (`layers.ts:147`, built in item (a), currently **uncalled**). The blurb says what the thread IS; the headline says what it means to this agent. (This also makes L4 in the ledger — the `scratchpad_write` description's present-tense "shows beside this thread's shared digest blurb" — finally true.)
- Preserve the cache-layout invariants: layer 9 stays in the **volatile USER section**, nothing new above layer 6, no timestamps in the stable prefix, manifest stays diffable.

### 4. Delete `runWake`'s post-reply regen call

Delete the `regenerateDigestLine(...)` call at `wakes/index.ts:482`. A thread's digest is now maintained **only for people outside the room** — by the sweep (piece 6) and their on-read checks (piece 7) — never by activity inside it. **This deletion ships in the SAME change as pieces 6+7** (Constraint A) — never a commit where the old refresh is gone and the new one isn't in yet.

### 5. New regen — whole-thread input, `'system'` cost attribution

Rewrite `regenerateDigestLine` (`digest/index.ts:28`) into the shared-blurb regen:

- **Feed Haiku the WHOLE thread** (read the thread's `agent_messages`), not the previous-line + delta. Today's prev-line+delta is a summary-of-a-summary that drifts indefinitely; whole-thread is generated from source every time — cheaper, loop-proof, **and more faithful.**
- Write the result to `thread_digests.blurb` + bump `updated_at` (keyed on `thread_id`, shared — not per-agent).
- **Cost attribution: `cost_events.agent = 'system'` for BOTH regen paths** (sweep and on-read) — consider (a). A shared-blurb regen is fleet infrastructure, not one agent's work; attributing it to whoever-woke-first-into-a-stale-room is luck, not behavior, and would be noise in the per-agent spend signal Emma watches. `cost_events.agent` is `text NOT NULL` (migration 005), so `'system'` is a valid actor string; no schema change.

### 6. Background 30-min sweep

Add a periodic sweep (the scheduler `cycle()` runs every 15s — gate the sweep to ~30-min intervals via a `lastSweepAt` tracker or equivalent; don't run it every cycle):

- Scan `thread_digests` where `last_activity_at > updated_at` (new posts since last regen) **AND `state != 'dormant'` AND the thread has ≥1 `thread_subscriptions` row** (the **L2 rule**: only real rooms get digests; pseudo-threads never do). Regen each (piece 5). Quiet threads (no new activity) cost nothing — the scan is a timestamp comparison, no LLM.

### 7. On-read staleness check at compose

In the compose path, for the agent's **other** member-threads:

- Compare `last_activity_at` vs `updated_at` (deterministic SQL, no model). Fresh → use as-is (the common case; the sweep keeps up). Stale → regen those threads **now**, in the compose path.
- **Regen in PARALLEL** across stale threads (one Haiku of wall-clock, not N).
- **In-flight dedup = double-checked staleness INSIDE the per-thread lock.** Use `withKeyLock` (`db/keyed-mutex.ts`) per thread, and **re-read the digest's staleness once inside the lock**: by then the first caller has refreshed it, so the second reads not-stale and **skips**. ⚠ Do NOT rely on `withKeyLock` to "await the in-flight result" — it *serializes* and then runs the second caller's callback anyway (verified in the source); the in-lock re-check is what actually makes it one regen instead of two. The sweep's regen and a compose-path regen for the same thread must contend on the **same** per-thread key.
- **~2-min per-thread floor:** never regen the same thread more than once per ~2 min (a seatbelt for pathological bursts).
- **Degrade LOUDLY:** on regen failure, compose the **stale blurb + an explicit marker** ("⚠ stale: N messages since this summary; refresh failed — pull the thread if it matters"). Never block the wake on a digest; never let the agent silently trust a stale line.
- **Where this lives:** `composeWake` is today pure-read + one manifest insert; adding an awaited, lock-guarded, failure-tolerant Haiku call changes its character. Decide the clean home with DC (a pre-compose step in the dispatch path vs. inside the composer) — flag if anything else assumes `composeWake` is fast/non-LLM.

### 8. Activity-keyed dormancy

Rewrite `sweepDormantThreads` (`digest/index.ts:131`, called `scheduler/index.ts:324`) to key on **real thread activity** — `thread_digests.last_activity_at < cutoff` (7 days) → set `state = 'dormant'` — **NEVER on digest-regen timestamps.** Today's sweep keys on `digest_lines.updated_at` (the *blurb's* last regen), which under v3 is more wrong than ever: a live-but-unread thread's digest can legitimately sit old for a stretch, so the old keying would age out **live** threads (and, once item (e) lands, fire "this thread ended" memories into live conversations). This is a **correctness requirement inside item (b)**, not a separate step. *(Item (e) later inserts the two-phase `closing`→`dormant` + the per-ever-member memory-write before `dormant`; item (b) just needs dormancy keyed on real activity and the three-value `state` in place.)*

## Constraints & traps (each is a QC check)

- **A — ATOMIC SHIP.** Pieces 4 (delete the post-reply call) + 6 (sweep) + 7 (on-read) land in **one change**. A top-down build that lands the deletion first opens a window where nothing refreshes any digest. Hard requirement.
- **B — L2:** the sweep and all digest behavior consider only real rooms (≥1 subscription row); pseudo-threads never get digests. Enforce via subscription-existence, **never** by parsing thread-id prefixes.
- **C — double-checked staleness INSIDE the lock** (piece 7) — not `withKeyLock`-as-await. This is the exact mechanic the design-of-record was corrected on.
- **D — whole-thread regen** (piece 5), never prev-line+delta.
- **E — `'system'` cost for both regen paths** (piece 5).
- **F — ever-member visibility** includes dismissed subscriptions (piece 3).
- **G — cache layout preserved:** nothing new above layer 6; no timestamp/counter in the stable system prefix; every wake manifest stays diffable (design-of-record §6.4, ARCHITECTURE invariant 2).
- **H — migration discipline:** idempotent-additive; DEV DB first, Justin-gated; keep `digest_lines` until soak; add `thread_digests` to `EXPECTED_SCHEMA` same trip (piece 2).

## Test discipline (the lane's hard-won rule — put it in the DC brief)

**Every new guard test must be watched FAILING against deliberately-broken real code before it is believed.** A test in this codebase will, by default, compare a thing to itself (it happened three times on item (a)). Concretely for item (b): a test that asserts "the current thread's digest is never composed" must be shown to FAIL when the exclusion is removed; a test for the in-lock double-check must FAIL when the re-check is dropped (proving it catches the two-regen case); the L2 exclusion test must FAIL when the subscription gate is removed. Typecheck strict-green + full suite green is necessary, not sufficient.

## Open design calls (ACf1's, within-lane — noted for visibility, not blocking)

1. **Table name.** The design-of-record calls it `thread_digests`; ACf's consider (c) suggested "naming the table for what it becomes" (a registry). **Lean: keep `thread_digests`** — every doc/comment/§ references that name, and renaming ripples for a cosmetic gain; the registry role is carried by the `last_activity_at` column + comments, not the name. (Reversible; flag if QC or Justin prefers `threads`.)
2. **Backfill.** Optional backfill of `thread_digests` from existing `digest_lines` rows on DEV. **Lean: skip** — DEV data is disposable and the sweep repopulates naturally; not worth the migration complexity.
