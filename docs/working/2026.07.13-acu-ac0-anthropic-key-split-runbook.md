## 1. Why we're doing this (the problem the split solves)

Evryn's **production backend** (`evryn-backend`) authenticates to Anthropic with an API key in the env var **`ANTHROPIC_API_KEY`** (required at boot — `src/config.ts` line ~318, `anthropicApiKey: required("ANTHROPIC_API_KEY")`). Justin confirmed (2026-07-13) that this key currently draws on the **same Anthropic account/billing** that AC/DC and other dev-side API work use. That shared arrangement has three costs:

1. **Blended billing** — prod runtime spend (the real product) and dev/experimentation spend land on one invoice line, so you can't cleanly see "what does running Evryn-for-Mark actually cost per month?"
2. **Shared blast radius** — one key leaking, or one runaway loop, threatens both prod and dev at once. A per-product key limits the damage to that product.
3. **It muddies the spend-cap picture** — the M1 circuit-breaker plus the Anthropic-console monthly ceiling (see §3) are supposed to be a *per-product* backstop. When the key is shared, a console-level cap can't distinguish prod runtime from dev noise, and the per-event `llm_usage` cost data (ADR-038) is the only clean per-product signal we have.

**The goal:** give the production backend its **own** Anthropic API key, scoped to its **own** Anthropic **workspace**, with its **own** monthly spend limit — so prod billing, blast radius, and spend caps are cleanly isolated from dev/team work.

## 3. The monthly spend ceiling — where it lives (console-verify, not guessed)

The operator-guide (`evryn-backend/docs/operator-guide.md`, "Two one-time setups") says: *"Anthropic monthly spend ceiling — at console.anthropic.com, set a monthly cap. This is the code-independent final backstop."* The docs **do not state whether the ceiling Justin set pre-live is account-level (org-wide) or workspace/key-level.** Do NOT guess — **this is a console-verify step for Justin** (§4, step J1).

**What the Anthropic console supports today** (confirmed via web search, current as of April 2026 — Justin should re-confirm in the live console, capabilities move):
- **Workspaces** exist and each can carry its **own monthly spend limit** and rate limits, set **below** the org limit (they can't exceed it).
- **Per-workspace API keys**: keys are created *inside* a workspace, so a key inherits its workspace's limits.
- Spend limits are **console-UI only** — there is no Admin API endpoint to set them programmatically (the Rate Limits API, shipped 2026-04-24, only *reads* limits).
- The **Claude Code workspace** additionally supports *per-user* monthly limits (not relevant to an API-key runtime, but good to know).

**Implication for the split:** the clean shape is a **dedicated "Evryn Production" workspace** with (a) its own monthly spend limit and (b) a key minted inside it. That gives per-product billing + a per-product console backstop in one move. Whether Justin's *existing* pre-live ceiling is org-level (stays as an umbrella) or was already set on a workspace determines only whether step J1 is "verify + leave" or "verify + move" — it does not change the rest of the runbook.

## 4. The runbook (ordered steps, each with an owner)

**Owner tags:** **[JUSTIN]** = console/click or local-env action only Justin can do · **[AC0]** = env/coordination/doc work · **[DC]** = code (none required here — this is env-only, no `src/` change) · **[VERIFY]** = post-change check.

### Pre-flight (before touching anything)

- **[AC0] P1 — Confirm no team-runtime interaction.** Re-confirm (this doc, §2) that `evryn-team-runtime` is subscription-auth and unaffected. Nothing to change there. State this to Justin at dispatch so he knows the blast radius is `evryn-backend` only.
- **[AC0] P2 — Timing gate (DO NOT skip).** This split is an **env-var change that rides a `railway up`** (§ step A2). **Do NOT run the deploy step while another agent's in-flight backend work is mid-merge or mid-deploy** — a key-swap deploy must not collide with, or silently carry, someone else's unmerged branch. Check `docs/current-state.md` + the deploy-log for an active wave; if a cost-lane / security-lane deploy is imminent, either (a) piggyback the env change onto **that** deploy (preferred — see A2), or (b) wait for a clean window. Coordinate through Justin.
- **[AC0] P3 — Confirm local vs. Railway inventory of `ANTHROPIC_API_KEY`.** The var lives in: **Railway prod** (the production service), each **dev's local `.env`**, and *possibly* a **Railway staging service** IF one exists separate from prod. As of the deploy-log, backend deploys are a single prod service; staging is an `EVRYN_ENV=staging`-mode run against a dedicated DB (ADR-045), not necessarily a standing separate Railway service. **[JUSTIN] to confirm** whether a standing staging Railway service exists — if yes, its `ANTHROPIC_API_KEY` is a second Railway target below; if no, skip staging.

### Console setup (Justin, ~10 min)

- **[JUSTIN] J1 — Verify where the current spend ceiling lives.** In console.anthropic.com, check whether the pre-live monthly cap is set at **org (account) level** or on a **workspace**. Note which — it decides J2's umbrella logic. (This is the console-verify from §3.)
- **[JUSTIN] J2 — Create an "Evryn Production" workspace.** In the console, create a workspace dedicated to the prod backend. Set its **monthly spend limit** to the prod-runtime ceiling (whatever the pre-live number was — carry it here; if the current cap was org-level, keep the org cap as the umbrella and set the workspace cap at or below it).
- **[JUSTIN] J3 — Mint a new API key *inside* the Evryn Production workspace.** Name it clearly (e.g. `evryn-backend-prod`). **Copy the value into Railway directly (step J4) — do not paste it into chat, a doc, a ping, or a commit.** Refer to it only as "the new prod key" everywhere else.
- **[JUSTIN] J4 — Set the new key on Railway prod.** In the Railway prod service's variables, replace the value of **`ANTHROPIC_API_KEY`** with the new prod-workspace key. **The var NAME does not change** — only its value. (If a standing staging Railway service exists per P3, decide whether staging gets the same prod key, its own staging-workspace key, or the old dev key — recommended: staging stays on the **dev/team** key so staging spend doesn't hit the prod workspace cap.)
- **[JUSTIN] J5 — (optional, hygiene) Rotate the old key's role.** The old `ANTHROPIC_API_KEY` value now belongs to "dev/team" only. Optionally rename/relabel it in the console as `dev-team` so the two are never confused later. Do NOT revoke it — AC/DC local `.env` files still use it.

### Deploy coordination (the env change must ride a deploy)

- **[AC0 + JUSTIN] A1 — Why a deploy is required.** A Railway env-var change **does not take effect on the running process until the service restarts/redeploys** — Railway applies new variable values on the next deployment. So J4 alone doesn't flip the live key; it's staged until a `railway up` (or a Railway-triggered redeploy) picks it up. (This is the same "env changes can't skip a deploy" pattern the deploy-log operates under — one `railway up` = one `v0.2.N` bump.)
- **[AC0 + JUSTIN] A2 — Ride-along, don't solo-deploy (preferred).** Because this is a **value-only** change to an existing var (no code, no migration), the cheapest path is to let it **ride the next scheduled `v0.2.N` deploy** (e.g. the AC2 ③ runtime-bookkeeping ship that's next in the deploy-log). The env value is already staged in Railway from J4; the next `railway up` boots on the new key with zero extra risk. **If no deploy is imminent** and Justin wants the split live now, do a **standalone `railway up`** from `evryn-backend/` per the operator-guide deploy steps — it bumps its own `v0.2.N` and tags the commit. Either way: **one deploy, on Justin's explicit go.**
- **[AC0] A3 — Assign the deploy number + note it.** Whichever deploy carries the flip, record it in `docs/deploy-log.md` as the `v0.2.N` that "swapped `ANTHROPIC_API_KEY` to the dedicated Evryn-Production-workspace key." Write the deploy-log row + CHANGELOG entry (deploy-log/CHANGELOG/ADR are the excluded-from-approval docs — AC may edit; but **still get Justin's go before the commit**, per commit discipline).

### Verify (after the deploy boots)

- **[VERIFY] V1 — Clean boot.** Confirm the deployment reaches SUCCESS and boots clean (Supabase / Slack / polling up) via `railway logs --since 10m` (or the Railway dashboard). A **bad key surfaces as boot success but failing Anthropic calls** — so V2 matters more than V1.
- **[VERIFY] V2 — A real Anthropic call succeeds on the new key.** The most direct proof: after boot, confirm a live triage actually completed (a forwarded email got triaged, or the proactive cron fired) — i.e. Evryn's tool-calling path hit Anthropic and got a real response. A GO that only proves "the process started" is not enough here; the thing under test is *auth against the new key*. Check `#dev-alerts` for triage activity, or watch `llm_usage` for a fresh row (see V3).
- **[VERIFY] V3 — `llm_usage` is still capturing.** Query the `llm_usage` table (ADR-038 per-event cost capture) for rows with `created_at` **after** the deploy — a new row proves the runtime is authenticating, calling, and recording spend under the new key. This is also the per-product billing signal the split exists to sharpen. (The DB-side capture is key-agnostic, so it should keep working — V3 confirms it.)
- **[VERIFY] V4 — Console: prod spend now lands in the Evryn-Production workspace.** After a day of real traffic, confirm in the console that prod usage is accruing against the new workspace's limit (and dev/team usage against the old key), proving the billing split is clean.

### Rollback

- **[JUSTIN] R1 — Instant rollback = restore the old key value in Railway prod.** If the new key misbehaves (auth failures, wrong workspace, a mis-set spend cap that throttles prod), set **`ANTHROPIC_API_KEY`** back to the old dev/team value in Railway prod and redeploy (or `railway up`). Because the split is value-only, rollback is a one-variable revert — no code to undo. The old key stays valid (J5 said don't revoke), so this is always available.
- **[AC0] R2 — Note the rollback.** If rolled back, record it in the deploy-log so the next instance doesn't think the split shipped when it didn't (the "intent ≠ proof it shipped" discipline).

## 5. Steps that MUST NOT run while other agents' work is mid-flight

Called out explicitly (from P2, restated as a hard gate):

- **The `railway up` that carries the key swap (A2)** must not fire while another agent's backend branch is mid-merge or an unrelated `v0.2.N` deploy is in progress. A prod deploy picks up **all** currently-staged Railway env changes and the current `main` tree — so an ill-timed deploy could carry someone else's half-finished work or a stale tree. **Coordinate the deploy window through Justin.**
- **J4 (setting the new value in Railway prod)** is safe to stage anytime — it doesn't take effect until a deploy (A1). But once it's staged, **any** `railway up` by any agent will flip the live key. So either stage J4 *and* deploy in the same coordinated window, or tell every agent that a key-swap is staged so nobody deploys unaware. **Preferred: stage J4 and deploy in one window.**
