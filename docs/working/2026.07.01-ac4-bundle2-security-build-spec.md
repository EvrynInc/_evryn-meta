## 1. What this is + why (the bulkhead frame)

Evryn has two **generic** database tools, `supabase_read` and `supabase_upsert`, in `src/triage/classify.ts`. They run as `service_role` â€” which **bypasses Postgres RLS** â€” and accept an arbitrary `table` + `filters` + `record`. The per-user firewall ("only *this* user's data is loaded into her context") protects the *normal, typed* pathways; but this generic tool is a **door around that firewall**: a prompt-injected Evryn could call `supabase_read({ table: 'users', select: '*' })` and RLS would not stop it â€” a cross-user bulk dump.

Bundle 2 adds an **inner door** on the **read** side: even if an attacker breaches Evryn's judgment (injection), the generic read can't be turned into a bulk / cross-user dump. This is the v0.2 *interim* â€” the full structural fix (an RLS-respecting role, or typed read tools replacing the generic surface) stays v0.3 (**S1**).

**Bundle 2 is READ-side ONLY. Do NOT touch `supabase_upsert`.** The raw *write* surface is removed separately by AC2's **Step 57** (runtime-does-bookkeeping), which deploys *after* Bundle 2. The two compose into one agreed end-state (below); they do not overlap. Touching `supabase_upsert` here would collide with Step 57 â€” leave it exactly as it is.

**The agreed end-state (AC0 brokered this with AC2 â€” build to it):** after Step 57 **+** this Bundle 2, Evryn's DB surface =
- **{typed tools}** â€” `create_user`, `record_pass`, `append_pending_note`, `append_cross_user_note`, `register_gatekeeper_address`, + the operator-gated `set_thread_scope` / `rescope_messages` / `redact_user_from_message`;

## 2. The policy to build (exact)

Add a **pure policy function** that the `supabase_read` handler consults before it runs the query. Mirror the shape of Bundle 1's `evaluateWebToolCall` in `web-guard.ts` (a pure, total, never-throwing core + a thin handler that wires it) â€” see Â§4 for the reference file.

**Signature (adjust names to fit the file's style, keep the shape):**
```
evaluateSupabaseRead(args: { table: string; filters: Record<string, string> | null; limit?: number })
  â†’ { decision: "allow"; effectiveLimit: number }
  | { decision: "deny"; reason: string }
```

**Rules (all four):**

1. **Table allow-list.** `supabase_read` may read ONLY these tables; anything else â†’ **deny** (reason names the table):
   - `users`, `messages`, `emailmgr_items`, `gatekeeper_inbound_addresses`.
   - Deliberately EXCLUDED (deny): `llm_usage`, `notify_queue`, `evryn_knowledge`, `story_versions`, and any unknown/unlisted table. (Rationale: llm_usage/notify_queue = ops/cost, no conversational need; evryn_knowledge = loaded by slug elsewhere; story_versions = the Reflection archive, not conversational. If a legit need for one surfaces later, it's a one-line allow-list add + a note to AC4 â€” do NOT add speculatively now.)
   - The allow-list lives as a named `const` set (single source of truth), env-overridable is NOT required (keep it simple + explicit).

2. **`users` requires a row-scoping filter â€” the load-bearing anti-dump rule.** A read of `users` MUST include a `filters` object carrying an equality on **`id` OR `email`**. An unfiltered `users` read, a `users` read with `filters === null`, or a `users` read filtered ONLY on a non-identifying column â†’ **deny** (reason: "users read requires an id or email filter â€” no unfiltered/bulk reads"). This is the exact cross-user-dump vector (`select:* from users`), so it's the rule that matters most; get it exactly right. (Reading a single user's full row *by id/email* â€” `select:*` included â€” is legit and must be ALLOWED; the firewall concern is *breadth of rows*, not columns, at v0.2. Column-level constraints are v0.3.)

3. **Hard row-cap.** Cap the effective row limit at **50** regardless of what Evryn passes. If she passes `limit > 50`, clamp to 50 (and note it in the log). If she passes no limit, apply 50 as the default cap. Return `effectiveLimit` from the policy; the handler applies it via `.limit(effectiveLimit)`. This bounds a bulk read on ANY allow-listed table (belt-and-suspenders behind the users-filter rule â€” see Â§6 flag on messages/emailmgr_items).

4. **Log every call + deny-with-reason + alert-on-deny.**
   - **Log every read** (allow AND deny) as structured JSON to stdout â€” mirror `web-guard.ts`'s `console.log(JSON.stringify({...}))` and `classify.ts`'s existing `logToolCall`: include `type: "supabase_read_guard"`, timestamp, table, a truncated view of filters, decision, reason, effectiveLimit, and the pathway/scope from `toolLogContext`. So the door is *observed*, not silent.
   - **On deny:** return the reason to Evryn in the tool result (she SEES the refusal and can react â€” never a silent empty result), AND fire `notifyDev` (best-effort, swallowed) â€” an attempted bulk/cross-user read is signal we want to see, not bury. (This mirrors AC0's Bundle-1 add: `notifyDev` on a web-guard deny.)
   - **On allow:** proceed with the clamped `effectiveLimit`.

**Fail-closed:** the policy fn is **pure + total â€” it NEVER throws.** Malformed/unparseable input (e.g. `filters` that isn't an object) â†’ treat as no usable scoping filter â†’ **deny** for `users` (no id/email), and for other allow-listed tables allow-with-cap (they're not the id-scoped case) â€” but if in doubt, deny. Wrap the handler's use of the policy so that if anything unexpected throws, the read is denied + logged, not allowed. Over-block beats under-block on a security door.

## 2a. Post-QC amendment (2026-07-01) â€” the `select` embed door + filter-value tightening

> **Added after QC's adversarial review found the `select` vector CONFIRMED-EXPLOITABLE against the FK graph (NO-GO blocker).** This **supersedes** Â§2.2's parenthetical that a bare `select:*` is fine: bare `*` and plain column lists remain fine; a resource-EMBED select does NOT. AC4 owns this refinement (QC asked AC to confirm it â€” confirmed).

5. **`select` MUST NOT carry a resource-embed â€” the load-bearing addition.** The original guard validated `table` + `filters` + `limit` but passed `args.select` through raw into `db.from(table).select(args.select ?? "*")`. PostgREST **resource embedding** via `select` (e.g. `select=*, related_table(*)`, `select=*, alias:users!fk(*)`) reaches FK-related tables in one query â€” and because the runtime queries as `service_role` (RLS-bypassing), an embed reaches:
   - **non-allow-listed tables** â€” `supabase_read({ table:'emailmgr_items', select:'*, llm_usage(*)' })` passes the allow-list (top-level table = `emailmgr_items`) yet reaches the deliberately-excluded `llm_usage` (FK `llm_usage.emailmgr_item_id`);
   - **other users' rows** â€” `supabase_read({ table:'users', filters:{id:'<self>'}, select:'*, referrer(*)' })` passes the allow-list AND the id-filter rule yet pulls a *different* user's full row via the `users.referrer â†’ users.id` self-FK (defeating the anti-dump rule);
   - and the **row-cap does not bound embedded rows** â€” PostgREST `.limit(n)` bounds only top-level rows, not the joined/embedded set.

   So `select` is a cross-table / cross-user dump vector that defeats both the allow-list and the cap. **Fix (deny-by-shape â€” same spirit as the rest of the guard):** pass `select` into `evaluateSupabaseRead`, and **DENY any `select` that carries embed syntax.** A PostgREST embed **always introduces a parenthesis** (`table(cols)`) â€” the relationship forms (`alias:users!fk(*)`, `...spread(*)`) all still contain `(` â€” so **the presence of `(` in `select` is the reliable, complete embed signal**: deny on `(`. A plain column list (`id,email,status`), a bare `*`, or an omitted `select` has no `(` â†’ allow. Deny-with-reason: *"embedded/related-table selects aren't allowed â€” read one table at a time."* This closes the cross-table reach while preserving every legit v0.2 read (the operator `messages`-by-`scope_user_id` pattern uses `*` or a plain column list â€” no parens). *(Column-LEVEL select allow/deny stays v0.3 per Â§7 â€” this rule is specifically the cross-TABLE embed reach, a v0.2 cross-user-dump vector, not a column-hygiene nicety.)*

6. **`users` filter must map to a non-empty, non-null value (tightening â€” QC NB2).** `filtersScopeUsersRead` originally checked key PRESENCE only, so `{id:null}` / `{id:''}` returned `allow` (fails safe *today* â€” `.eq('id',null)` matches no rows â€” but it's latent looseness a future refactor could turn load-bearing). Require the `id`/`email` key to map to a **non-empty, non-null primitive** (string/number). Keep it pure/total (a non-primitive value â†’ treat as no usable scoping â†’ deny for `users`).

**Test additions (fold into Â§5):** `users + {id} + select:'*, referrer(*)' â†’ deny`; `emailmgr_items + select:'*, llm_usage(*)' â†’ deny`; `messages + select:'*, sender:users!sender_id(*)' â†’ deny`; legit `select:'*'` / `select:'id,email'` / omitted `select` â†’ allow; and (NB2) `users + {id:null}` / `{id:''}` â†’ **deny at the decision level** (not merely no-throw).
