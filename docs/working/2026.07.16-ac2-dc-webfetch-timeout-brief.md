## 1. The problem (verified against the runtime)

`runEvrynQuery` (`src/triage/classify.ts`) drives Evryn's triage via `for await (const message of query({...}))`. Its options carry `maxTurns: 20` but **NO wall-clock / abort timeout.** The built-in `WebFetch`/`WebSearch` tools are gated only by the shape-based `web-guard` PreToolUse hook (allow/deny — no timeout). So a **WebFetch to a slow/unresponsive host that never returns** blocks the `for await` loop **forever**.

Because the poll pipeline is **single-threaded and fully awaited** — `startPolling → pollOnce → planPollBatch → handle(email) → handleNewEmail → processEmail → runForwardTriage → runEvrynQuery` — a single hung fetch **wedges the entire poll loop.** No new mail is processed. And the existing silent-death guards do NOT catch it:
- The **polling-dead escalation** (`safety/polling-dead.ts`) keys on `consecutiveErrors` — a HANG throws no error, so it never increments, so that escalation never fires.
- The **Healthchecks heartbeat** is pinged only *after a successful `pollOnce`* — a wedged loop stops pinging → Healthchecks alarms after ~15 min → **manual Railway restart** is the only recovery.

**Real Mark-live risk:** Evryn web-researches essentially every real sender at triage. One dead/slow domain in one forward wedges the whole pipeline until a human restarts it. (Observed live: the Gate-A a12 fixture hung `runEvrynQuery` ~27 min on a fetch before AC2 killed it.)

## 2. The fix — an AbortController wall-clock cap on `runEvrynQuery`

Add a wall-clock budget to `runEvrynQuery`'s `query()` call via the SDK's **`abortController`** option (confirmed supported: `@anthropic-ai/claude-agent-sdk/sdk.d.ts` — `abortController?: AbortController` in the query Options). On expiry, **abort the query** so the underlying Claude Code subprocess + the hung fetch are **terminated** (not just ignored), the `query()` iterator rejects, and control returns to `runEvrynQuery`'s existing `catch` → the item follows the **normal error/retry path** instead of wedging the loop.

**Why AbortController, NOT a bare `Promise.race`/`withTimeout` (the refinement over "mirror the Haiku screen"):** the Haiku screen's `withTimeout` (`haiku-screen.ts`) uses `Promise.race`, which on timeout *ignores* the still-pending `query()` — leaving the underlying subprocess running. That's fine for the Haiku screen (`maxTurns:1`, no tools, finishes fast). For `runEvrynQuery` (20 turns, tools, web fetches) a `Promise.race` would leave a **hung Claude Code subprocess zombie per timeout** — a resource leak that compounds. `abortController.abort()` actually kills it. So: use the AbortController, not `Promise.race`.

### Shape (illustrative — DC owns the exact code)
```ts
const ac = new AbortController();
const timer = setTimeout(() => ac.abort(), QUERY_TIMEOUT_MS);
try {
  for await (const message of query({ prompt, options: { /* …existing… */, abortController: ac } })) {
    // …unchanged…
  }
  // …existing usage capture…
} catch (error) {
  // …existing classifyQueryError handling… (see §3 — verify the abort branch)
} finally {
  clearTimeout(timer);
}
```

### The timeout budget
- **Env-tunable:** `EVRYN_QUERY_TIMEOUT_MS` (parse in `config.ts`, mirror the existing `parse*` helpers; default to a generous few minutes). **Recommended default: 180000 (3 min)** — comfortably above a legit multi-turn Opus triage with several web searches (the Gate-A gold/edge fixtures each completed in ~1–2 min with real WebSearch/WebFetch), and far below "wedged forever." **Confirm the default with AC2/AC0 before shipping** (too tight = false-aborting a legit deep triage; too loose = a longer per-item stall on a hang). It applies to the whole `runEvrynQuery`, so it also caps the rare legit slow triage — that's acceptable (an aborted item errors + retries; a legit triage that genuinely needs >3 min is vanishingly rare and better retried than allowed to wedge).
- This is a **whole-query** cap, deliberately — a per-`WebFetch` timeout is NOT cleanly achievable (the built-in WebFetch tool exposes no timeout via the SDK/hook; the PreToolUse hook can only allow/deny, not time-bound a tool). The whole-query cap catches ANY hang (fetch or otherwise) and is the achievable, robust fix. (Note this in the ADR — it's the "which is achievable" answer AC0 asked for.)

## 3. Interactions to get right (DC must verify each)

1. **The abort error must classify to the NORMAL ERROR/RETRY path, NOT a breaker halt.** `runEvrynQuery`'s `catch` runs `classifyQueryError(error)` (`src/safety/errors.ts` — **read it**). An abort error (AbortController → typically an `AbortError`/`DOMException`, or an SDK abort rejection) is neither a billing nor an auth error, so it *should* fall to the `"rethrow"` action → re-throws → `processForward`/`processDirect`'s catch stamps the item `'error'` → returns `"completed"` → **poll loop continues** (exactly the goal). **Verify** `classifyQueryError` doesn't mis-bucket the abort. If it does (or to make it explicit + observable), add a dedicated timeout branch that: (a) fires ONE `notifyDev` (`"runEvrynQuery aborted after {ms}ms — likely a hung web fetch on {pathway}/{item}; item errored, will retry"`) so a recurring hang pattern is VISIBLE, and (b) re-throws so the item errors + retries. Do **NOT** route a timeout into `haltForBreaker` — a hang is a slow external dependency, not a runaway; it must not trip the M1 breaker or park like a halt.
2. **M1 breaker gate is untouched.** The breaker check is at the TOP of `runEvrynQuery`, *before* `query()`; the timeout wraps the `query()` consumption only. A tripped breaker still short-circuits first (zero spend); the timeout only governs a query that actually runs. No change to the halt/park semantics.
3. **`llm_usage` capture:** on an abort, the `result` message carrying usage likely never arrives → `capturedUsage` stays undefined → the existing best-effort `recordLlmUsage` writes a row with null token fields (or is skipped) — same as the existing mid-stream-throw limitation (QC NB1). Fine; no new work.
4. **The `finally { clearTimeout(timer) }`** must run on every path (success, error, abort) so a completed query doesn't leave a dangling timer that aborts nothing.
5. **All other `query()` call sites:** `runEvrynQuery` is the money-path one that wedges the poll loop — that's the target. The **Haiku screen** already has its own 10s `withTimeout` (leave it). The **Reflection consolidation** and any cron pathway also go through `runEvrynQuery` (they call it), so they inherit the cap for free — good. No other raw `query()` call needs this (grep to confirm: only `classify.ts runEvrynQuery` + `haiku-screen.ts defaultRunQuery` call `query()` directly).
