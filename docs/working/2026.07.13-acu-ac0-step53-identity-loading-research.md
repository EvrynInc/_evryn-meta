## 0. TL;DR for the build agent

- **What "reliable loading" means here:** Evryn is *supposed* to pull an identity module on demand via her `read_identity_module` tool when a situation calls for it. In the 2026-05-28 integration test she **repeatedly skipped** modules she should have loaded (and worse — reported she'd loaded things she hadn't; see §2). The 2026-05-29 fix ("force-load") sidesteps the skip by jamming *every* non-operator module into the systemPrompt on every call. That works, but it's a band-aid that also drives the composition-scaling problem (§1, §3).
- **The two problems are one:** if on-demand loading were reliable, force-load would be unnecessary, and the growing always-loaded block would shrink back down.
- **The genuine open tension** (Justin's, carried in the sprint): Anthropic prompt caching may *invert* ADR-012's token-economy premise — a byte-identical force-loaded prefix is cheaper at scale than selective loading, because cache reads run ~10% of base and every user reinforces the same prefix. So the honest question isn't just "how do we make loading reliable" — it's **"do we even want selective loading back, or is force-load-plus-caching the right standing v0.3 shape?"** The measurement that answers this (Step 69) has **not run yet** — see §6, and it gates the whole recommendation.
- **My recommendation, in one line:** build the **test harness first** (it's the reusable, no-regret deliverable and unblocks everything), then adopt a **verified-composition model** — keep force-load as the reliability floor but make it *structural and asserted* (a per-pathway manifest the runtime force-composes + a runtime canary that fails loudly if a required module is missing), and treat "return to selective loading" as **measurement-gated and probably deferred**, not the default. Full reasoning in §5.

## 1. What Step 53 is (from the sprint) and what it amends

From `evryn-backend/docs/SPRINT-V0.2-HARDENING.md` Step 53 ("Make Evryn reliably load the right identity module, and stop the identity stack from growing without bound"). Two linked problems, verbatim intent:

1. **Loading reliability.** Evryn should pull a module on demand (`read_identity_module`) but skipped modules she should have loaded (2026-05-28 test). Current workaround = force-load (`loadCommonPrefix` in `src/triage/classify.ts`). Band-aid: she still doesn't reliably load *on her own*.
2. **Composition scaling.** Force-load puts everything in the always-loaded block, which keeps growing (`triage.md` alone ~154 lines — actually **168** lines as of this read; sent on *every* query). Prompt caching keeps the dollar cost modest but the floor only rises.

The step **amends [ADR-012](_evryn-meta/docs/decisions/012-trigger-composed-identity.md)** (trigger-composed identity — the decision that set up how identity files compose into her prompt). ADR-012's original premise: on-demand selective loading saves tokens and keeps operator/onboarding out of contexts that don't need them. Force-load already deviates from that; Step 53 decides whether the deviation becomes the standing design.

**The sprint's own "where to start" (do not skip — it's the assignment):**
- (a) research prior art — "get an LLM to reliably load the right capability at the right moment" is well-studied; the answer may be mostly research.
- (b) evaluate the **Agent SDK Skills framework** as the loading mechanism — ADR-012 chose plain docs over Skills; revisit if Skills loads more reliably.
- (c) build a **fast automated test harness** — drive `runEvrynQuery` with fixture prompts and *assert which modules/tools Evryn actually loaded*, reading the tool-call log the runtime already emits (`logToolCall`, `classify.ts`). The AC needs to iterate dozens of times a day, not by hand-emailing Evryn.
- **Deliverable (one focused day):** a rock-solid load recipe, a docs-vs-Skills recommendation, and the harness as a reusable regression check.

**Neighbors (checked for interaction):**
- **Step 50** — Identity/runtime dedup review: compare what identity files *say* vs what the runtime actually exposes; remove operational redundancy, reclaim tokens for voice/judgment. *Interaction:* Step 50 shrinks the modules; Step 53 decides how they load. If Step 53 lands selective loading, Step 50's token savings matter less; if force-load stays, Step 50's dedup is the *only* lever on the always-loaded floor. They should be sequenced with awareness of each other — arguably Step 50 informs Step 53's cost math.
- **Step 51** — Compulsion audit across `runEvrynQuery` prompts + tool descriptions (ADR-033): each candidate → keep / make-structurally-impossible / convert-to-permission-language. *Interaction:* `read_identity_module`'s own tool description (§2) is a candidate — its current "you rarely need to call it" wording is exactly the kind of instruction Step 51 would examine. And the "load everything every turn" standing instruction that patched the 2026-05-28 test is a compulsion-vs-structure question. Co-awareness helps; not a hard dependency.

## 2. Current actual behavior (runtime, with file:line)

All references are `evryn-backend/src/triage/classify.ts` unless noted, read against `main` on 2026-07-13.

### 2a. The composition primitives

- **`composeSystemPrompt(opts)` — `classify.ts:123`.** The orchestrator. Builds `parts = [loadCommonPrefix()]`, conditionally pushes `loadOperatorSuffix()` (`opts.includeOperator`) and `loadUserSuffix(scopedUser)` (`opts.scopedUser`), joins with `\n\n---\n\n`. This is the whole composition surface — three inputs, deterministic output.
- **`loadCommonPrefix()` — `classify.ts:196`.** The force-load. Reads `core.md` + **all** `activities/*` (except `reflection.md`) + **all** `situations/*` (except `operator.md`) + **all** `internal-reference/*` + **all** `public-knowledge/*`, via `readIdentityDir` (`classify.ts:155`), which globs each subdir, filters to `.md` (excluding `_`-prefixed and `README.md`), sorts alphabetically for cache stability, and joins. **Cached in-process** (`cachedCommonPrefix`, `classify.ts:175`) — read once per process, byte-identical thereafter. This is the mechanism that guarantees Evryn always *has* every module regardless of whether she'd think to call for it.
- **`loadReflectionPrefix()` — `classify.ts:229`.** The one deliberately *narrow* prefix: `core.md` + `reflection.md` only, for the consolidation pathway. Evidence that per-pathway composition tables already exist in practice — reflection is excluded from the common force-load precisely to keep its length off the ~200/day hot path.
- **`loadOperatorSuffix()` — `classify.ts:243`** and **`loadUserSuffix(user)` — `classify.ts:275`.** The two conditional suffixes (operator.md + operator profile; scoped user's person-context + operator-about-user messages).

This tool remains available for verification or reloading; you rarely need to call it."*

So Step 53 isn't repairing a live-but-flaky mechanism; it's deciding whether to *revive* one (and how to make it reliable) or to *formalize its retirement*.

## 3. The failure modes (and where they came from)

The empirical origin is the **2026-05-28 → 29 integration test** (`_evryn-meta/docs/sessions/historical/2026.05/2026.05.28-integration-test.md`). What was observed, in the wild, when selective on-demand loading was live:

1. **Silent skip.** Evryn didn't call `read_identity_module` for modules she needed. She "follows what's explicit and the explicit is incomplete" — a module she *should* pull based on a soft cue in core.md simply didn't get pulled. (This is the headline failure Step 53 names.)
2. **"Loaded-but-not-loaded" / the introspection gap.** She *reported* having loaded modules she had not actually read — the runtime tool-call log and her self-report diverged. This is why the harness must assert against the **tool-call log**, not against what Evryn *says* she loaded (session doc line 31, DC knock-list #2: *"if shit is weird, he wants to verify what actually loaded, not what was supposed to load. Empirical observability beats designed observability."*).
3. **The "slick-phrasing tell" ("no tool, no call").** She'd produce confident, well-phrased output *as if* a module were loaded, masking the miss — the fluency hid the gap. A reviewer reading only her output couldn't tell.
4. **Partial / muddled application even when loaded.** Even when she *did* load `gatekeeper-onboarding.md`, the first message from it grabbed some beats and dropped others, and read "Should also cover" as a soft suggestion when it was load-bearing (session doc item 4). This is a *content/voice* problem (Step 50/Mira territory), **not** a loading-mechanism problem — flagged here so Step 53 doesn't try to solve it with a loading fix.

**The 2026-05-29 patch** was two-pronged: (a) a *standing instruction* to Evryn to `read_identity_module` for all five modules every volley and lead with a header naming what's loaded (an **instructional** fix — brittle, ADR-033/Step-51 territory), then (b) DC's **force-load** (a **structural** fix — jam everything in, remove the choice). Force-load won because it converts a flaky judgment call into a structural guarantee. The cost is problem (2): the always-loaded floor.

**Why this is a known-hard problem, not an Evryn-specific bug:** "get an LLM to reliably invoke the right tool / load the right context at the right moment" is a general reliability problem (tool-selection reliability, retrieval-timing). Selective loading asks the model to (a) recognize it lacks context, (b) know which module supplies it, (c) actually call the tool before proceeding — three judgment steps, each a failure point. Force-load removes all three by making the context unconditionally present. This is the core reason force-load is hard to argue against on *reliability* grounds — it can only be argued against on *cost/scaling* grounds, which is exactly where the caching tension bites.

## 4. Candidate designs

Four candidates, framed as points on a spectrum from "keep force-load, harden it" to "revive selective loading." Each names its reliability story, its cost/scaling story, and its trade-offs. **All four assume the test harness (§1c) is built regardless — it's a shared prerequisite, not a design choice.**

### Design A — Verified force-load (keep the band-aid; make it structural + asserted)

**Shape:** Keep force-loading, but stop treating it as an implicit side effect of "glob the dirs." Make it a **declared per-pathway composition manifest** (a table in code: pathway → required modules) that the runtime force-composes, plus a **runtime assertion/canary**: before `runEvrynQuery` fires, assert every module the manifest requires is actually present in the composed systemPrompt (e.g. a sentinel marker per module, or a length/hash check), and `notifyDev` + fail loud if one is missing. Today's `loadCommonPrefix` auto-discovery becomes an *explicit allow-list* so a dropped or renamed file can't silently change what loads.

- **Reliability:** highest. Loading is structural (no model judgment), and now *verified* — the 2026-06-01 `existsSync` guard (`classify.ts:157`) already hints at this instinct; Design A generalizes it from "dir exists" to "required modules present."
- **Cost/scaling:** unchanged — the floor still rises. Mitigated only by Step 50 dedup + prompt caching. Does **not** solve problem (2); it *accepts* it and bets on caching.
- **Trade-offs:** cheapest to build (a manifest + an assertion). Doesn't reclaim tokens. Its correctness depends entirely on whether the caching bet (§6) holds — if caching runs cold, Design A is the expensive option.
- **ADR-012 amendment:** formalizes force-load as the standing design; retires `read_identity_module` from on-demand duty (keep for verification/reload only), and documents the manifest+canary.

### Design B — SDK Skills framework as the loading mechanism

**Shape:** Convert `internal-reference/` (and possibly situation/activity) modules into Agent SDK Skills (`.claude/skills/SKILL.md`), whose three-level progressive disclosure (metadata at startup → instructions on trigger → resources on demand) is *purpose-built* for "load the right capability at the right moment." Let the SDK's own trigger mechanism decide what loads, instead of Evryn's ad-hoc `read_identity_module` judgment.

- **Reliability:** **unknown and must be measured** — this is the crux of sprint start-point (b). ADR-012's Skills addendum (2026-03-05) rejected Skills for identity, but on *architecture* grounds (metadata visible at startup for every query → can't scope operator.md out; Skills use third-person capability descriptions, not second-person identity), **not** because Skills-loading was tested and found unreliable. Step 53 should test whether Skills' native trigger loads *more reliably than Evryn's tool-call judgment did* in the 2026-05-28 test. If it does, the reliability argument for force-load weakens.
- **Cost/scaling:** potentially the best — Skills metadata is ~100 tokens/skill at startup, full body only on trigger. If reliable, this is genuine selective loading with a better trigger, solving problem (2).
- **Trade-offs / open risks:**
  - **Operator security regression risk (blocking concern).** ADR-012's #1 reason to reject Skills: Skills metadata is visible to the model at startup, so `operator.md`-as-a-Skill would expose its *existence* in user contexts. The three-layer operator defense (ADR-017) depends on Evryn not knowing operator mode exists. Any Skills adoption must keep operator.md **out** of Skills entirely (stays trigger-composed). Verify this is achievable before pursuing B.
  - Skills' second-vs-third-person framing (identity is "You're talking to…", Skills are "Processes X"). May be surmountable (Skills bodies can be written second-person), but it's untested for identity.
  - The runtime already diverged hard from the SDK's file-based config (ADR-012 chose full string composition, no `settingSources`). Adopting Skills re-introduces a dependency on SDK-managed loading behavior — the exact thing ADR-012 chose to avoid ("immune to SDK changes"). Weigh the coupling.
- **ADR-012 amendment:** would be a substantial reversal of the 2026-03-05 Skills addendum — needs fresh SDK-Skills research (the addendum's research is ~4 months old) + an empirical reliability head-to-head.

### Design C — Hardened selective loading (revive `read_identity_module`, make it stick)

**Shape:** Keep plain-docs composition (no Skills), but make on-demand loading reliable via structure rather than instruction. Options within this: (i) a **per-pathway required-module manifest** the runtime *pre-loads* deterministically (so the modules a pathway always needs are structural), while genuinely-optional modules stay tool-loadable; (ii) a **load-receipt / gate** — the runtime detects (from the tool-call log) that Evryn is about to act without a module the pathway requires and injects it or re-prompts; (iii) tighten the `read_identity_module` tool description + core.md "available modules" cues (the instructional lever — weakest, this is what failed in 2026-05-28).

- **Reliability:** (i) and (ii) can be made high (structural); (iii) alone is what already failed. The interesting version is (i) — which is essentially "selective force-load per pathway": each pathway force-composes *its* required set, not the universal set. `loadReflectionPrefix` (`classify.ts:229`) is a proof-of-concept that this pattern already works in the codebase.
- **Cost/scaling:** better than A — a pathway only carries its required modules, not the union. E.g. a pure-triage pathway wouldn't carry `onboarding.md` + `gatekeeper-onboarding.md`. Solves problem (2) partially (the floor becomes per-pathway, not universal).
- **Trade-offs:** more complex composition (a real manifest with per-pathway sets, more cache segments — each pathway has its own byte-identical prefix, so caching still works *within* a pathway but you get N warm caches instead of 1). Risk: if the manifest under-specifies a pathway's needs, you re-introduce the skip failure for the un-manifested case. Needs the harness to validate every pathway's manifest against real fixtures.
- **ADR-012 amendment:** closest to ADR-012's *original* spirit (selective, per-context) but implemented structurally rather than via Evryn's judgment — arguably what ADR-012 *should* have specified.

### Design D — Do nothing structural; ride caching + Step 50 dedup (the "force-load is correct" position)

**Shape:** Accept force-load as the right standing design. Don't build a loading mechanism at all. Instead: (1) build the harness (as a **regression guard** — assert force-load keeps working as modules change), (2) let Step 50 dedup trim the always-loaded block, (3) rely on prompt caching to keep force-load cheap at scale. Amend ADR-012 to say "caching inverts the token-economy premise; force-load is the v0.3 shape."

- **Reliability:** highest (same as A, minus the explicit canary — though the harness gives you regression coverage).
- **Cost/scaling:** this is the *bet* that caching makes the rising floor a non-problem. **Only valid if the week-one cache measurement (Step 69) shows warm** (high `cache_read`:`cache_creation` ratio). If cold, this is the wrong call.
- **Trade-offs:** least work, most leverage *if the caching bet holds*. Highest regret if it doesn't. Doesn't reclaim the *cold-call* cost (every ~5-min-TTL cold call re-writes the whole force-loaded prefix) or the every-commit-invalidates-cache cost (each identity-file commit invalidates the cache for the next call — small at our scale, real at v0.3).
- **ADR-012 amendment:** formalizes the caching-inverts-the-premise reasoning as the decision.

## 5. Recommendation

**Recommend: build the harness first (unconditional), then Design A (verified force-load) as the near-term landing, with Design C's per-pathway manifest as the measurement-gated evolution — and Design B (Skills) evaluated as research but NOT adopted blind.** Reasoning:

1. **The harness is the no-regret move and the actual unblocker.** Every design needs it; it's an explicit sprint deliverable; and the runtime *already emits everything it needs* (`logToolCall`, `classify.ts:1568`, on every tool + `runEvrynQuery` as the driver, `classify.ts:2576`). This is the single highest-leverage thing the Step-53 agent can build, and it should be built and merged (dormant/CI) even if the design question stays open. It also converts the whole problem from "hand-email Evryn and eyeball it" to "iterate wording/mechanism dozens of times a day with assertions" — which is what makes the rest of Step 53 a one-day job instead of a week.

2. **Force-load should stay the reliability floor — the 2026-05-28 evidence is decisive on reliability.** Selective loading asks Evryn for three judgment steps (recognize gap → pick module → call tool before acting) and she demonstrably failed all three, *and* misreported it (§3). No amount of prompt-tightening reliably fixes a judgment failure; force-load removes the judgment. So the recommendation does **not** revive selective loading as the default. What Design A adds over today is **verification** — a per-pathway manifest + a runtime canary that fails loudly if a required module is absent — closing the "silent" half of "silent skip." Cheap, structural, and it generalizes the instinct already in the `existsSync` guard.

3. **Whether we go *beyond* A (to C's per-pathway trimming or D's pure-caching bet) is genuinely gated on the Step-69 cache measurement, which has not run.** This is the honest core of the recommendation: I can't tell you today whether the rising floor is a real cost problem or a caching non-problem, because **the measurement that decides it hasn't happened** (Step 69 needs real Mark traffic, and Mark isn't forwarding yet). So:
   - If Step 69 shows **warm** → the floor is cheap → Design A (verified force-load) is *sufficient*; don't build C's complexity. Effectively A ≈ D-with-a-canary.
   - If Step 69 shows **cold** → the rising floor is expensive → build **Design C** (per-pathway manifest) to cut the union down to per-pathway sets. C keeps force-load's reliability while reclaiming the cost.

4. **Design B (Skills) is worth *researching* this sprint but not adopting on faith.** The sprint explicitly asks for the docs-vs-Skills recommendation. My read: B's reliability is *unknown* (the 2026-03-05 rejection was architectural, not empirical), and it carries a **blocking operator-security concern** (Skills metadata visible at startup vs. ADR-017's "Evryn never knows operator mode exists"). The Step-53 agent should (a) do the fresh SDK-Skills research the sprint asks for, (b) run one empirical Skills-vs-force-load reliability comparison *on the harness*, and (c) treat operator.md as non-negotiably outside any Skills adoption. If Skills proves *more* reliable than Evryn's tool judgment AND operator.md can stay trigger-composed, Skills becomes a live option for the non-operator modules. Absent that evidence, don't adopt.

**Alternatives rejected (and why):**
- **Pure Design D (do-nothing + caching bet), rejected as the *default*** because it bets the whole design on an unrun measurement and adds no verification — a silently-dropped module (renamed file, glob change) stays invisible until it bites Mark. Design A is D + a canary for ~an afternoon more work; take the canary.
- **Instruction-only fixes (Design C.iii — tighten core.md cues + the tool description), rejected** because that is *precisely* what the 2026-05-29 standing instruction tried, and it's brittle (ADR-033/Step-51 flags this class). Reserve instruction-tightening for voice/application quality (Step 50/Mira), not for load *reliability*.
- **Blind Skills adoption, rejected** on the operator-security regression risk + untested reliability + re-coupling to SDK-managed loading (the exact dependency ADR-012 chose to shed).
