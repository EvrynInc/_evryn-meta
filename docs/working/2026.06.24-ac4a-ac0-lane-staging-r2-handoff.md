## 1. The finalized staging design

A second, isolated full Evryn stack that mirrors prod's *shape* but can't touch prod's *data* or *people*. Validate a release candidate end-to-end on staging, then promote the **exact same build** to prod (no rebuild between validation and prod).

| Piece | Prod | Staging | Provisioned by |
|---|---|---|---|
| **Compute** | Railway service → `main` | A **2nd Railway service** (separate service, not just an env — cleaner isolation), deploy target for the RC branch before prod | OC + Justin (account) |
| **Database** | Evryn Product (West Coast) | **A dedicated 4th Supabase project** (us-west-2), reset-able/disposable, seeded from a prod-shape `pg_dump` — **NOT the dev DB** (ADR-045) | OC + Justin (account) |
| **Inbox** | polls `evryn@evryn.ai` | a **test inbox** (`systemtest@` if it has a pollable OAuth, else stand up `evryn-staging@`) | OC verify / Justin (creds) |
| **Slack** | Evryn app → real channels | a **separate staging Slack app + channels** so staging can't ping real operator channels | Justin (creates app) + OC (wires) |
| **Send safety** | `SEND_ENABLED=true`, no filter | `SEND_ENABLED=true` **+ the send-recipient allowlist** (built — §3) so staging sends REAL email but only to test addresses | DC (built) ✓ |

**The validation flow it buys:** assemble the RC → deploy to staging → run the pre-flight isolation gate → run a create-from-zero + Phase-6-style live-fire end-to-end (against staging's DB, emailing only allowlisted test addresses) → confirm the whole interlocking system works → **promote the identical build to prod.** That's the difference between "QC says the code looks right" and "we watched the assembled system do the real thing."

### The full credential-isolation checklist (the cross-wiring invariant)

Every one of these env vars must be **proven to point at non-prod before staging's first boot** — a staging service that silently inherits one prod credential becomes prod-under-another-name. Prove each; never assume:

- `SUPABASE_URL` + `SUPABASE_SERVICE_KEY` → the **4th (staging) project**, never prod (`wvaaqwapueycyxyhxdnh`) or dev (`maqkdesopsskptpxjbqs`).
- `GMAIL_CLIENT_ID` / `GMAIL_CLIENT_SECRET` / `GMAIL_REFRESH_TOKEN` / `GMAIL_USER` → the **test inbox's** OAuth + address, never `evryn@evryn.ai`.
- `SLACK_APP_TOKEN` / `SLACK_BOT_TOKEN` / `SLACK_WEBHOOK_URL` → the **staging Slack app**, never the real `#evryn-approvals` app.
- `SLACK_DEV_WEBHOOK_URL` → a staging dev-alerts channel.
- `SLACK_EMERGENCY_WEBHOOK_URL` → a staging emergency channel (or unset — see M1 below).
- `SEND_ENABLED=true` (staging WANTS real send, to validate rendering/threading) — **made safe only by the next line.**
- `STAGING_RECIPIENT_ALLOWLIST=<test addresses>` → **the safety net. Its mere presence is what makes the runtime "staging."** Fail-closed: empty/garbled = send nothing. (See §3.)
- `ANTHROPIC_API_KEY` → fine to share, but staging spend then shares the Anthropic *bill* (not the analysis — that lands in staging's own DB; see §4). A separate staging key gives clean billing separation if wanted.
- `POLL_INTERVAL_MS`, `PROACTIVE_CHECK_HOUR_PT`, `QUIET_START_HOUR_PT`/`QUIET_END_HOUR_PT`, `NODE_ENV`, `PORT`, `CLAUDE_CODE_PATH` → staging-appropriate values.
- M1's `HEALTHCHECK_URL` / `HEALTHCHECK_AFFIRM_URL` → **staging's own** Healthchecks.io checks if you want staging's silent-death covered, or omit (staging is operator-watched during a validation run, so this is optional for staging).
- Lane A's `INTERNAL_SENDER_DENYLIST` → has a safe default; no staging-specific change needed.

## 4. The dashboard work — staging visibility + real green-lights

Two distinct pieces, both surfaced by Justin this session:

**(a) A staging dashboard tab.** The dashboard's product-vitals API (`dashboard/api/product.ts`) reads the product DB via its own env vars. A **staging tab** = a duplicate endpoint (e.g. `api/staging.ts`) pointed at **staging's DB** (the 4th project), showing staging's actions + costs only. Staging becomes **visible but isolated** — different DB = zero pollution of prod's numbers. (This also resolves the old "does staging spend pollute `llm_usage` analysis?" question: staging's `llm_usage` rows land in staging's own DB.) Small DC/dashboard build; lands in `_evryn-meta/dashboard/`.

**(b) Replace the hollow `/health` light with real per-subsystem green-lights (Justin's instinct; OC ops-reviewed → build-spec-ready below).** `/health` (`src/index.ts`) always returns 200 regardless of liveness (audit F067); the dashboard's "● Live" inherits it (`product.ts` `fetchProcessUp`). We already *have* the real signals — they just aren't wired. The design (DC build spec, OC-reviewed):

- **Make `/health` real + 503-when-wedged.** Replace the unconditional-200 with a handler that checks DB (cheap bounded `limit(1)` query, short `AbortController` timeout, never hang), reads the in-process Slack-socket-connected flag, and reads a `lastSuccessfulPollAt` timestamp (a tiny new write in `poll.ts` after each successful `pollOnce`); return **200** when all critical subsystems healthy, **503** when any is wedged; body `{ status, uptime, checks: { db, slack, lastPollOk, lastPollAgeSec } }`. **Why 503 is load-bearing:** an unconditional-200 means Railway's healthcheck-failure alerting (and M1's external-watchdog path) **can never fire even when Evryn is wedged** — so that whole alerting layer is currently inert. A 503-when-wedged `/health` is what *arms* it.
- **Five dashboard lights** (extend `product.ts`'s existing `lastActivityAt` + `recentErrors`):

  | Light | Signal (the "real" part) | Green / Yellow / Red |
  |---|---|---|
  | **L1 DB reachable** | a live round-trip query succeeds | succeeds / — / fails |
  | **L2 Slack socket** | socket *currently* connected (not "at boot") | connected / reconnecting <60s / down ≥60s |
  | **L3 Poll loop** | **last *successful pollOnce* completed** (NOT last DB write) | <5 min / 5–15 / ≥15 min |
  | **L4 M1 heartbeat** | Healthchecks.io ping within grace | within / — / grace exceeded (~15min / ~26h affirm) |
  | **L5 24h errors** | `error`/`escalated` items in last 24h (already computed) | 0 / 1–3 / ≥4 or any item stuck >24h |

  **The load-bearing catch (OC):** L3 must key on *last successful poll cycle*, **not** `lastActivityAt`/last-DB-write — during legitimate quiet hours Evryn correctly does nothing and writes nothing, so a DB-write-based light goes stale and cries wolf (the F067 mistake one level down). The poll loop runs continuously 24/7 (quiet hours gate notifications, not polling — ADR-040), so a "last successful pollOnce" signal is always meaningful. *(Verify the continuous-cycle assumption survives clustering, Step 58 — forwards still land continuously; only triage batches.)*
- **L6 (later, with clustering):** "cluster ran today" — fed by the "cluster ran: screened N, escalated M, drafted K" heartbeat the M1 handoff + playbook call for; the dashboard already stubs `cluster_heartbeat: null` in `product.ts`. Design the first build's light-set to extend to it.
- **Composition (the trustworthy two-layer model):** `/health` (real, 503-when-wedged) = the runtime's self-report; **Healthchecks.io = the external dead-man's-switch** that catches total death self-report can't (armed by *silence*, survives a dead box); the **dashboard composes both** into operator-readable lights. The dashboard tells you *which* subsystem is sick when you look; M1/Healthchecks *wakes you* when it all dies and you're not looking — you need both. This also closes SPRINT Step 45 (a correct silent pass is indistinguishable from an outage from the operator's side): with these lights, a correct-but-silent pass shows L1–L4 green; an outage shows red.
- Small DC build (`src/index.ts` + a health module + the `poll.ts` timestamp; `product.ts` + frontend) + one OC config task (point Railway healthcheck-failure alerting at the now-real `/health`). **SPRINT step — §6. Build alongside M1 Stage 2 (Step 4) + Step 45.**

## 5. The standup provisioning sequence (OC-owned; OC ops-reviewing the detail now)

OC ops-reviewed the sequence (GO with sharpenings, folded in). Governing principle: **isolation-first, sends-last** — stand up + *prove-isolated* each input wire before the next; flip real send on **last**, behind the allowlist. The traps below are OC's.
0. **Pre-stage the env-var manifest** (every var from §1, with its *staging* value) BEFORE creating anything. **Trap:** a var you forget to override silently takes a prod value or a code default (`config.ts` has `?? <default>` patterns — a missing staging var doesn't error). The manifest IS the pre-flight checklist.
1. **Create the 4th Supabase project** (us-west-2, RLS on every table). **Trap:** the catastrophic failure is staging booting against the *prod* ref — prove the staging `SUPABASE_URL` by reading the live env, not by trusting you set it. Don't reuse the *dev* ref either (re-creates the two-writers problem ADR-045 solves).
2. **Seed it — prod SHAPE, not prod DATA** (load-bearing). Take a schema/DDL `pg_dump` (schema + RLS + functions + constraints + grants + `COMMENT ON` — the restorable kind, per `backups/README.md`); seed rows **synthetically** (the fictional-gatekeeper fixtures). **Trap:** once Mark is live, a prod *data* dump into staging is a PII leak — bake the shape-not-data discipline now. (Also: the post-2026-10-30 Supabase Data-API grant cliff — a captured-grants schema dump handles it; a hand-rebuilt schema can miss `GRANT … TO service_role` → silent `42501`.)
3. **Create the 2nd Railway service** (separate service, not env) → deploy target = the RC branch.
4. **Wire the env (apply the manifest); boot LOG-ONLY first** (`SEND_ENABLED=false`). **Trap:** booting `SEND_ENABLED=true` before the allowlist is proven is the single most dangerous misstep.
5. **Test inbox** — resolve `systemtest@`-has-pollable-OAuth vs. stand up `evryn-staging@`; its Gmail creds must be the test account's. **Trap:** sharing `evryn@evryn.ai`'s OAuth = two runtimes fighting over the same real inbox + `historyId` (the inbox-version of two-writers).
6. **Staging Slack app + channels** (separate app; mirror the 3-app structure). **Trap:** shared prod tokens → staging drafts land in your real operator channel + staging emergencies pierce your phone DND (destroys the emergency channel's signal value). **Staging's emergency channel must NOT be VIP'd on your phone.**
7. **Set `STAGING_RECIPIENT_ALLOWLIST`** (test inbox + **`review@evryn.ai`** per QC), **prove it blocks a non-listed address**, THEN flip `SEND_ENABLED=true`. *(The allowlist is already BUILT + QC-GO this session — commit `7705675`, §3; OC reasoned from round-1 docs that it wasn't, but it is — so staging consumes it rather than waiting. Until it's deployed to the staging service, staging runs `SEND_ENABLED=false`.)*

**Three runbooks staging needs (OC to author at standup):**
- **A — Stand up staging:** steps 0→7 above. Add: confirm `railway logs` targets the *staging* service (else you debug prod's logs by mistake — `railway logs --deployment` alone streams; use `--since`/`--lines` for history).
- **B — Validate through staging:** = the existing validation playbook (assemble RC → deploy that exact build → pre-flight gate → create-from-zero + live-fire → promote the identical build). Plus the §5 delta below.
- **C — Reset staging to clean (NEW, the whole point of a disposable DB):** stop runtime → drop/reseed the 4th project from the shape dump + synthetic seed → reset the test-inbox cursor if stuck → restart + pre-flight → **verify the clean state** (a reset you didn't verify might have half-failed). This is exactly the mechanical/verifiable procedure a future autonomous OC should own.

### Validation playbook delta (fold into the playbook at standup)
- **Allowlist contents (QC, load-bearing):** the staging allowlist must include the test inbox address(es) **AND `review@evryn.ai`** (or the staging review address) — the review@ send passes through the allowlist too, so omitting it silently breaks staging's review/approval flow (review emails never arrive). Pre-flight: confirm `review@` is in `STAGING_RECIPIENT_ALLOWLIST` before the first validation run.
- **Validate by real behavior, not `/health`.** Since `/health` is hollow, prove staging is alive by observed behavior (DB rows written, Slack posts arriving, real allowlisted email sent) + the per-subsystem lights / M1 heartbeat — never the green light.
- **Validate the CLUSTERED model, breadcrumbed.** Clustering (Step 58 — forwards pool → ~10am daily drain → one morning package) is decided as a super-fast-follow BEFORE Mark, so Mark's first experience IS the clustered model. **Clustering is NOT built yet** (the runtime is real-time per email today). So: the playbook validates real-time *now*, and gains the daily-drain validation *once clustering ships*. **Breadcrumb (do not lose):** when clustering lands, the staging suite must exercise the once-a-day drain (forwards pool as `new` → drain → batch triages → one morning package) AND prove a silent cluster-failure is caught (M1's heartbeat must carry a positive "cluster ran: screened N, escalated M, drafted K" — the dashboard's `cluster_heartbeat` stub in `product.ts` is where it surfaces). Shared test surface with M1.
