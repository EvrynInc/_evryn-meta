## LAYER 1 — DURABLE (the mental model; slow-changing)

### 1.1 What this system is, and what it is for

`evryn-team-runtime` is **the autonomous harness for Evryn's 8 founding-team agents** ("mains": Lucas, Soren, Emma, Mira, Marlowe, Nathan, Dominic, Thea). It is a **separate codebase from the Evryn product** — its own repo, its own Supabase project ("Evryn Team", `azugibmylktinddyqkhd`, us-west-2), its own Railway service. It shares no code with `evryn-backend`.

The problem it solves: today the founding team is operated *by hand* — Justin opens a Claude Code session as Soren, and Soren exists for that session. The runtime makes them **persistent and self-advancing**: Justin summons any of them on Slack; they wake on schedules and events; they coordinate through an audited message database; and they **only ever propose** — nothing external or resource-committing happens without Justin's explicit authorization, enforced by code rather than instruction.

Two structural facts that explain most design choices:

- **Identity never lives here.** The agents' definitions and memories live in `evryn-team-workspace`; this repo is harness code only. The runtime clones that repo and composes prompts from it, so the autonomous agent, a local run, and an interactive Claude Code session are *the same agent* — zero drift by construction (`ARCHITECTURE.md` invariant 1; `src/boot/workspace.ts:206` `syncWorkspace`).
- **The mains run on Justin's Max 20x subscription, not the API.** Near-zero marginal cost. This is the load-bearing economic premise of the whole build, and it carries a policy risk (§1.7).

### 1.2 The architecture of record

**ADR-050 (accepted 2026-07-10, Soren co-signed) — SDK mains + a firewalled worker tier.** Two tiers:

- **Mains (8 agents):** Claude Agent SDK, pinned `0.3.207`, **Opus**, authenticated by a 1-year `claude setup-token` OAuth credential, running as **one Railway service** (not eight). Fresh-composed per-wake prompts with logged composition manifests — deliberately **not** long-lived SDK sessions, because SDK compaction is *observable but not steerable* and silently shaves precision.
- **Workers (Phase 4, not built):** cleared specialists on Anthropic + cheap open-model contract workers on Mastra, **structurally firewalled** — no secrets, no workspace, no DB access; sanitized task slices in, verified outputs back.

**ADR-050's seven cardinal invariants** (`ARCHITECTURE.md:21-29`), each enforced by construction rather than instruction:

1. Identity lives in `evryn-team-workspace`, never in this repo.
2. **Deterministic composition** — the runtime concatenates file contents into the prompt, so an agent *cannot* skimp on its own loading. Every wake logs a manifest (component list, sizes, SHA-256 hashes, workspace git SHA).
3. **Proposal-gated side effects** — external-effect tools are exposed to a wake only when its job carries the matching authorization. Hook-enforced.
4. **Everything durable is in the database** — process memory holds nothing that matters; crash → restart → re-arm losslessly.
5. **Subscription auth only** — agent processes carry `CLAUDE_CODE_OAUTH_TOKEN` and never an API key. No model ever silently downshifts; an agent that can't proceed stops and says so.
6. No worker at any tier holds secrets, workspace access, or DB access.
7. **Evryn is the only user-facing surface** — team agents work on the company's behalf and never serve end users.

**ADR-050's amendment (Justin-ratified 2026-07-11) — the two-tier data rule.** The S6 spike returned CONDITIONAL, and the conditions became the operating rule:

- **Tier A — allowed on the subscription path:** identity files, internal deliberation, ordinary company-confidential material. Basis: with model-training opted out (Justin visually verified his toggle is OFF), consumer retention is ~30 days — the same floor as the standard commercial API.
- **Tier B — NEVER on the subscription path:** attorney-client privileged material (Nathan's work) and trade-secret-critical material, until counsel affirmatively clears it. Basis: safety-flagged sessions are retained up to 2 years, may be human-reviewed and used for safeguards training **even when opted out**, and the Consumer Terms carry no confidentiality commitment plus sole-discretion law-enforcement reporting.

**ADR-052 (accepted 2026-07-15) — memory + wake economics.** The reframe that matters: the memory *design* was already mature; the gaps were that (a) the runtime never **fires** consolidation, so notes accumulate untouched, and (b) the "meta-meeting token bomb" — an @mention subscribes an agent and every subsequent message full-Opus-wakes it, so 8 agents in one thread means 7 full wakes per message. The load-bearing rulings:

- **The thread IS the session** — load the current thread in full; remove the 30-message window.
- **Wake economics = membership, not a model gate** — only agents *in* a thread wake on it; run it like a good meeting (invite, `leave_thread`, late-invite-one-pass). This replaced a first-draft Haiku "should I engage?" gate; Justin rejected guessing in favour of determinism.
- **One shared digest per thread + a per-agent scratchpad headline**, refreshed on a 30-min sweep plus an on-read staleness check ("shape v3", ratified 2026-07-16).
- **Per-agent wake serialization** — one wake at a time per agent ("one mind, one moment").
- **Dormancy memory-write**, thread-ID-stamped (the "echo rule").
- **Two-lane consolidation** (Phase 2): Lane A runs the autonomous/reversible parts; Lane B parks Protected-list items for Justin's async approval.

### 1.3 How a wake actually works, end to end

This is the single most useful thing to hold. A "wake" is one composed, gated, metered LLM turn.

1. **Trigger.** Four kinds, all deterministic — no LLM decides to wake anyone:
   - Slack `app_mention` → subscribe + mirror thread history + wake (`src/slack/index.ts:182`).
   - Slack message in a subscribed thread, or any DM (`src/slack/index.ts:194`).
   - The scheduler's 15-second loop finds a due `wakeups` row or an unprocessed addressed `agent_messages` row (`src/scheduler/index.ts:207-246`).
   - A `node-cron` daily check-in (`src/scheduler/index.ts:321`).
2. **Dispatch gate.** `dispatch()` calls `isDispatchAllowed(db, agent)` (`src/scheduler/index.ts:117` → `src/safety/kill.ts:33`), which fails the wake if **either** the `__global__` row or the agent's own row has `active=false`. Then a per-(agent, thread) in-flight lock is acquired; trigger consumption happens **only after** the lock is held, so a contended trigger is never consumed-without-running (`src/scheduler/index.ts:124-137`).
3. **Composition.** `composeWake` (`src/composer/index.ts:123`) reads the stable layers from two clones and the volatile layers from the DB, then delegates assembly to a pure function (`src/composer/layers.ts:190`). The 12-layer stack, ordered stable → volatile so the prompt cache holds:
   - **SYSTEM:** (1) team-common = workspace `CLAUDE.md` + `runtime-ops.md` · (2) the Hub · (3) team current-state **in full, never windowed** · (4) spokes per agent · (5) identity verbatim · (6) memory (`MEMORY.md`).
   - **USER:** (7) skill index · (8) required-skill bodies · (9) open-threads digest · (10) this thread · (11) tray · (12) wake context.
   - **Missing identity is fatal** — a wake without its identity file is not that agent, so composition refuses (`src/composer/index.ts:182`). Missing current-state is *loud but non-fatal* — it fires the notifier, rate-limited to once an hour (`src/composer/index.ts:158-167`).
   - Every wake writes a manifest row: per-component `{name, chars, sha256}` + workspace SHA + meta SHA (`src/composer/index.ts:291-302`). **This is what makes "the agent got dumb and I don't know why" a diffable event.**
4. **Execution.** `runWake` (`src/wakes/index.ts:119`) calls the SDK `query()` with: an explicit `model` (Opus — the subscription default is Sonnet, which would silently under-brain the mains), an allowlist-built env carrying only the OAuth token (`src/wakes/env.ts:52`), the in-process MCP server, and a **structured output contract** `{reply?, notes, type_tags?, followup_wakeup?}` (`src/wakes/index.ts:87`).
5. **The gate — on every tool call.** `createToolGate` (`src/wakes/tool-gate.ts:62`) is the *only* authoritative fence. It checks, in order: hard-pause → loop signature → infrastructure tools → MCP capability grant → Read capability + path containment → **default deny**. The entire body is wrapped in try/catch that returns an explicit deny (`src/wakes/tool-gate.ts:135-141`) — see §1.4.
6. **Outcome.** Cost is captured from the SDK's own `usage`/`total_cost_usd` (populated with API-equivalent valuation even on subscription) — recorded **even for parked wakes**, because spend happened either way (`src/wakes/index.ts:298`). Then one of five terminal paths: loop-kill, hard-pause park, gate-error park, generic park, or success. Every park writes a lossless park note carrying everything needed to resume (`src/wakes/index.ts:429`).
7. **Success side effects.** Notes append to `MEMORY.md` through a canary-guarded writer (`src/wakes/memory-writer.ts:86`), which git-commits and pushes so cloud-written memory survives a Railway redeploy. The reply posts to Slack, mirrors to `agent_messages` under a deterministic slack_ts-derived id, and regenerates the thread's digest line via one Haiku call.

### 1.4 The safety and control model

This is the part ACU most needs to trust, so here is what is actually enforced versus merely instructed.

**Dormancy — three independent switches, all in one table (`agent_state`):**

| Switch | Column | Effect | Enforced at |
|---|---|---|---|
| Global | `__global__.active` | Blocks **every** dispatch, fleet-wide | `src/safety/kill.ts:42` |
| Per-agent | `<agent>.active` | Blocks new wakes for that agent; in-flight **finish** | `src/safety/kill.ts:44` |
| Hard pause | `<agent>.hard_paused` / `__global__.hard_paused` | Also **parks in-flight** wakes at their next tool call | `src/wakes/capabilities.ts:74-81` → `src/wakes/tool-gate.ts:78` |

- **Boot-paused by default.** On first boot in *any* environment, if the `__global__` row is absent, boot inserts it `active=false` and pings Justin. It **never** auto-resumes an existing row (`src/boot/index.ts:123-146`). Resuming is always a deliberate human act.
- **Kill classes** exist as a stable dashboard contract: `killAgent`, `killAllMains`, `killAll`, and a deliberate Phase-4 `killWorkersOfManager` stub that no-ops loudly rather than being absent-then-added (`src/safety/kill.ts:96-191`).
- **A pause is not a kill** — `pauseAgent` explicitly nulls `killed_reason` while setting `state_reason`, so a re-pause can't inherit a stale kill reason (`src/safety/kill.ts:76`).

**Capability grants — fail-closed.** Every tool call checks a short-TTL (~5s) cached snapshot of `agent_capabilities`. **A missing grant row means NOT granted** (`src/wakes/capabilities.ts:107`). The cache is why a dashboard revoke lands within seconds rather than next wake.

**The tool gate fails closed *by construction*, and this is a subtle, hard-won property.** QC traced the SDK and found that a *thrown* hook callback becomes a control_response error to the compiled CLI — and what the CLI does with it (deny, or fall through to default-allow) is **unprovable statically** and was never tested in S2. So the gate never throws: the whole body is try/caught, any error returns an **explicit deny** and sets `flags.gateErrored`, and `runWake` then parks the wake losslessly (`src/wakes/tool-gate.ts:4-14`, `135-141`; `src/wakes/index.ts:333-340`). **A DB outage cannot open the gate.**

**The read gate.** Reads are contained to two roots (the workspace clone and the read-only meta clone) with defence in depth (`src/wakes/read-gate.ts:50`): reject any `..` segment in the raw input → resolve → require `path.relative()` to be non-absolute and not start with `..` → `realpath` both sides and re-check → **plus** a basename denylist that refuses `.env` / `.env.*` at any depth in either root, applied to both the raw path *and* the realpathed target (so a symlink alias can't launder it).

**Money.** Two mechanisms, and the distinction matters enormously:

- **Per-job hard caps + forecast + the 60% pitch gate** — the ADR-050 money governor. Every wake carrying a `jobId` shows spent/cap/% + forecast-at-completion; at 60% with a forecast breach the next wake carries a **mandatory pitch beat** (`src/composer/layers.ts:176-182`); at 100% the job freezes and parks (`src/money/ledger.ts:90-110`). Accrual is atomic via a Postgres function, so concurrent wakes can't lose spend (`src/money/ledger.ts:67`).
- **A global $/hour velocity ceiling** — the scheduler sums the trailing hour's `cost_events` and pauses the whole fleet at 100% of the ceiling, warning at 80% (`src/scheduler/index.ts:186-203`; `src/money/budget.ts:71`).
- **No model-downshift ladder exists anywhere.** By explicit design (ADR-050): an agent that can't afford to think **stops and says so**; it never gets dumber and keeps working.

**Loop detection.** What trips the breaker is **repetition of identical work**, never volume: the same tool + same args ≥4 consecutive, or a 2/3-length cycle repeating 5× (`src/safety/loop-signature.ts:34`). First trip warns the wake in-stream; a further trip kills and parks. A hard worker burning bright on varied work never matches a loop signature. **Scope: the detector is per-wake** (instantiated fresh at `src/wakes/index.ts:196`) — it cannot see a loop *across* wakes.

**Liveness.** The scheduler refreshes the singleton lock and pings Healthchecks.io at the **top** of every cycle, deliberately *above* the pause gate — because a paused runtime is still a live runtime, and pause gates work, not liveness (`src/scheduler/index.ts:169-179`). The singleton itself is a compare-and-swap lock with a bounded acquire-wait, so a Railway redeploy hands off cleanly instead of crash-looping (`src/boot/singleton.ts:61`).

### 1.5 The composition and memory model

**Six mechanisms, kept distinct so no single one bloats** (ADR-052 §2). This taxonomy is the durable core of the memory design:

| Mechanism | Durability | Load model |
|---|---|---|
| The current thread | ephemeral (until dormant) | always, in full |
| Per-thread scratchpad *(Phase 1, not built)* | dies at dormancy | only this thread's wakes |
| Open-threads digest | rolling | always — one shared blurb per thread + your headline |
| Recent Notes (`MEMORY.md`) | until consolidated | always, whole file |
| Story (`MEMORY.md`) | durable | always, whole file |
| The message ledger (`agent_messages`) | permanent | on demand only |

The discipline: **each item lives in exactly one mechanism, at its right altitude.** Turn-noise dies with the thread. Working state lives in the scratchpad. Cross-thread carry-forward is a note. Distilled understanding is the story. Everything verbatim is always in the ledger — *reachable but never ambient*.

**"One mind, many rooms"** (ADR-052 §4.6) is the frame that resolves most confusion: a thread is a *room*; a wake is the agent stepping into one room fully briefed. There are never "versions" of an agent, because memory (the trunk), digests, and the ledger ride into every wake. In Justin's git metaphor: **memory is the trunk; each wake commits to it; threads organize working context, not identity.**

**The honest token math** (ADR-052 §6, and I verified the measured sizes today — see Layer 2 §2.5):

- A base wake is **~64K tokens before the thread** — dominated by team-common (~17K), current-state (~25K), the Hub (~6K), memory (~6K).
- **Full-thread-load *increases* per-wake cost.** The `notes` field is nearly free (one field in the same structured output as the reply). So "write memory less often" saves ~nothing, and "load the whole thread" costs more.
- **The savings are entirely in (a) not waking on every volley** — membership, the ~70% lever — **and (b) shrinking the always-loaded base layers** via standup burn-down (current-state) and consolidation (memory).

### 2.5 Q5 — Risk register: what could bite when autonomy flips on

**First, the two prior QC blockers ACU asked me to verify against current code — both are genuinely FIXED:**

- **Read-gate path traversal (QC blocker 1) — FIXED, and thoroughly.** The old gate did `startsWith` on the raw model-supplied path, which let `<workspace>/../evryn-team-runtime/.env` reach the OAuth token. `src/wakes/read-gate.ts` now rejects `..` segments outright, resolves, requires non-escaping `path.relative()`, realpaths both sides and re-checks, **and** denylists `.env`/`.env.*` by basename on both the raw and realpathed target. I traced an absolute-path bypass (`/app/src/boot/env.ts`) and a symlink-alias bypass; both are denied.
- **Railway workspace-clone design gap (QC blocker 5) — FIXED and empirically proven.** `cloneOrPull` (`src/boot/workspace.ts:139`) clones on first boot using `WORKSPACE_GIT_URL`/`_TOKEN`, injects the token *per-command* so the on-disk `origin` stays token-free, and scrubs the token from every error string. **The live heartbeating lock proves it works on Railway**, since boot awaits both clones before acquiring the lock.

**Now — what nobody appears to be tracking. These are my findings, verified against source and live data.**

**🔴 RISK 1 — `/pause` and `/resume` are UNAUTHENTICATED. Any Slack workspace member can turn the fleet on.**

This is the highest-severity finding, and it sits on exactly the mechanism that makes the system dormant.

- `src/slack/index.ts:339` (the 👋 reaction handler) **correctly gates** on `config.slack.justinUserId`.
- `src/slack/index.ts:265` (`/pause`) and `src/slack/index.ts:311` (`/resume`) have **no authorization check at all**. I grepped: `justinUserId` appears exactly once in the whole file — line 339. `/resume all` calls `reactivateAgent(db, GLOBAL_KILL_AGENT)` with no check on `command.user_id`.
- **Blast radius:** `/resume all` from anyone in the workspace flips `__global__.active=true` — the single row that is the entire dormancy guarantee (§2.1). `/pause all` from anyone halts the fleet.
- **This exact bug class was already found and fixed in the Evryn product lane** — current-state 2026-06-24 records Step 62: "operator-channel auth gate [`SLACK_OPERATOR_USER_ID` checked on the live handler and the catch-up replay loop]". **The team runtime did not inherit that lesson.**
- **Honest scoping:** today's Slack workspace is effectively Justin plus bots, so the *practical* risk is low right now. And it's **unverified whether the slash commands are even registered** — ACf's session record shows `/dismiss` never appeared in Slack (a probable name-collision at install), so `/pause`/`/resume` may or may not be live. But the code path is fail-open, it contradicts the "structural, not instructional" posture ADR-050 sells, and it gets materially worse the moment any other human joins the workspace.
- **Also:** `src/boot/index.ts:31` warns when `SLACK_JUSTIN_USER_ID` is unset in production — but *only* about the 👋 reaction, and only in production. The runtime currently runs `EVRYN_ENV=dev`, so the warning never fires.

**🔴 RISK 2 — The per-job money governor is INERT. No wake the runtime fires carries a job.**

The entire ADR-050 money system — hard caps, forecast, the 60% pitch gate, freeze-at-100% — applies only to wakes carrying a `jobId`. **Every dispatch path in the runtime passes `jobId: null`:**

- scheduler loop → `src/scheduler/index.ts:271` · daily check-in → `src/scheduler/index.ts:328` · Slack `wakeForThread` → `src/slack/index.ts:169`.

Consequently in `src/wakes/index.ts:129`, `if (req.jobId)` is never true, so `budget` stays null → `maxBudgetUsd` is never set (`src/wakes/index.ts:251`, the SDK-native per-wake backstop) → `accrueJobSpend` never runs (`src/wakes/index.ts:310`).

**The live data confirms it exactly: 0 of 16 `cost_events` carry a `job_id`, and the `jobs` table has 0 rows.** Nothing in the runtime *creates* jobs — the jobs ledger has no producer. This is why pace criterion 5 requires ACf to hand-create a `$2-cap job fixture row`: there is no natural path for a wake to carry a jobId.

**So the only live spend control on a real wake is the global trailing-hour ceiling** (`src/scheduler/index.ts:186`), which fires *after* the money is spent and only pauses. Which leads directly to:

**🟠 RISK 3 — The global $/hour ceiling is mis-calibrated by roughly an order of magnitude for a meta-meeting.**

- Default is **$10/hour** (`src/boot/env.ts:145`), and `GLOBAL_SPEND_CEILING_USD_PER_HOUR` is **not set** in the local `.env`, nor in the 16-var Railway set ACf recorded — so the live value is the $10 default. *(Inferred from ACf's recorded var list, not from reading Railway's env directly — I deliberately did not print Railway values.)*
- **Measured reality: a real Lucas wake costs $0.16–$1.42, averaging ~$1.00** (16 live `cost_events`, $8.07 across 8 wakes). Note the Opus wakes at $0.95–$1.42 each even with heavy cache reads (207K cached tokens on one).
- **So the ceiling trips after ~10 wakes in an hour.** At Phase 3, an 8-agent meta-meeting costs ~7 wakes *per message* — the ceiling would `pauseAll` the entire fleet after **roughly one and a half messages**.
- The gate reads *trailing-hour* spend, so a `/resume all` inside that window gets re-paused within one cycle — a known operator trap already commented at `src/scheduler/index.ts:182-185`, but it compounds this.
- **This is a fails-safe direction** (it pauses, doesn't overspend) — but it means the Meta-Meeting would halt itself repeatedly unless the ceiling is deliberately raised first. ACf1's Phase-1 membership work reduces wake volume ~70%, which helps but doesn't close an order of magnitude.

**🟠 RISK 4 — The park-note self-wake loop can burn money unsupervised. Tracked, still open.**

`parkNote` inserts an `agent_messages` row with `to_actor: <agent>`, `channel: 'system'` (`src/wakes/index.ts:429-451`). The scheduler's candidate query excludes `heartbeat` type and `slack` channel — but **`system` channel with a real `to_actor` is a valid candidate** (`src/scheduler/index.ts:226-237`). So a wake that parks writes a note that wakes the agent, which may park again, which writes another note. **A deterministic failure becomes a self-perpetuating wake loop**, bounded only by the $10/hr ceiling (~10 wakes ≈ 10 minutes). The per-wake loop detector cannot catch it — it's instantiated fresh per wake (`src/wakes/index.ts:196`) and each wake's *internal* tool sequence is innocent.

This is **QC's NEW-3, correctly deferred** to the pre-autonomy hardening list ("add park-count backoff"). It is tracked. It is also **not yet fixed**, and it is exactly the shape of thing that bites at 3am. `boot/recovery.ts` also writes park notes, so a crash-restart can seed it.

*(There is precedent: the "current-state size nag" was a rogue wake generator of this exact family — it inserted an addressed message and spawned a full Opus wake every time current-state was large. It was found and removed on 2026-07-14. The live evidence is still in the DB: one `wake_manifests` row at `2026-07-14T23:22:52Z` with reason `addressed-message`, thread `system:current-state-size`.)*

**🟡 RISK 5 — The daily check-in cron fires at ~2am Pacific.**

`cron.schedule(\`0 ${config.dailyCheckinHour} * * *\`, ...)` at `src/scheduler/index.ts:321` passes **no timezone option**, so node-cron uses the process's local time — **UTC on a Railway container**. `DAILY_CHECKIN_HOUR` is unset, so it defaults to **9** (`src/boot/env.ts:140`) — meaning 09:00 **UTC** = **01:00–02:00 Pacific**. The design intent ("each agent's daily Opus check-in — 'here's what I'm on today'") is plainly a *morning* beat. Currently harmless (the global pause blocks it, and it posts to a test channel), but at autonomy-ON it fires in the middle of the night. Cheap fix; nobody appears to have it.

**🟡 RISK 6 — The message ledger has no clearance partition, and Tier-B material would be readable by every agent.**

`search_messages` and `read_thread` (`src/tools/tools.ts:143`, `:272`) query `agent_messages` with **no scoping to the requesting agent's membership**. Any agent can read any thread verbatim.

**This is deliberate for mains** — ADR-052 §4.1 is explicit: *"The ledger stays open: `search_messages` / `read_thread` reach anything if you get curious — it's just not clogging your context."* So it is **not a defect today**.

It becomes load-bearing at Phase 3, because ADR-050's two-tier rule says **Tier-B privileged material must never touch the subscription path** — and there is no mechanism preventing Nathan (once stamped) from writing privileged material into `agent_messages`, where (a) it composes into wakes and (b) every other agent can read it. The rule is currently enforced by *instruction only* (`runtime-ops.md` has no Tier-B line at all). **The structural answer is that Tier-B material must never enter the ledger** — worth being explicit about before Nathan stamps, not after.

**🟡 RISK 7 — Two gaps between the ratified `runtime-ops.md` and what the code enforces.**

Since ACU asked specifically about protocol-vs-code drift, beyond ACf1's B1/B2 (which are *forward*-looking — they become false when Phase 1 ships):

- **`runtime-ops.md` §2 tells agents:** *"tools with external effect are exposed to your wake ONLY when your current job carries that specific authorization."* **In the code, there are no external-effect tools at all** and no job-authorization check on tool exposure — the tool list is static (`src/wakes/index.ts:197-206`) and gated only by per-agent capability grants, not by job authorization. BUILD-PHASE-1 acknowledges this honestly ("Phase 1 has no external-effect tools anyway"), so the *invariant* is intact by absence. But the doc describes a mechanism that **does not exist yet**, and an agent reading it would believe a job-authorization layer is protecting it. Low blast radius today; a real spec-runtime mismatch to close before external-contact tools ship.
- **`runtime-ops.md` §3 describes the budget line** ("your wake context shows spent-to-date, % of cap, forecast") **as though it always appears.** Per Risk 2, it never appears — no wake carries a job. An agent is being told to self-assess at 60% of a cap it will never see.

**🟢 Things that correctly fail CLOSED (verified — the good news):**

- The tool gate on DB error → explicit deny + park (`src/wakes/tool-gate.ts:135`).
- Capability grants — missing row = not granted (`src/wakes/capabilities.ts:107`).
- Boot on missing/ambiguous env → refuses, never falls back to prod (`src/boot/env.ts:104`).
- `ANTHROPIC_API_KEY` present in the process env → boot **refuses** rather than silently stripping (`src/boot/env.ts:115`).

- Memory writer → canary-guarded, injection-guarded, atomic tmp+rename (`src/wakes/memory-writer.ts:60`).
- Singleton → CAS-guarded; a stalled instance that lost its lock exits loudly rather than double-running (`src/boot/singleton.ts:185`).
