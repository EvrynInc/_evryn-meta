## The trigger (the actual problem being solved)

The presenting problem is **context/compaction pressure**, not aesthetics: the backend runtime (~16,000 lines / 39 files) has grown to where a coordinating AC who needs the whole picture — ARCH + BUILD + work brief + all of `src/` on `main` + a worktree diff — is **at compaction before doing any real thinking.** One in-flight merge in particular (the ③ Runtime Bookkeeping wave) seems to require the coordinator to "hold it all in her head" — and Justin's hypothesis is that she has to *because* it's all so intertwined; decouple it and that requirement drops.

So this is really two questions:
1. **How do future ACs load only the *relevant* parts of the runtime** — reliably, without the "I'll skip xyz" → "turns out xyz was load-bearing" failure that keeps biting?
2. **Is the switch to v0.3 the right moment to re-think build structure deliberately** instead of continuing to grow organically?

## Diagnosis (what's actually wrong — verified by reading code)

The backend is **not** structurally collapsed, and it is **not** "no structure." It has real layers (`db/ notify/ safety/ security/ email/ triage/`), and at least one *deliberate* architectural move (a dependency-injection seam in `index.ts` that registers `poll.ts`'s `stopPolling` into the safety halt orchestrator, specifically to break a `classify → safety → poll → classify` import cycle). The problem is narrower and more fixable than "organic mess":

**One module became a god-module: `src/triage/classify.ts` (2,861 lines).** Its name says "the classifier." What it *actually* contains (verified via its exports + reading the tool-server assembly at lines ~2480–2700):
- **The core Evryn service** — `runEvrynQuery`, the **sole** Anthropic/LLM call in the entire system (the single spend door, where the safety circuit-breaker gate lives, where per-event cost capture lives).
- **Identity composition** — `composeSystemPrompt` + the prefix/suffix loaders that assemble Evryn's system prompt from the `identity/*.md` files.
- **The entire tool layer** — definitions AND handlers for all ~15 of Evryn's MCP tools (`submit_draft`, `supabase_read`, `create_user`, `find_user_by_name`, `append_pending_note`, `record_pass`, `register_gatekeeper_address`, the operator-only `set_thread_scope`/`rescope_messages`, etc.), assembled by `buildEvrynMcpServer`.
- **A pile of deterministic recorders/validators** — `recordPass`, `recordIgnore`, `registerGatekeeperAddress`, `validateUserExists`, `decideUpsertRouting`, note-turn resolution, etc.

This is why loading feels all-or-nothing: **almost any task that touches Evryn's behavior is forced to load 2,861 lines**, and the file's name actively misleads — an AC hunting for "gatekeeper-address registration" or "user validation" would never guess it lives in `classify`.

**Why it's a god-module (the giraffe-nerve framing — Justin's, and it's apt).** `classify.ts` almost certainly started as *just the triage classifier* (back when triage was ~all Evryn did). Then `runEvrynQuery` got added there because that's "where the important stuff was," then the tools, then the recorders. The **name is a fossil** of its origin. This is the software version of the recurrent laryngeal nerve: a connection made when two structures were *adjacent* that becomes a forced detour as the body grows around it. Standard name for the accumulated version: **technical debt**; standard fix: **refactor** (restructure the insides, *behavior unchanged* — NOT a rewrite).

## The key reframe (unlocked by Justin's "triage is not the spine")

Justin's steer, load-bearing for the whole analysis: **the email-triage/gatekeeper-inbox pipeline will likely run forever, but it is *far* from the spine of the organism.** It's *a channel/feature*, not the core.

If that's true, then the current arrangement is backwards: **the organism's core service (`runEvrynQuery` — "talk to Evryn") is currently buried *inside* a soon-to-be-non-central feature file (`triage/classify.ts`).** The spine lives inside a limb. That is precisely the nerve-detour.

**"Straightening the nerve" = extract the core service to its own home** (something like `src/agent/` — the real Evryn engine: `runEvrynQuery` + identity composition + the tool layer), evict the deterministic recorders/validators to where they belong, and let `triage/` go back to being *only triage*.

**Important refinement (so the fix isn't overshot):** the goal is NOT "eliminate the hub." A single chokepoint for the LLM call is *good* architecture — one spend door, one breaker site, one cost-capture site. The debt is the **unrelated tenants** co-located with the hub, not the hub itself. Evict the tenants; keep the genuine core central.

## Rewrite vs. refactor (settling the wording)

Justin's word "rebuild" was never "throw it away / start fresh" (a **rewrite**) — it was "break modules apart and re-home them so they make more sense" (a **refactor**: same behavior, better-organized insides). This distinction matters because a genuine rewrite here would be **actively dangerous**, and the reasons are worth keeping:

- **The runtime's real value is invisible and un-rewritable: the scars.** The dense code comments encode fixes for real production incidents — the throw→cursor-hold contract, the M1 lossless halt/park, dedup-resume, the 2026-04-27 runaway-loop, the DND finding. That hard-won correctness lives in the *code*, not in ARCH. A rewrite deletes those lessons and re-ships the bugs.
- **We don't fully hold *why* every weird bit is there** (Justin's own words: "I know precisely fuck all about coding"; AC was only lightly loaded). That's the worst precondition for a rewrite (Chesterton's Fence: don't remove the fence until you know why it was built).
- **A live pilot user (Mark) makes a multi-week rewrite stall especially costly**, and rewrites are a classic way early software efforts stall out.

**Refactor = yes. Rewrite = no.** The pattern that gets you to clean structure *without* a big-bang switch-off is the **strangler fig**: grow the new, well-structured modules around the old runtime and let them take over incrementally, so the working system is never turned off.

## Timing + sequencing (the part that's genuinely tricky)

There are **two differently-timed moves**, and conflating them is the trap:

**Move 1 — The map (do soon; cheap; near-zero risk).** Docs + tooling, no code change, directly reduces what a coordinator must hold in head — so it helps the *current* pain immediately. Two pieces:
- **A regenerable dependency skeleton** — a script that emits the import graph + fan-in/fan-out + tier buckets, so an AC gets the structural skeleton in seconds without reading source. Never goes stale (generated from the code). Starter commands used this session (from `evryn-backend/`):
  - File sizes: `find src -name '*.ts' | while read f; do printf "%5d  %s\n" "$(wc -l < "$f")" "$f"; done`
  - Import edges: `find src -name '*.ts' | while read f; do echo "### $f"; grep -oE "from ['\"]\.\.?/[^'\"]+['\"]" "$f"; done`
- **A hand-maintained "Code Atlas"** — the concept→file→coupling layer ARCH deliberately doesn't carry: the always-load core, a per-task "if you're working on X, load {these files}, beware {these invariants}" table, and an **invisible-edge registry** (the DI seams + cross-file behavioral invariants that neither ARCH nor the import graph shows).
  - *Caveat AC flagged:* the atlas is itself a doc that can drift — the exact failure mode we're fighting. The skeleton-script stays true automatically; the atlas needs a discipline (regenerate-and-diff at `#lock`, or a currency gate) or it rots. Decide that up front.

**Move 2 — The `classify.ts` extraction (the real fix; schedule into a *quiet gap*, NOT on top of live builds).** Behavior-preserving, test-guarded, QC-reviewed. **The sequencing risk is concrete and current:** the in-flight ③ Runtime Bookkeeping wave already rewrites `classify.ts` (it removes `record_pass` and the raw `supabase_upsert` surface). A big extraction dropped on top of the in-flight worktrees would **collide hard and make the merge problem *worse* before better.** It wants a moment when `main` is stable and no worktree is deep in that file. And bluntly: **the extraction cannot rescue the *imminent* merge** (it can't land in time) — for that one we lean on the map to cut cognitive load, or squeak through and let the extraction pay off on the *next* wave.

**On "is v0.3 the moment?"** Partly. v0.3 is mostly *additive* (matching, web app, payments, identity verification attach to the existing spine; the triage pipeline keeps running). ARCH already anticipates the "three brains collapsed into one agent" separating as complexity grows — so **most of the restructuring win comes for free by building v0.3's new subsystems with real module boundaries from day one**, rather than piling them into existing files. But the compaction pain is *now*, so Move 1 shouldn't wait for v0.3, and Move 2 may be worth doing before v0.3 proper (in the first quiet gap) rather than as part of a v0.3 grand-restructure.
