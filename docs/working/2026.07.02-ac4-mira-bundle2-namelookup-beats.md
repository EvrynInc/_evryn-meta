## Why this exists (the runtime change — the *what*, plainly)

Bundle 2 (v0.2.5) puts a security lock on Evryn's generic `supabase_read` tool. The one change that touches Evryn's *instructions*:

- **A `users` read via `supabase_read` now REQUIRES a filter on `id` or `email`.** A **name-based** `users` read (e.g. "find the user whose name is Alice") through the generic tool is now **refused** (it was a cross-user-dump vector).
- **New typed tool: `find_user_by_name`.** This is the safe, purpose-built home for *name → id* resolution. Give it a name (or email fragment); it returns a small, capped list of candidates with **thin, non-sensitive fields only**: `id`, `display_name`, `full_name`, `role`, `status`, and `is_stub` (true = a not-yet-onboarded record whose UUID is still canonical). It **never** returns email, story, notes, or any profile content. So: resolve the name to an id with `find_user_by_name`, then act on the id.

**Net for the identity layer:** two existing beats currently tell Evryn to look a user up *by name* via `supabase_read`. That instruction now points at a refused path. Repoint the **name** path to `find_user_by_name`; leave the **email** path on `supabase_read` (email is still an allowed filter). Nothing else changes — email lookups, the `create_user` flow, `append_cross_user_note`, `set_thread_scope` all work exactly as before.

**The requirement:** the "by name" half now goes through `find_user_by_name`; the "by email" half stays on `supabase_read`. Everything else in item 2 (the blind write, user-isolation discipline, "you never load the subject's profile") stays exactly as-is — this is a tool-routing repoint, not a rethink of the beat.

**Shape to voice (your words, not mine):** something like — *to resolve the subject to their `user_id`: if you have their email, a scoped `supabase_read` on `users` by email; if you have their name, `find_user_by_name` (it returns just the id + a few non-sensitive fields to pick the right person). Then `append_cross_user_note`.* Keep it tight; the id is "the minimum you need," and `find_user_by_name` is designed to give exactly that without loading the subject's profile — so it actually *reinforces* the blind-write discipline (it can't leak the subject's story even if you wanted it to).

## Beat 2 (NB-2) — `evryn-backend/identity/situations/operator.md` (verify-and-lock, ~line 26)

**Current text (the relevant clause):**
> "Look them up in the `users` table; present the block — full name, brief description, company/context, role, status (nothing sensitive); ask the Operator to confirm. If the name is ambiguous (multiple matches), ask for the UUID."

**The requirement:** in verify-and-lock the Operator almost always names the user — so "look them up in the users table" is a *name* lookup, now refused via the generic tool. It becomes a clean **two-step**, and the two-step already *composes* with the verification block:
1. **`find_user_by_name`** (name → the canonical `id` + `full_name` / `role` / `status` — which is most of the verification block already), then
2. a **scoped `users` read by that `id`** to load the profile for the "brief description, company/context" part of the block.

The "if the name is ambiguous, ask for the UUID" instinct stays — and `find_user_by_name` makes it *better*: it returns a small capped candidate list, so Evryn can see the ambiguity directly (multiple candidates) and either disambiguate from the thin fields or ask the Operator for the UUID.

**Shape to voice:** clarify the lookup as *find_user_by_name to resolve the name to the id (and see if it's ambiguous), then a scoped read by id for the profile you present in the block.* The catastrophic-wrong-user framing and the confirm-gate stay untouched — this is only *how she finds the id*, not the discipline around confirming it.
