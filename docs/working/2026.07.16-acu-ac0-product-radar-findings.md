## 1. R1 — an untracked correctness bug in halt-resume (**REPORTED** — please confirm)

**Plain statement:** `resumeParkedItem` (`evryn-backend/src/email/process.ts:483`) does **not** call the Step-61 gatekeeper lane resolver. `resolveGatekeeperByInboundAddress` appears exactly **twice** in `process.ts` — the import, and the single call site inside `processEmail`.

**The failure:** on a **halt-resume** of an item that arrived via a *registered non-primary* inbound lane, the resume path skips lane resolution → creates a **phantom lead** and triages them against an **empty profile**. That is verbatim the bug Step 61 exists to eliminate.

**Why it matters beyond itself:** `ARCHITECTURE.md` states the lane-resolution guarantee **absolutely**. If this holds, the runtime does not honor it on one path — a spec/runtime mismatch of the exact class QC exists to catch.

**Reachability is narrow** — needs a halt (M1 breaker trip or restart) **plus** an item from a registered secondary lane; forwards additionally need clustering **OFF**. Not a Mark-live blocker on its own.

**It survives ③ byte-identical** (the reporter diffed the `ac2/step57-runtime-bookkeeping` branch) and appears in **no doc** — not SPRINT, not ARCH, not a lane brief.

**ACU's only sequencing note:** it touches `process.ts`, which ③ also touches — so if you do take it, **after ③**, not alongside. Everything else is yours.
