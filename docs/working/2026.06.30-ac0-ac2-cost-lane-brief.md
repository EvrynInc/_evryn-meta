### ① Haiku pre-screen (Step 44) — the #1 lever (~$1,800/mo). NEARLY DONE (only deploy remains).

**What it is:** a separate, cheap `claude-haiku-4-5` call placed *in front of* the Opus forward-triage (**forwards only**; direct messages to Evryn stay full-Opus). It gates whether the expensive Opus query runs at all. It loads **NONE** of Evryn's identity/dossier/history — just this gatekeeper's tight criteria (from their `profile_jsonb` story/notes) + the (untrusted-fenced) email. It returns **exactly one of THREE verdicts:**
- **`ignore`** = OBVIOUS junk it's certain of (spam / mass-marketing / automated/vendor/bot — not a real person). In active mode → the runtime deterministically `record_ignore`'s the item **terminal-dead, NO contact stamp** (junk is not a person to service); Opus never runs.
- **`pass`** = a REAL person, but an OBVIOUS, certain no-fit for THIS gatekeeper. In active mode → `record_pass` (the contact **IS** stamped and enters the v0.3 serviceable backlog — a pass is a real Evryn user, just not this gatekeeper's connection); Opus never runs.
- **`escalate`** = anything else (doubt / thin info / ambiguity / whiff of potential / anything smelling like a bad actor / any failure) → the existing full Opus triage, unchanged.

**⚠️ THREE verdicts, not two (Justin's 2026-07-03 catch — this session's rebuild):** the earlier two-verdict screen labeled junk as `pass`, which **polluted the v0.3 outreach backlog + wrote a spurious "Evryn chose not to connect you" stamp on spam records.** The fix files junk as `ignore` (terminal-dead, no stamp), keeps `pass` for real off-target humans, escalates the rest.

**🔴 The whole safety model is asymmetric error.** A **false-FILTER** — the screen `ignore`'d OR `pass`'ed something Opus would have called gold/edge — is **silent and un-backstoppable**: a filtered forward produces no draft, so the approval gate never sees it, and a real gatekeeper connection is dropped with nobody noticing (the Gate-B class). A **false-escalate** just costs one Opus query. So: **(a)** the instruction biases HARD to escalate on any doubt; **(b)** the RUNTIME fails safe to escalate on EVERY failure (API error, thrown exception, timeout, unparseable output, or a verdict not exactly `ignore`/`pass`). A failure must NEVER become a silent filter. We never trust the screen until a shadow trial proves it ~never false-filters.

**Mode gate:** `HAIKU_SCREEN_MODE` env = **`off`** (default — screen never runs; forward path unchanged) | **`shadow`** (runs + logs its would-be verdict, but NEVER acts — Opus always runs) | **`active`** (acts). Config-only flip, no redeploy. Ships `off`; the `shadow→active` flip is a **separate Justin-gated decision** after the shadow trial. A typo'd/garbage value falls to the safe `off`.

**The 3 observability adds (the "who actively looks" watch surface — a logged-but-unread store answers nothing):**
1. **Dashboard readout on the Evryn Product tab** — screened N / would-skip K / false-filter count + the list. It's an OC/Soren dashboard piece; coordinate via AC0 (pairs with the Step-46 cost-dashboard).
2. **Fresh-false-filter `notifyDev`** — the instant a false-filter row lands in shadow, alert (item + Haiku reason + Opus verdict) → catch a systematic blind spot on day 2, not at the day-7 readout.
3. **Haiku's own spend tagged in `llm_usage`** (pathway `haikuScreen` → activity `screen`) so the flip decision proves **NET** savings (Opus-avoided − Haiku-added), not a flattering gross number.

**Shadow store:** new `haiku_screen_shadow` table (migration `backups/add-haiku-screen-shadow-table.sql`; RLS on + `service_role`-only; carries a `mode` column so the dashboard tallies false-filter over **shadow rows only** — active rows never run Opus, so they'd read a structural "0"). **Measurement subtlety (load-bearing):** the backfill derives Opus's forward verdict from `triage_result` (set for `pass`) OR the ignore terminal-state (`sender_type=ignore`/`status=ignored`) OR `metadata.draft.classification` (`gold`/`edge`) — via `deriveOpusVerdict` — because **gold/edge do NOT set `triage_result` today** (they live only in draft metadata). Reading `triage_result` alone would miss a false-filter on a real gold lead — the exact case the trial exists to catch. (③, below, canonicalizes `triage_result` for all three; this is the interim.)

**M1 interaction:** a tripped breaker **SKIPS the screen** (no Haiku spend during a halt) and falls through to the Opus path, which performs the lossless halt. M1 semantics stay intact.

**Validation gates:** pre-live = the 18 synthetic fixtures in DEV (known answer key) → ~0 false-filter. Live (Mark-gated) = ~1 week of real Mark traffic in `shadow` → ~0 false-filter on real golds/edges → **Justin's EXPLICIT sign-off** → flip `active`. **Never auto-flip.** Note (expectation-setting): the ~$1,800/mo **lags ~1 week into Mark-live** — there's no real traffic to shadow-screen pre-Mark (dev = 18-fixture calibration only), and it's full-Opus until the flip.
