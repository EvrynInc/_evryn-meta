### ① Haiku pre-screen — Step 44 (ON TOP — Justin's #1; DECIDED)
Kills the ~5,650 Opus passes/mo (≈ **~$1,800/mo** for a ~$15–30/mo screen). You can't tier *within* a `query()` (it's pinned to one model), so this is a **separate cheap Haiku call that gates whether the Opus query runs at all**:
`forward → Haiku screen ("obvious no-fit for this gatekeeper?") → obvious pass → record_pass (deterministic), Opus never runs; else → full Opus triage.`
- **Self-contained:** loads NONE of Evryn's identity — just the gatekeeper's tight criteria + the email + a one-line instruction.
- **🔴 Safety rule (non-negotiable, asymmetric error): Haiku may only PASS, never gold/edge, and must over-escalate on any whiff.** A false-pass is silent + un-backstoppable (the Gate-B class); a false-escalate costs one Opus query. Bias hard toward escalate.
- **Validation = SHADOW MODE.** Build it to **log what it WOULD decide (not acting)** against Opus's real verdicts + the 18-fixture answer key, for ~a week. ~0 false-pass on golds/edges → flip it on. This IS the ADR-020 model-tier analysis, done cheaply — write it as an **ADR-020 amendment.**
- **⭐ Ship the build + shadow-logging FIRST so the week-of-data clock starts immediately.** This is the one item gated on real-time-elapsed, so it leads the lane. It's additive (gates the Opus call) → it can deploy early, ahead of clustering/bookkeeping.

Haiku-screens the pile (① above) → Opus the survivors back-to-back (the first pays the cold cache write, the rest run warm) → gold/edge drafts land in the approval queue together → **the gatekeeper gets ONE package per cluster.**
