## 0. The 90-second orientation (read this first)

- **What the engine does.** v0.2 already captured *everyone* — every forwarded email is an `emailmgr_items` row with a classification (`sender_type` + `triage_result`) and reasoning, and every sender has a `users` row (at `status = 'lead'`). But Evryn **contacts no one** today. The outreach engine is the thing that turns that passive record into **active, tier-tailored replies** on Mark's behalf: gold ("you're on Mark's radar"), edge ("tell me more so I can judge the fit"), pass/redirect ("not the right fit for Mark right now, but finding connections is what I do"). (`BUILD-EVRYN-v0.3.md:17-27`; message templates at `gatekeeper-flow.md:72-82`.)
- **Two populations, one engine.** (1) The **Phase-I backlog** — the ~200/day people logged during v0.2 calibration, never contacted, reached retroactively based on their stored classification. (2) The **go-forward stream** — new forwards get outreach shortly after triage once Mark is live. Same three-tier messaging; the backlog is just a big one-time batch the same machinery drains. (`gatekeeper-flow.md:127-137`.)
- **Why this is the revenue-earliest slice.** It runs on data we *already have* — no new traffic required to *build or even run* it against the backlog. It's time-sensitive (Justin wants outreach within ~2 weeks of MVP, while senders still remember emailing Mark — `BUILD-EVRYN-v0.3.md:25`). And it's the first point where Evryn *engages a paying-funnel user*, so it's the front door to first revenue.
- **The single most important runtime fact.** The outbound path Evryn already uses for gold notifications is *exactly* the seam the outreach engine plugs into: **`submit_draft` → `submitDraftForApproval` → `status='pending_approval'` → Slack ping → Justin approves → `executeApproval` → `sendEmail`** — and `sendEmail` has a **provenance guard that refuses any send not originating from the approval gate** (`src/email/client.ts`, Step 86e/ADR-047). This means outreach **cannot bypass Justin's approval** without deliberately dismantling a security invariant. The engine's job is to *feed* this pipe at volume, not replace it.
- **The single biggest design fork (detail in §5, Decision 1).** How does Evryn compose each outreach message? A **full Opus pipeline query per contact** (~$1.70 gold-draft, the current path — rich, but ~$340/day for the backlog at 200/day) vs. a **cheaper deterministic/light-model composition** ($0.06–0.32/contact — the tier message is nearly a template with 2–3 slots). ACU's read: **light-composition for pass/redirect, Opus only for edge (and optionally gold).** Rationale in §5.

## 1. The runtime + schema facts this rests on (verify before building against them)

Everything below was read directly from `evryn-backend/src/` and `evryn-backend/docs/schema-reference.md` on 2026-07-13. File:line refs so a builder can re-verify — the runtime is the *actual* shape and diverges from ARCH in load-bearing ways.

### 1a. The data the engine reads

- **`users`** (`schema-reference.md:15-33`): one row per human. `status` values: `active / inactive / suspended / lead / bad_actor`. **Every discovered/forwarded sender defaults to `status = 'lead'`** (`schema-reference.md:26`; confirmed in runtime by Step 66's note that `'lead'` is the default for every discovered user — `SPRINT-V0.2-HARDENING.md:202`). `profile_jsonb` holds `.story` (narrative, append-only), `.pending_notes`, `.structured` (location/profession/company/industry). `email` is the outbound target. `last_proactive_check_at` (`schema-reference.md:32`) is a per-user "last time the daily cron evaluated this user" timestamp — **directly reusable as the outreach-dedup timestamp** (see §1c).
- **`emailmgr_items`** (`schema-reference.md:36-56`): one row per forwarded inbound. The classification the engine keys on:
  - `sender_type` — `lead / ignore / bad_actor` (`schema-reference.md:55`). **Only `lead` gets outreach**; `ignore` (spam) and `bad_actor` never do (`gatekeeper-flow.md:42` — "Spam, scams, and bad actors are filtered out during triage and are not part of any outreach pathway").
  - `triage_result` — `gold / pass / edge`, populated only when `sender_type = 'lead'` (`schema-reference.md:56`). **This is the tier selector** for which template Evryn uses.
  - `triage_reasoning` — Evryn's prose explanation of the classification (`schema-reference.md:53`); the audit trail + the "why" she can surface if the sender or Mark later asks.
  - `status` lifecycle — `new → processing → pending_approval → delivered → matched/passed/no_gk_response`, terminals `matched / passed / ignored / bad_actor / no_gk_response` (`schema-reference.md:50`). Every status change appends to `metadata.lifecycle`.
  - `original_from` — the *original* sender's email (NOT the forwarding gatekeeper) (`schema-reference.md:46`). This is who the outreach is addressed to.
- **`messages`** (`schema-reference.md:78-97`): all conversational messages, `sender_id`/`recipient_id`/`scope_user_id`. Outreach *replies* (when a contacted person writes back) land here and flow into the normal conversation pipeline (see §4).
- **`llm_usage`** (`schema-reference.md:134-154`): per-`runEvrynQuery` cost/token capture with a `pathway` tag. A new outreach pathway tag (e.g. `outreachCompose`) drops straight in — this is how we'll *measure* real per-contact cost once the engine runs, closing the estimate-vs-actual gap the cost model flags.

### 1b. The outbound seam the engine plugs into (do NOT build a second send path)

The whole approval + send machinery already exists and is battle-tested for gold notifications. Trace it:

1. **`createEvrynInitiatedPlaceholder(recipientEmail)`** (`src/approval/flow.ts:90-115`) — for an **Evryn-initiated outbound with no inbound trigger**, creates an `emailmgr_items` placeholder (`metadata.type = 'evryn_initiated'`, `status='processing'`). **It throws if no `users` row exists for the recipient** (`flow.ts:94-98`) — an FK guard. For the backlog this is a non-issue: every sender *already* has a `users` row. This is the exact primitive a cold-open outreach uses.
2. **`submitDraftForApproval(...)`** (`src/approval/flow.ts:186`) — stores the draft in `emailmgr_items.metadata`, flips `status → 'pending_approval'`, sends a review email to `review@evryn.ai`, and pings `#evryn-approvals` on Slack with an approval handle (the item's short_id). Has a **duplicate-submission no-op guard** (`isDuplicateDraftSubmission`, `flow.ts:62-80`) so an accidental re-submit doesn't re-send/re-ping.
3. **Justin approves** in Slack ("approve `<short_id>`") → **`executeApproval(itemId)`** (`src/approval/flow.ts:472`) — re-checks `status === 'pending_approval'` (a hard gate; refuses otherwise, `flow.ts:491-494` + `533-537`), then calls **`sendEmail`** with up to 3 retries.
4. **`sendEmail`** (`src/email/client.ts`) is the **only outbound exit**, and it carries a **provenance assertion that fail-closed refuses + emergency-alerts any external send not originating from the approval gate** (Step 86e / ADR-047, `SPRINT-V0.2-HARDENING.md:259`). Also has a **staging recipient allowlist** (`isRecipientSetAllowed`, Step 74) — when `STAGING_RECIPIENT_ALLOWLIST` is set the runtime only emails allowlisted test addresses (fail-closed), so **the whole outreach engine can be dry-run on staging against real backlog data without emailing a single real person.** This is the safe way to validate the backlog run before it goes live.

**The load-bearing consequence:** the outreach engine is *upstream* of this pipe. It decides *who* to contact, *what tier template* to use, and *composes the draft* — then hands off to `submitDraftForApproval`. **Approval gating, send retries, provenance, and staging safety are already solved.** Any candidate architecture that invents a parallel send path is re-litigating a solved (and security-critical) problem — don't.

### 1c. The scan pattern to copy: `checkProactiveOutreach`

`checkProactiveOutreach` (`src/email/poll.ts:1054-1145`) is a **per-user daily cron scan** and is the closest existing runtime analogue to the outreach engine. It:
- Gates on the PT hour (`PROACTIVE_CHECK_HOUR_PT`, `poll.ts:1063`) so it runs once/day, not every 30s poll.
- Enrolls users via `findCareEligibleUsers()` (`status='active'` AND not a system actor — `poll.ts:1072`).
- Per user, checks `shouldRunProactiveCheck({ now, lastCheckedAt: user.last_proactive_check_at })` (`poll.ts:1090`) — a **23h-minimum per-user interval gate keyed on `last_proactive_check_at`** — so a user isn't re-hit every cycle.
- Wakes Evryn with the user's context via `runEvrynQuery(...)` (`poll.ts:1122`); any outbound flows through the normal `submit_draft → approval` path (`poll.ts:1056-1058`).
- **Bumps `last_proactive_check_at` in a `finally` whether the check produced an outbound, decided "nothing," or threw** (`poll.ts:1133-1143`) — so a permanently-failing user doesn't re-trigger forever.

**Two things to note for the engine.** First, this is a *per-contact full-`runEvrynQuery`* loop — which is exactly the expensive-composition model we want to *avoid by default* for the pass majority (§5). Second, it enrolls only `status='active'` — and **every backlog contact is `status='lead'`, DELIBERATELY excluded from proactive care at v0.2** (`SPRINT-V0.2-HARDENING.md:202,266` — "bringing leads into care is v0.3 contact-outreach work"). So the outreach engine needs its *own* enrollment query keyed on the backlog (`emailmgr_items` where `sender_type='lead'` and not-yet-contacted), NOT `findCareEligibleUsers`. The `last_proactive_check_at`/interval-gate *pattern* is reusable; the *enrollment filter* is different.

### 1d. How verdicts are recorded today (`record_pass` and the Step-57 future)

- On a `pass` classification Evryn calls **`record_pass`**, and the **runtime constructs a deterministic stamp server-side** (`buildPassStamp`, `src/triage/classify.ts:576-589`) written to the *contact's* `pending_notes` — no LLM prose on the ~6,000/mo pass majority (cost lever; `classify.ts:472-493`). Two structural invariants: the stamp lands on the **contact's record only, never the gatekeeper's** (`classify.ts:483-487`), and the status transition routes through `updateEmailmgrItem` (lifecycle-appending), not a raw upsert (`classify.ts:488-492`). Idempotency predicates `isAlreadyPassed` / `isAlreadyIgnored` key on `status` (`classify.ts:509-530`).
- **This matters to outreach because the pass-stamp is the model for cheap deterministic composition** (§5, Candidate 2): a tier-templated outreach draft is the same *shape* of problem — a small deterministic string with a few slots, built server-side without an Opus turn. The pass-stamp proves that pattern is already accepted and works.

### 1e. The Finding-12 collision (a real bug the design must fix)

`SPRINT-V0.3-CANDIDATES.md:19` (Finding 12 / Proposal 01): **a contact-outreach `submit_draft` currently re-uses the gold forward item's single draft-slot** — `draft_to` flips gatekeeper→contact, overwriting the prior gold-notification draft. Harmless at v0.2 (approval gate + the gold notif was already audited before the overwrite), but **at v0.3 the outreach draft/recipient/lifecycle must NOT overwrite the forward's.** The fix is the **connection-record vs. connection-event split** (Proposal 01): separate the forward-item (the intelligence: who emailed, how classified) from the connection-event (the commerce: Evryn reached out to this contact, they replied, etc.). **The outreach engine's data model depends on this split existing** — it's a schema prerequisite (package B in the pre-read), not something the engine can paper over.

## 2. What the message actually is (the composition surface, per tier)

From `gatekeeper-flow.md:72-82`, the three tier messages are near-templates with a small number of variable slots. This is the single most important fact for the cost design — **the message is mostly fixed voice with 2–3 fill-ins**, not a from-scratch composition:

| Tier | Trigger (`triage_result`) | Message shape (slots in `[...]`) | Composition need |
|------|--------------------------|----------------------------------|------------------|
| **Gold** | `gold` | "Hi `[name]` — you reached out to Mark about `[topic]`. I work with Mark... I've put you in touch. He's got a lot on his plate... you're on his radar. In the meantime, finding the right connections for people is what I do — I'd love to hear more about what you're looking for." | Low. `[name]` + `[topic]` fills. Topic derivable from `subject`/`triage_reasoning`. |
| **Edge** | `edge` | "Hi `[name]` — you reached out to Mark about `[topic]`. I work with Mark... I'd love to hear more about what you're looking for and see if there's a great fit here." | Low-medium. Same slots, but edge *invites a conversation Evryn must then judge* — the reply handling is where the intelligence is, not the opener. |
| **Redirect (pass)** | `pass` | "Hi `[name]` — you reached out to Mark Titus about `[topic]`. I work with Mark... Unfortunately this isn't the right fit for Mark right now — but finding the right connections for people is what I do, so I'd love to hear more... and see if I can help." | Low. Same slots. |

**The insight:** the *opener* is nearly deterministic across all three tiers (same voice, same 2–3 slots). The tier only changes a sentence of framing. Where real Evryn *judgment* is required is **not the opener** — it's (a) getting `[topic]` right and warm rather than robotic, and (b) handling the *reply* when the contact writes back (which is just the normal conversation pipeline, §4). This is why per-contact full-Opus composition is over-provisioned for the opener (§5).

**Personalization tension (a Mira call — §6):** a purely templated opener is cheap but risks feeling like a mail-merge — the opposite of Evryn's whole promise. The design lever is *how much per-contact warmth* each tier's opener carries: a light-model pass that adapts one sentence to `triage_reasoning`/`story` may buy most of the warmth at a fraction of the Opus cost. That warmth-vs-cost dial is Mira's voice call informed by ACU's cost mechanics.

## 3. Candidate architectures

Three candidates, each a *complete* engine (enrollment → composition → approval → send → reply-handling). They differ mainly on the **composition** axis (the cost driver) and the **approval-batching** axis (Justin's throughput). All three share the §1b send seam and the §4 reply flow.

### Candidate 1 — Per-contact full-Evryn query (the "checkProactiveOutreach shape")

**How it works.** A cron-scan (copy `checkProactiveOutreach`, `poll.ts:1054`) enrolls not-yet-contacted `lead`s from the backlog, and for **each contact** runs a full `runEvrynQuery` with the contact's context, letting Evryn compose the outreach in her own voice and `submit_draft` it. Approval per draft, exactly like today's gold notifications.

- **Pros:** Zero new composition machinery — it's the existing pathway pointed at a new population. Maximum per-contact warmth and judgment (Evryn reads the story, adapts fully). Lowest *build* risk (the pattern is proven and live).
- **Cons:** **Cost.** ~$1.70/contact (gold-draft modeled, `evryn-cost-analysis.md:85`) → **~$340/day at 200/day**, ~$10k to drain a ~6,000-item backlog. Massively over-provisions the pass/redirect majority, where the opener is near-templated. **Latency + approval flood:** 200 individual Opus queries/day + 200 individual Slack approval pings would bury Justin (the approval flood is as much a blocker as the dollar cost — see Candidate 3).
- **Verdict:** Correct for **edge** (where the conversation Evryn is opening genuinely needs her judgment) and *arguably* gold. **Wrong for the pass/redirect majority.**

### Candidate 2 — Tiered composition: deterministic/light for pass+gold, Opus for edge (RECOMMENDED)

**How it works.** Split composition by tier, mirroring the `record_pass` deterministic-stamp philosophy (§1d):

- **Pass/redirect** → **deterministic template** (server-side, like `buildPassStamp`) with `[name]`/`[topic]` slots filled from `users.display_name` + `emailmgr_items.subject`/`triage_reasoning`. **~$0/contact** (no LLM), or a **light-model (Haiku) single-sentence warm-up pass** at **~$0.06/contact** (`evryn-cost-analysis.md:185`) if a pure template feels too mail-merge.
- **Gold** → **light-model composition** (Haiku or a trimmed Opus turn), ~$0.06–0.32/contact — the gold opener is also near-templated, and the *real* gold value (the connection) is Mark's, not the opener's prose.
- **Edge** → **full Opus** query, ~$1.75/contact (`evryn-cost-analysis.md:84`) — this is the tier where Evryn is opening a judgment conversation, so the warmth + adaptivity is load-bearing.

Enrollment is a backlog-scan; composition dispatches on `triage_result`; all three tiers hand off to `submitDraftForApproval`. Approval is **batched** (see §5, Decision 2) — a digest of N drafts, not N pings.

- **Pros:** **Cost-matched to value.** At Mark's likely ratio (pass is the silent majority, `classify.ts:475`), the blended per-contact cost collapses from ~$1.70 toward **well under $0.30** — roughly a **5–10× reduction** on the backlog drain. Preserves full Evryn judgment exactly where it matters (edge). Reuses `record_pass`'s already-accepted deterministic-composition pattern, so it's not a novel risk.
- **Cons:** Two composition paths to build + test (template + light-model) rather than one. The pass/gold template quality is a **Mira voice dependency** (§6) — a bad template is worse than no outreach. Requires measuring Mark's real tier ratio to confirm the blended-cost win (available ~1 week into traffic).
- **Verdict:** **ACU's recommendation.** It's the outreach analogue of the cost-lever thinking that already shaped triage (`record_pass`, the Haiku pre-screen), applied to the send side.

### Candidate 3 — Clustered digest + batch approval (the throughput-first framing)

**How it works.** Not a *different* composition model but a different **operator-flow** model, and it composes with Candidate 2. Instead of streaming drafts as they're composed, the engine **clusters** the day's outreach (mirroring the twice-daily clustering Justin leaned toward for triage — `SPRINT-V0.2-HARDENING.md:214`) into **one batched approval digest**: "Here are today's 200 outreach drafts — N gold, N edge, N redirect. Approve all / approve-by-tier / review individually." Justin approves a *batch* (or a tier), not 200 pings.

- **Pros:** **Solves the approval-flood** that would otherwise make *any* per-contact engine unusable at 200/day — this is arguably the harder bottleneck than dollars (Justin can't click 200 approvals/day). Fits the existing morning-sweep/digest muscle the runtime already has (`poll.ts:1147+` morning sweep). Batch-by-tier lets Justin spot-check the cheap-templated pass batch and read the edge drafts individually.
- **Cons:** Batch approval **weakens the per-draft review gate** — Justin approving 150 redirect drafts in one click trusts the template, so the template quality (Mira) and the pre-send safety checks (dedup, opt-out, allowlist) carry more weight. Requires a batch-approval UI/flow that doesn't exist yet (today's approval is per-short_id). A bad template in a batch-approved run is a 150-person mistake, not a 1-person one — so **batch approval must be gated on a validated template + a staging dry-run** (§1b allowlist).
- **Verdict:** **Necessary complement to Candidate 2 for the backlog drain**, but its batch-approval half is the piece that most needs a deliberate safety design. Recommend: **Candidate 2's tiered composition + Candidate 3's batched approval, with per-draft approval retained as the default for edge and any low-confidence draft.**

## 4. How replies flow back (this is mostly already solved)

When a contacted person replies to Evryn's outreach, their reply is an **inbound to Evryn** and flows through the **normal conversation pipeline** — it is *not* a special outreach path:

- The reply arrives as a `messages` row (or a `direct_message` `emailmgr_items` item, `flow.ts:133-137`) and Evryn responds via `processDirect` — the same pathway that handles any direct message, at ~$1.03–1.13/query (`evryn-cost-analysis.md:45-46`), through the same `submit_draft → approval` gate.
- **The contact is already a `users` row** (they were logged in v0.2), so there's no new-user creation at reply time — Evryn just continues the relationship. Over time the `lead → active` transition (an open v0.3 design item, `SPRINT-V0.2-HARDENING.md:202`) formalizes "this person is now an engaged Evryn user," which then *also* enrolls them in normal proactive care (`findCareEligibleUsers`).
- **Design implication:** the outreach engine's responsibility **ends at the first outbound** (open the conversation). Everything after is the existing pipeline. So the engine is genuinely *bounded* — enrollment + tiered composition + hand-off to approval — which is why it's a clean, shippable slice. The one thing the engine must set correctly is the **conversation-opening context** so Evryn's *reply* handling knows this thread began as a Mark-forward outreach (carry the `original_from`/`triage_reasoning` context forward — this rides the Finding-12 connection-record split, §1e).

## 5. The two forks that most shape the build

### Fork A — Composition model (cost)

Covered by the candidates. **ACU's read: tiered (Candidate 2).** The cost delta is decisive and the `record_pass` precedent de-risks it. The numbers (all from the HARD-anchored `evryn-cost-analysis.md`, `:83-86,185-187`):

| Tier | Full-Opus per-contact | Light/template per-contact | Backlog-drain cost @ that tier's share |
|------|----------------------|---------------------------|----------------------------------------|
| Pass/redirect (majority) | ~$1.92 | **~$0–0.06** | This is where the 5–10× win lives |
| Gold | ~$1.70 | **~$0.06–0.32** | Small population, either is affordable |
| Edge | ~$1.75 | ~$1.75 (stays Opus by design) | Keep full — judgment is load-bearing |

The task brief's "~$1.36/query full pipeline" is a reasonable *blended* figure; the decomposition above is the defensible per-tier reality and is what the design should be costed against. **Caveat (honest):** these are `n=4` HARD captures + modeled floors (`evryn-cost-analysis.md:202`) — a single captured *cleaned-outreach* run replaces the estimate with data, and the new `llm_usage` `pathway` tag (§1a) is how we'll capture it once the engine runs. Don't lock the final scope against these numbers without the week-one measurement.

### Fork B — Approval model (Justin's throughput + the safety gate)

Covered by Candidate 3. **ACU's read: batched digest for the backlog drain + the cheap tiers; per-draft retained for edge and low-confidence drafts.** The approval flood is as real a blocker as the dollar cost — 200 individual pings/day is unusable — but batch approval trades away per-draft review, so it must be gated on a validated template (Mira) + a staging dry-run (the §1b allowlist makes this safe). **This is the single biggest open scoping question for Friday** because it determines whether the backlog can drain in days or dribbles out one-approval-at-a-time.

## 6. What needs Mira (identity/voice — flagged, NOT written here)

Per the orchestration rule, identity files are Mira's docs — any build that touches one is *coordinated with her*, never folded in quietly. The outreach engine has real identity/voice surface:

1. **The three tier templates (gold/edge/redirect openers).** These are Evryn's *voice* in the highest-stakes first-impression moment. If the design uses deterministic/light-model composition (Candidate 2), the template *is* the voice — so the template text is a Mira artifact, not a runtime constant a builder writes. **This is the load-bearing Mira dependency.**
2. **The warmth-vs-template dial (§2).** How much per-contact personalization each tier carries (pure template vs. one adapted sentence) is a voice judgment — a mail-merge-feeling opener undercuts Evryn's whole promise. ACU owns the *cost mechanics* of each option; Mira owns *which reads as Evryn*.
3. **Opt-out / unsubscribe language.** Every outbound needs a CAN-SPAM unsubscribe (`BUILD-EVRYN-v0.3.md:51` — "CAN-SPAM unsubscribe link in every outbound email") and Evryn should honor a conversational "stop" — the *wording* of both is Mira's.
4. **Backlog time-delay framing.** A backlog contact emailed Mark *weeks* ago; the opener may want a light acknowledgment of the delay ("you reached out to Mark a little while back") so it doesn't read as robotic-late. Voice call.
5. **The `lead → active` transition's identity meaning** (if it lands in v0.3) — when Evryn starts treating a replied-to contact as a full user — touches how she frames the relationship. Coordinate.

**ACU's mechanics-first duty (per the identity-runtime rule):** derive the *mechanical* requirements first (which tier maps to which template, what slots exist, that the unsubscribe link must be present and functional, that opt-out must actually suppress future outreach), brief Mira the *what*, then re-check the voiced templates against the runtime on the way back. QC's spec-runtime-match check is the independent backstop and covers the identity/template files too.

## 7. Invariants the engine must hold (design these in, don't bolt on)

These are the operational-requirements checklist for the eventual DC spec — surfaced now so they shape the architecture, not patch it later:

- **No-double-contact (dedup).** A contact must never receive two outreach messages for the same original email. Reuse the `last_proactive_check_at`-style per-user/per-item timestamp gate (§1c) — a `contacted_at` (or a `connection_event` row per Finding-12's split, §1e) that the enrollment query filters on. Idempotent like `record_pass`'s `isAlreadyPassed` (`classify.ts:509`). **This is a trust invariant, not a nicety** — double-contacting reads as spam and burns the exact goodwill outreach exists to build.
- **Opt-out is durable and suppresses future contact.** A conversational "stop" or an unsubscribe click must set a persistent flag that the enrollment query respects **for all future outreach** (backlog *and* go-forward). CAN-SPAM requires it; trust requires it more.
- **Never contact `ignore`/`bad_actor`.** Enrollment filters strictly on `sender_type='lead'` (§1a) — spam and bad actors are structurally excluded (`gatekeeper-flow.md:42`). A bad-actor who later flips must not silently re-enter the outreach pool.

- **Approval gate is non-negotiable.** Every outbound routes through `submitDraftForApproval → executeApproval → sendEmail`; the `sendEmail` provenance guard (§1b) refuses anything else. Batch approval (Candidate 3) is a *UI over* this gate, never a bypass of it.
