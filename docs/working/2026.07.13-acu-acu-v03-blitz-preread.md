## 1. What v0.3 is, as currently documented

The scope lives primarily in `evryn-backend/docs/BUILD-EVRYN-v0.3.md` (mechanics, moved verbatim out of the MVP spec on 2026-07-07) plus the six staging proposals it references. The documented surface breaks into **eight areas**:

1. **Sender / cast-off outreach.** Evryn engages the people in a gatekeeper's inbound — all three tiers, tailored per tier (gold: "you're on Mark's radar"; edge: a conversation to learn more; pass: honest redirect to Evryn's broader services). Includes the **Phase-I backlog**: the ~200/day people logged during v0.2 calibration who were never contacted, reached retroactively. **Time-sensitive** — Justin wants outreach within ~2 weeks of MVP, while senders still remember emailing Mark. (`BUILD-EVRYN-v0.3.md` → Sender Outreach + Phase I Backlog.)

2. **The web app.** The primary user surface. Design mandate (Hub + technical-vision): the *first* thing a user meets is a **conversation with Evryn**, not a form/dashboard. Scope includes:
   - **OAuth account creation** (Google — self-service ~20 min; Apple Sign In — needs a $99/yr Apple Developer account + ~2-day approval, Justin must register).
   - **Pre-account ephemeral sessions** — tab-scoped, no cookie persistence (prevents cross-person data bleed on shared browsers), full dual-track Evryn from exchange one, server-side session store keyed to a tab-scoped token, nudge→warn→require account-creation cadence (~20/~30/40 exchanges).
   - **`consent_events` table** — audit trail for every consent event (ToS, PP, OAuth, data-share, verification).
   - (`BUILD-EVRYN-v0.3.md` → Web App — Additional Scope Items.)

3. **Matching pipeline.** Evryn's core paid act. Two layers (`ARCHITECTURE.md` §Judgment & Matching): an **algorithmic layer** (PostGIS/structured pre-filters → vector similarity search — pure computation, no LLM) and an **analytical layer** (Opus evaluates the top candidates, writes the *why*). Matching is **profile-to-intent, not person-to-person**, bidirectional. **Event-driven, never scheduled** (fires on new-user-onboarded and on profile-change; costs scale with meaningful change, not DB size — ADR-019). Implies a `match_candidates` cache table (schema TBD, ADR-019 Consequences). **Day-one reality: judgment alone** — no scoring data, no outcome data yet; the computational layers develop as data accumulates.

4. **Payments (Stripe).** Trust-based pricing at the point of connection (free favor → $10k exec match), **100% satisfaction refund guarantee**, Stripe Connect for post-match transactions. (Hub → Business Model; `BUILD-EVRYN-v0.3.md` Principle Breadcrumbs row "100% satisfaction guarantee / refund flow".)

5. **Identity verification (iDenfy).** Required **before any connection**, but **not at account creation** — Evryn asks during the Connection Dance (Step 4 of the connection flow), the first time a user is about to be connected; once verified, always verified. iDenfy stores the documents; Evryn stores only **verification status + timestamp + a non-reversible trust-fingerprint hash** (the hash persists through account deletion — the trust imprint that blocks delete-and-recreate reputation resets). (`BUILD-EVRYN-v0.3.md` → identity verification; `ARCHITECTURE.md` line 106.)

6. **Security hardening — v0.3 structural end-states.** The v0.2 *inner-doors* shipped (v0.2.4/v0.2.5, ADR-047); the full **end-states** are v0.3 and are **prerequisites for the multi-user web app** because that's when `anon`/`authenticated` roles first appear: (a) **S1** — replace the RLS-bypassing generic `supabase_read`/`supabase_upsert` (service_role) with typed tools or an RLS-respecting role; remove the raw `supabase_upsert` write surface (Step 57); (b) **S2** — a WebFetch domain-allowlist wrapper; (c) explicit RLS policy on `evryn_knowledge`; (d) change the project default-privilege rule so new `public` tables don't auto-grant `anon`/`authenticated`, and retro-`REVOKE` existing ones; (e) swap the dashboard's product-DB `service_role` key for a least-privilege read role. (`BUILD-EVRYN-v0.3.md` → Security Hardening.)

7. **Evryn writes her own Linear tickets.** A three-stage build (runtime tool + identity discipline + slim-form Linear protocol in her identity stack) letting Evryn capture runtime/instruction/voice observations as Linear tickets routed to the right team member. **Explicitly flagged as possibly ship-able as a small post-Mark wave *before* the full web-app build** if friction warrants (`BUILD-EVRYN-v0.3.md` line 111). Touches **Mira's identity files** → must be coordinated with her, not folded in quietly.

8. **The six (really eight) staging proposals.** AC2-produced design proposals in `evryn-team-workspace/shared/projects/product/research/v03-design/`, each with a key open question for Justin (`BUILD-EVRYN-v0.3.md` → Staging table): 01 connection-graph schema, 02 proactive behavior / `care_queue`, 03 multi-gatekeeper scoping, 04 feedback-loop formalization, 05 memory scaling / Reflection, 06 trust architecture at scale, 07 multi-party orchestration, 08 capability-vs-constraint. **These are research, not decisions** — the Friday session's job is to react to them and let decisions flow into the v0.3 BUILD doc.

Plus the **granular candidates** (`SPRINT-V0.3-CANDIDATES.md`) — ~20 line-item bug/polish/hardening items deferred from v0.2 (recipient-label mislabel, binding-TTL audit sweep, connection-record/event split, care-population of leads, etc.) that get **triaged into the sprint once scoped** — not blitz headline work, but several are *unblocked by* the schema/outreach packages below.

## 4. Parallel-lane-blitz-fit vs. serial-deep-design

The blitz's power is **parallel lanes** (multiple DC/QC loops on independent slices). But some packages are **serial deep-design** — they need concentrated architectural thinking *before* any lane can run, and splitting them across parallel workers just fragments the reasoning. Sorting the packages:

**Suits a parallel-lane blitz (independent, bounded, low collision risk):**
- **C — Security end-states.** Several discrete, mostly-independent doors (WebFetch wrapper, RLS policy, REVOKE fix, dashboard key) — classic fan-out.
- **B — Schema foundation.** Once shapes are decided (A), each table is an independent migration.
- **I — Evryn-writes-Linear-tickets.** Small, bounded, self-contained — a clean standalone lane (with Mira coordination on the identity half).
- **The granular candidates** — many are one-file fixes that ride alongside their related package.

**Needs serial deep-design first (concentrate, don't fragment):**
- **A — Decisions pass.** This is *Justin + ACU thinking*, not a DC lane. It's the front-loaded serial work that unlocks the parallel lanes. Reacting to eight interlocking proposals (schema choices in Prop 01 constrain outreach in D; Reflection in Prop 05 gates half a dozen deferred items) is exactly the "collision between lanes" reasoning that must happen up front.
- **F — Matching pipeline.** Two-layer, event-driven, embedding-model-dependent, and the intellectual core of the product. Deep design first; then it *can* fan out into algorithmic-layer vs. analytical-layer lanes, but not before the shape is settled.
- **E — Web app.** Large and internally serial (auth → session → onboarding chain, each depending on the prior). Can parallelize *within* once scaffolded (frontend vs. session-store vs. consent wiring), but the scaffold is a serial predecessor.

**ACU's read on blitz shape:** Friday's *serial* work is **A (decisions) + the deep-design of whichever build slice Justin picks** (ACU recommends D-outreach). The *parallel* work — the lanes that actually consume the blitz's throughput — is **B + C + I** (+ granular candidates), which are genuinely independent and don't need traffic. This is why an outreach-first scope fits the blitz: it front-loads the traffic-independent lanes and defers the traffic-dependent quality tuning (F) to when Mark is live.

### Decision 1 — Blitz scope: whole v0.3, or revenue-earliest slice?

- **Context.** v0.3 is a large surface (8 areas + 8 proposals). A Fri–Sun blitz can't land all of it. The outreach backlog is *time-sensitive* (reach senders while they remember Mark). Much of v0.3 (matching quality, payments-live, verification-live) needs traffic and externals that won't be ready this weekend anyway.

Defer the web app (E) deep-build, matching (F), payments-live (G), verification-live (H) — they need traffic/externals not present this weekend.
- **Why:** it converts the blitz's parallel throughput into the packages that are *actually unblocked now*, targets the one time-sensitive deliverable (outreach), and produces revenue-path progress without waiting on Stripe/iDenfy/hosting/Fenwick.
- **Alternatives rejected:** (a) *Scope the whole v0.3 BUILD doc and start the web app this weekend* — the web app is large, serial, and blocked on hosting + OAuth + security roles; starting it now fragments effort against a dependency wall. (b) *Do only the decisions pass, no build* — wastes the blitz's parallel-lane capacity on work (B/C/I) that's ready to run.
- **Question:** **(a)** outreach-first slice (ACU's rec) · **(b)** whole-v0.3 scoping incl. web-app kickoff · **(c)** decisions-pass only?

### Decision 2 — Reflection module: v0.3 or stretch?

- **Context.** Reflection (memory consolidation, story re-synthesis, self-audit) is Proposal 05's open question and **gates a cluster of deferred items**: auto-consolidation, binding-TTL audit sweep, `evryn_knowledge` global-pattern consolidation, cross-scope operator-resolution auto-propagation. It's also the precondition for *autonomous* matching/care.
- **Recommendation:** **Stretch, not core v0.3.** Ship v0.3 with Reflection *stubbed* (schema fields present, module deferred); build it as a fast-follow once there's real per-user story data to consolidate.
- **Why:** Reflection consolidates *accumulated* experience — with one gatekeeper and near-zero connection history at v0.3 launch, it has almost nothing to consolidate yet. Building it before the data exists is planning against an empty table. The team's own consolidation pilots (Lucas/Mira) are the test bed and aren't proven yet.
- **Alternatives rejected:** *Reflection in core v0.3* — front-loads a large, data-hungry module before its inputs exist, and the deferred items it gates are themselves not launch-blocking.
- **Question:** **(a)** stretch/fast-follow (ACU's rec) · **(b)** core v0.3?

### Decision 3 — Apple Sign-In in v0.3 scope (start the Developer-account clock now)?

- **Context.** OAuth account creation is web-app scope. Google is self-service (~20 min). Apple Sign-In needs a $99/yr Apple Developer account with a ~2-day approval — a human long-pole.
- **Recommendation:** **Include Apple, and register the Developer account *now*** (this week), independent of when the web app builds — so the 2-day clock isn't on the critical path later. If web-app is deferred (Decision 1a), registering costs $99 and nothing else.
- **Why:** it's a cheap, slow, parallelizable dependency — exactly the kind to start early. iOS users disproportionately expect Apple sign-in; skipping it adds onboarding friction on the demographic most likely to convert.
- **Alternatives rejected:** *Google-only for v0.3* — viable, but leaves a known friction point; easy to add now, awkward to bolt on later.
- **Question:** **(a)** include Apple + register now (ACU's rec) · **(b)** Google-only for v0.3 · **(c)** defer the call?

### Decision 4 — Connection-graph: are relationship edges ever deleted, or only marked inactive?

- **Context.** Proposal 01's core question, and it shapes the schema (package B) that outreach (D) and matching (F) both build on. The two-table design is `relationships` (intelligence) + `connection_events` (commerce).
- **Recommendation:** **Never delete — mark inactive** (soft-delete / status flag).
- **Why:** relationship history *is* the trust intelligence — a severed edge is signal, not noise (Evryn learns from connections that ended as much as ones that held). It also aligns with the "user data deletable, system learnings not" separation (Hub → Data Layers) and the trust-imprint-on-deletion principle. Hard-deletes destroy the training substrate the whole model compounds on.
- **Alternatives rejected:** *Hard-delete edges* — irreversibly loses relationship-outcome signal and complicates the trust graph's "user can delete personal data but not Evryn's learnings" boundary.
- **Question:** **(a)** soft-delete / mark inactive (ACU's rec) · **(b)** hard-delete · **(c)** defer to schema deep-design?

### Decision 5 — Embedding model: OpenAI vs. Voyage (and is pgvector in v0.3)?

- **Context.** Proposal 05 + ARCHITECTURE Embedding Strategy. The matching pipeline's algorithmic layer needs an embedding model; the choice gates package F, and pgvector is "optional v0.3, necessary v0.4."
- **Recommendation:** **Defer the final model pick to the matching deep-design (package F), but decide *now* that pgvector ships in v0.3** (so the schema — package B — provisions the vector column and index from the start rather than a v0.4 migration).
- **Why:** the embedding-model choice depends on cost/quality analysis that belongs in F's deep-design (and interacts with the PII-anonymization decoupler — the provider never sees raw identity). But *whether pgvector exists* is a schema decision that B needs now; provisioning it early is cheap and avoids a later migration. This decouples the reversible-later model pick from the expensive-later schema pick.
- **Alternatives rejected:** (a) *Pick the model at kickoff* — premature; the analysis isn't done and it's reversible. (b) *Defer pgvector to v0.4* — forces a schema migration mid-stream and blocks any matching experimentation in v0.3.
- **Question:** **(a)** pgvector in v0.3 + model TBD in F (ACU's rec) · **(b)** pick model now · **(c)** pgvector deferred to v0.4?

### Decision 6 — Ship "Evryn-writes-Linear-tickets" as a standalone early wave?

Touches Mira's identity files. Needs a Linear-seat call (~$10/mo for proper provenance vs. owner-attributed `[from Evryn]` tag).

- **Why:** it's genuinely traffic-independent, self-contained, and gives Evryn a real capability she already reaches for (she's tried to write tickets as Slack messages). It's a clean parallel lane that doesn't collide with the outreach/schema work. Coordinate the identity half with Mira.

- **Question:** **(a)** early standalone lane + buy the seat (ACU's rec) · **(b)** fold into v0.3 proper · **(c)** early lane but owner-attributed / no seat?
