## CRITICAL mechanism facts (so you don't relearn them the hard way)

1. **The sandbox is built in the PARENT process at startup.** The VSCode extension runs a long-lived parent; each chat is a *child* (`CLAUDE_CODE_CHILD_SESSION=1`). Opening a **"new chat" does NOT rebuild the sandbox** — the child inherits the parent's already-initialized one. **Only a full window reload / VSCode restart activates settings.json sandbox changes.** This is why Justin's first "I opened a new session" still hung. *(Strong inference from env vars + extension architecture, not doc-confirmed — but it fit every symptom.)*
2. **`dangerouslyDisableSandbox: true` + Justin approving the prompt WORKS** for one-off network commands. Earlier "it hung" was him not catching/approving the prompt, not the flag failing. Use this as the escape hatch if you ever need a network command before a config change is live.
3. **Local git ops (status/diff/commit) run fine sandboxed** — only *network* (push/fetch) is blocked. So you can always commit; pushing needs either the live allowlist or the override above (or Justin pushes via the Source Control panel, which bypasses the Bash sandbox entirely).
4. No env var or `managed-settings.json` is forcing the sandbox on — so project `settings.json` is authoritative once the parent re-reads it.
