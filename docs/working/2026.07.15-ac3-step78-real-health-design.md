## The problem (audit F067)

`evryn-backend/src/index.ts:65` — today `/health` returns **HTTP 200 + `{status:"ok", uptime}` unconditionally**, even when the poll loop is wedged or the process is up-but-not-ingesting. So no external watchdog — a Railway health check, Healthchecks.io, or the dashboard — can tell "healthy" from "wedged." This is what leaves the M1/Railway watchdog *inert*: the very silent-death shape M1 exists to catch reads as green. Step 78 makes `/health` tell the truth, and surfaces it as operator-facing status lights.

## Scope — two halves, Half A first

- **Half A (backend, `evryn-backend`):** the real `/health` endpoint + the liveness signals it reads. The safety-critical, self-contained half. Ships first.
- **Half B (dashboard, `_evryn-meta/dashboard`):** 5 per-subsystem status lights on the Evryn Product tab (`api/product.ts`), reading the new `/health`. Follows Half A.

Non-colliding with AC2's Step 57 (which rewrites `classify.ts` / `process.ts` / `triage.md`): Half A touches `src/index.ts` + a new `src/safety/liveness.ts` + a one-line record call in `poll.ts` + read-only getters in `slack.ts`/`breaker.ts`; Half B is a separate codebase. Safe to run in parallel.

## Half A — the `/health` contract

New response body (keeps `status` + `uptime`; adds `checks`):

```jsonc
{
  "status": "ok" | "wedged" | "halted",
  "uptime": 12345,
  "checks": {
    "db":           { "ok": true },
    "slack_socket": { "ok": true },
    "poll":         { "ok": true, "last_success_at": "2026-07-15T17:00:00Z", "age_seconds": 22 },
    "m1":           { "breaker_tripped": false, "reason": null }
  }
}
```

**HTTP status code (the load-bearing part):**

- **200 `"ok"`** — healthy.
- **503 `"wedged"`** — the poll loop has NOT completed a successful cycle within `HEALTH_WEDGED_THRESHOLD_MS` (default ~15 min — above the 5-min max poll backoff, aligned to the Healthchecks grace), **OR** the DB is unreachable. These are the genuine "up but not doing its job / a restart would help and won't loop" states. This closes F067.
- **200 `"halted"` (deliberately NOT 503)** — the M1 circuit-breaker is tripped. **Rationale (load-bearing, do not "fix" to 503 without re-reading this):** a breaker trip is a *deliberate* M1 halt that already fired its own emergency alert, and M1's invariant is **no auto-reset — restart is the manual reset** (ADR-041). If `/health` returned 503 on a halt *and* an external restarter (a Railway health check) acted on it, a persisting cause would re-trip → restart-loop, defeating M1's over-halt-beats-under-halt design. So a halt returns **200 with `status:"halted"` + the reason** — the dashboard shows it RED, but no external restarter loops against M1. **← OC confirms Railway's actual behavior; see the open question.**

**⭐ PRECEDENCE (load-bearing — evaluate in THIS order; added 2026-07-15 after AC review caught a wedged-first bug):**

1. **breaker tripped → `halted` (200)** — FIRST, *regardless of poll staleness*. When the breaker trips, `haltForBreaker` calls `stopPolling()`, so the poll loop stops and its last-success timestamp goes stale ~15 min later. That stale poll is EXPECTED — the halt *caused* it — so `halted` subsumes it. Returning `wedged` (503) here (the naive "poll is stale → wedged" check running first) would flip a halted process to 503 after the threshold and re-introduce the exact M1 restart-loop the halted-→-200 rule exists to prevent. **Halted beats wedged.**
2. **else, poll stale OR db down → `wedged` (503).**
3. **else → `ok` (200).**

**Startup grace (never-polled ≠ wedged):** a freshly-booted process that has NOT yet completed its first poll is **`ok`/starting, not `wedged`**, until `process.uptime()` exceeds the wedged threshold. A process is only `wedged`-by-poll when it has EITHER polled successfully once and then gone stale, OR been up past the threshold with zero successful polls. This keeps a normal ~30s restart window from reading as a 503 "wedge."

## The liveness signals to wire

1. **`poll` — the wedged signal (NEW; the F067 fix).** Add `src/safety/liveness.ts` holding `lastSuccessfulPollAt` (epoch ms, null until first success). `poll.ts` already calls `notePollSuccess()` at the top of the success branch of the loop (right after `consecutiveErrors = 0`) — add a one-line `recordPollSuccess(now)` there. Expose `getPollLiveness()` → `{ lastSuccessAt, ageSeconds, ok }` where `ok` = age within threshold. **Keys on the last SUCCESSFUL poll *cycle*, NOT the last DB write** — so quiet hours (no mail → no DB write, but polls still succeed) do not false-red. Pure threshold predicate is unit-testable. `poll.ts` is not in Step 57's set → low collision.
2. **`db`.** A cheap `select 1`-style reachability check with a short timeout, run per `/health` request (optionally memoized ~few-seconds so a monitor polling frequently doesn't hammer Supabase). Fails to `ok:false` on error/timeout, never throws.
3. **`slack_socket`.** Expose the Socket Mode connected state from `slack.ts` (the disconnect monitor + the `slackApp` connection already know it) via a read-only getter. Read-only; no behavior change to the socket path.
4. **`m1`.** `isBreakerTripped()` + `getBreakerState().reason` from `safety/breaker.ts` (both already exported, pure reads).
5. **`errors_24h` — dashboard-side, NOT in `/health`.** The Evryn Product tab already renders a 24h error count from the product DB (shipped 2026-06-16). Half B reuses that existing source for the 5th light; `/health` does not need to newly expose it. (Noting it here so the "5 lights" tally is explicit: `/health` provides db / slack_socket / poll / m1; the dashboard already has errors_24h.)

## Operational requirements (DC gates on these)

- **`/health` must NEVER throw and must be fast + non-blocking** — a monitor endpoint that hangs is worse than a wrong answer. Every check is wrapped, times out, and fails to a conservative state (`ok:false`), never propagates.
- **The DB check is cheap + short-timeout** (and may be briefly memoized) so frequent `/health` polling can't hammer Supabase or slow the endpoint.
- **No new import cycles.** `liveness.ts` is a leaf state-holder (like `breaker.ts`); `index.ts` composes the checks.
- **Backward-compat:** `status` + `uptime` stay; `checks` is additive. But the status CODE now varies — anything today treating a bare 200 as "alive" must handle 503. (This is the intended change; call it out at deploy.)

## Half B — dashboard (follow-on, after Half A ships)

The Evryn Product tab (`dashboard/api/product.ts`) already reads Railway `/health` for the liveness banner. Extend it to render **5 lights** from the new `checks`: **db / slack_socket / poll / m1**, plus the **errors_24h** light from the existing product-DB source. Aggregates-only / PII-firewalled (unchanged posture).

**⭐ The colour scheme (CORRECTED 2026-07-16 — this line previously said `m1.breaker_tripped` → an "amber HALTED", which CONTRADICTED the rationale block at the top of this doc ("the dashboard shows it RED"). QC caught the self-contradiction at the Half-B review; the build followed the RED rationale and is correct. Do NOT re-introduce amber for m1):**
- **RED = an actual fault** — including **`m1.breaker_tripped` → RED "HALTED" (+ the reason)**. A halt is a real state requiring a manual operator restart (no auto-reset — ADR-041), so red is the honest signal; amber would under-call it.
- **AMBER (`warn`) = an EXPECTED DOWNSTREAM CONSEQUENCE of another light's fault**, not an independent fault. The one live case: **the poll light when `m1.breaker_tripped && !poll.ok`** — a breaker trip calls `stopPolling()`, so poll staleness ~15min later is *caused by* the halt. It renders amber with a causality sub-label ("expected — M1 halted") so ONE halt reads as ONE fault, not two. **A genuine wedge (poll stale, breaker NOT tripped) MUST stay RED** — that's the real alarm and must never be softened. This mirrors `decideHealthStatus`'s halted-beats-wedged precedence at the presentation layer.
- **GREEN = healthy · GREY/idle = no data** (the graceful-degrade state when `checks` is absent — i.e. the pre-Half-A-deploy old `/health` shape).

## ⭐ The open ops question — RESOLVED by AC3 via config inspection (2026-07-15); OC/Justin confirm at deploy

**RESOLVED:** `evryn-backend` has **no Railway health-check configured** in the repo — no `railway.json`/`.toml` `healthcheckPath`, no `Dockerfile HEALTHCHECK`, no `restartPolicy`. So **nothing restarts the service on a `/health` non-200 today** (the sprint's "inert watchdog"). The dashboard reads `/health` for its banner; Healthchecks.io uses the *separate* in-loop `pingHealthcheck` (fired after a successful poll — so it ALREADY catches the wedged case independently, which makes the `/health` 503 *additive* operator-visibility + a Railway-*armable* watchdog, not redundant). **Consequence for the design:** the `503 wedged` vs `200 halted` split is forward-safe today (inert) and becomes *load-bearing the moment the watchdog is armed* — i.e. if arming Step 78 includes setting Railway's `healthcheckPath=/health` so Railway restarts on a 503, then a 503-on-*halt* + auto-restart would be exactly the M1 restart-loop the halted-→-200 rule prevents. **So the design holds; Half A can build now.** **Residual (flag at deploy, does NOT block the build):** a health check set in the Railway *dashboard UI* isn't visible from the CLI — Justin/OC confirm none exists and decide whether *arming* the watchdog = setting `healthcheckPath=/health`. Given this is resolved, a full OC spin isn't needed pre-build; the operator-action decision rides the deploy.

- **If Railway does NOT act on `/health` at runtime** (only Healthchecks.io + the dashboard read it): the 503 is purely informational, the loop risk is moot — but the `wedged` vs `halted` distinction still matters for the dashboard + for any Healthchecks integration.
