## 1. Why (plain)

Forwarded emails today are stored **only** in `emailmgr_items`, which auto-deletes rows after 6 months. So once a row ages out, the contact's original email is gone from our DB (it still lives in Mark's Gmail). That breaks the v0.3 reach-back plan (contact everyone Mark forwarded, while they still remember emailing him). This build writes a permanent copy — as a **contact↔Evryn** message — which (a) survives the TTL, (b) never enters *Mark's* history (Mark is neither sender nor recipient, so it's outside his `getRecentMessages` window — verified), and (c) is correctly in the *contact's* history so Evryn has it as real context when she later works with that contact.

## 2. The design (settled — from the ratified 1-pager §4/§5)

On `processForward` (`src/email/process.ts`), **in addition to** the existing `createEmailmgrItem`, write a `messages` row:

- `sender_id` = the reified contact (`originalFromUserId`, "Eva")
- `recipient_id` = `EVRYN_USER_ID`
- `source` = `"email"`
- `content_raw` = the forwarded body (same value passed to `emailmgr_items.content_raw`, i.e. `email.body`)
- `metadata` = `{ origin: "forward", intended_for: <gatekeeper user_id = sender.id>, emailmgr_item_id: <item.id> }`

**Write model (§5.1): eager, at forward-time** — record it the moment the forward arrives, so the never-conversed contacts the v0.3 reach-back targets are all preserved (lazy-backfill would miss exactly them). ~200 rows/day is trivial storage, and it's the contact's-own-history, not bloat.

**Failure posture:** the persistence is defense-in-depth for a *future* reach-back — it must **never break the live triage pipeline**. Mirror the codebase's best-effort audit-write pattern (`recordLlmUsage`, the `executeApproval` audit write): if the `messages` write fails, log + (optionally) `notifyDev`, but do NOT throw — the forward still triages. (Do not adopt the "throw → poll.ts holds cursor" posture here; that's for load-bearing ingest steps, and this is an additive preservation write.)

## 3. ⚠️ THE OPEN FORK (awaiting Justin — do not build past this line until resolved)

**§5.2 says "persist for every real person — gold/edge/pass/bad_actor — skip only `ignore`." But the verdict (gold/edge/pass/ignore) isn't known at forward-time** — triage runs *after* `processForward`. And `processForward` reifies the original sender for *any* parseable `From` (including spam), so anchoring purely on "a contact record exists" would also preserve a forward that Evryn later classifies `ignore`. The ratified 1-pager is internally inconsistent here (it assumed `ignore → no user record`, but the runtime reifies pre-triage).

**Resolution options (Justin picks):**
- **(A) [AC3 rec] Write at forward-time, delete on `ignore`.** Write the message eagerly (catches every real person, defense-in-depth); have the deterministic ignore path (`recordIgnore` in `classify.ts` — also the Haiku active-ignore path) *delete* the preserved message for that item (findable via `metadata.emailmgr_item_id`). Net: real people preserved, `ignore` skipped — exactly the stated intent, one clean delete hook.
- **(B) Simplest — write at forward-time, leave it.** Accept that a spam forward with a readable sender also leaves a preserved message. Marginal harm: the spam *lead record already exists* regardless of 54b; this only adds a message row, and the v0.3 reach-back can filter by the `ignore` classification already on the item. Cheapest.
- **(C) Verdict-keyed — persist only at the real-person verdicts.** Write in `recordPass` + the gold/edge draft-submission + the bad_actor path; never in `recordIgnore`. Conceptually cleanest (keys exactly on "real person"), but spreads the write across ~3 sites and **bad_actor has no deterministic recording home today**, so it'd need one built.

**Once Justin picks, the §2 write + the chosen ignore-handling become the final `<task>`.**

## 4. §5.2 backstop (secondary requirement, applies under A or C)

A forward whose `From` didn't parse (`originalFromUserId` null) but where Evryn IDs a *real* human during triage via `create_user` → that real person should also be preserved. This is the rare parse-hiccup edge. **Hook:** the backstop must be scoped to *forwards* (not every `create_user`, which also fires for operator-introduces-gatekeeper / cold opens). Flag to DC as a design point — if it's non-trivial, it can be a fast-follow to the common-case forward-time write. (Under option B, the null-FK case simply isn't preserved — accept as the parse-hiccup floor.)

## 5. Tests (QC will verify; DC writes)

Pure-unit-testable per the codebase pattern: a pure builder for the `messages` row shape (mirrors `buildLlmUsageRow` / `buildPassStamp`) so the row (sender=contact, recipient=Evryn, metadata shape) is unit-testable without a live DB; a test that the write is best-effort (a throwing db does NOT break `processForward`); and — under option A — a test that `recordIgnore` removes the preserved message.
