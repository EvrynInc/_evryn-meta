#### M-resilience-1 — [BLOCKER / high] Slack operator mode has NO sender-identity check — any human in #evryn-approvals gets full operator privileges
- **Location:** src/notify/slack.ts:298-348 (message handler) + 605-721 (handleOperatorMessage)
- **Category:** cross-user / security / spec-runtime-mismatch
- **What the code does:** The `slackApp.message` handler filters only `message.subtype` (drops bot + system messages), then routes EVERY remaining human message to handleApprovalMessage / handleNotesMessage / handleOperatorMessage. `slackUserId` (message.user) is captured but only ever written to metadata (line 624) and logged — it is never compared against an allowlist / configured operator ID anywhere in src/ (grep for SLACK_OPERATOR/authorized/allowedUser returns zero matches). handleOperatorMessage then composes the full operator dossier (operator.md + Operator profile + discipline_notice + any scoped user's full profile) and exposes set_thread_scope + rescope_messages.
- **Why it matters:** ARCHITECTURE.md (lines 123, 711, 861), ADR-014, and operator.md line 90 all state operator mode is a HARD security boundary gated by 'the Operator's verified Slack user ID' — 'the code decides before Evryn wakes up' and 'their identity was verified before this module loaded.' The runtime does not implement that gate. Anyone who can post in the channel the bot resolves by name (#evryn-approvals) — a second workspace member, a future-added teammate, a compromised account — can drive Evryn in full operator register: approve/reject outbound drafts (`approve <id4>`), read any user's full profile by opening a scoped thread, set/rescope user scope, and issue instructions Evryn treats as the business owner's. operator.md explicitly tells Evryn 'The Operator has full access because their identity was verified before this module loaded. That's not a trust decision you make.' — so Evryn will NOT re-check; the structural check the identity layer relies on does not exist. This is the single largest gap between the documented bulkhead architecture and the runtime.
- **Fix sketch:** Add a `SLACK_OPERATOR_USER_ID` (or set) to config; in the message handler, if `message.user` is not the configured operator, do not route to handleOperatorMessage/approval/notes — either ignore or notifyDev. The approval/notes paths especially must be gated. Until then, the only protection is Slack channel-membership hygiene, which is not a structural guarantee and contradicts the spec.
- **Verification (reviewer):** Read full slack.ts; grepped src/ for SLACK_OPERATOR|OPERATOR_SLACK|operatorSlackId|allowedUser|authorized (no matches) and for message.user usage (only metadata + logging + handler routing). Cross-checked against ARCHITECTURE.md ADR-014/Operator Security section and operator.md 'full access because verified before this module loaded.'
- **Sprint overlap:** novel

#### M-resilience-4 — [NON-BLOCKER / high] executeApproval audit write mislabels the outbound recipient as Evryn for forwards and Evryn-initiated drafts
- **Location:** src/approval/flow.ts:442-456 (executeApproval audit createMessage)
- **Category:** audit-trail / silent-failure
- **What the code does:** After a successful send, the best-effort audit write does `const recipient = await findUserByEmail(draft.draft_to); createMessage({ sender_id: EVRYN_USER_ID, recipient_id: recipient?.id ?? EVRYN_USER_ID, ... })`. For a triage notification the draft.draft_to is the GATEKEEPER, so this usually resolves correctly. But when `findUserByEmail` returns null (recipient email has no users row — e.g. an internal/review address, a contact whose record is keyed on a different address, or any miss), recipient_id falls back to EVRYN_USER_ID — recording an outbound message as Evryn→Evryn.
- **Why it matters:** The messages table is the single audit trail of all outbound (ARCHITECTURE.md 'All messages should be recorded'). A self-addressed audit row is a corrupted history record: getRecentMessages(realRecipient) won't surface it, so the conversation-history that later loads for that user silently omits what Evryn actually sent them, and any future cost/spend/relationship analysis attributes the send to Evryn. It fails silently — there is no notifyDev on the fallback branch (only on a thrown audit error). At v0.2 single-gatekeeper scale the gatekeeper's record exists so this rarely fires, but it is latent and gets worse at v0.3 (recipient ≠ gatekeeper, cold-opens).
- **Fix sketch:** If findUserByEmail returns null, don't silently self-address — either notifyDev so the gap is visible, or store the recipient email in metadata and leave recipient_id unresolved/flagged. The submitDraftForApproval path already derives the authoritative user from item.user_id; reuse that rather than re-looking-up by draft_to.
- **Verification (reviewer):** Read executeApproval lines 442-456 in full; cross-checked EVRYN_USER_ID constant (flow.ts:22) and findUserByEmail (db/users.ts:20-32, returns null on no-match). No notifyDev on the null-recipient branch.

#### M-resilience-5 — [NON-BLOCKER / medium] checkProactiveOutreach / checkFollowUps cron paths run a FULL operator-suffix dossier per user with no per-run dedup guard against double-fire
- **Location:** src/email/poll.ts:680-760 (checkProactiveOutreach) + 524-599 (checkFollowUps)
- **Category:** idempotency / race
- **What the code does:** checkProactiveOutreach gates on PT-hour == PROACTIVE_CHECK_HOUR_PT and a 23h per-user interval via users.last_proactive_check_at, bumped in a finally. checkFollowUps gates only on an in-memory `lastFollowUpCheck` (1h) and re-queries delivered items >7d each run, issuing a runEvrynQuery per delivered item with NO per-item dedup — a delivered item that Evryn doesn't transition (decides 'nothing to send', or her supabase_upsert to no_gk_response fails) stays `delivered`, so it is re-picked every hour and re-drafts a follow-up indefinitely until it happens to terminalize.
