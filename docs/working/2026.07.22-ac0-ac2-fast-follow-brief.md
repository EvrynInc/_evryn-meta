## §3 — THE WebFetch DESIGN QUESTION (Justin's — recommend-first, THINK don't rubber-stamp)

Justin pushed on the fix's fundamental shape; **his instinct may be a better design than the shipped flat whole-query cap.** Evaluate before you finalize WebFetch:

- **(a) Per-TURN / inter-message-silence watchdog instead of a flat whole-query cap.** A flat cap bites *legit* long multi-turn triages (Gate-A fixtures took ~1-2 min with real research; a deep multi-search isn't far from 3). A per-tool WebFetch timeout is *not* achievable (no SDK knob) — but a **per-turn watchdog IS:** at the `for await (message of query())` level, **reset the abort timer every time the loop yields a message; abort only if no message arrives for N minutes.** A hung tool blocks the yield → silence → caught; a long-but-*progressing* query keeps resetting → never falsely aborted. **Bonus: this largely dissolves NB-2** (a steadily-generating Reflection keeps yielding → won't abort → may make a separate Reflection budget unnecessary). v0.2 trade to name for him: a slow-but-alive 20-turn query occupies the single-threaded loop a while (no concurrency at v0.2) — acceptable at one-gatekeeper scale; the real answer at scale is concurrency, not a tighter cap.
- **(b) Don't just fail-and-stop — note, retry, and on a REPEAT of the same failure escalate/switch, not loop.** A deterministic hang (permanently dead domain) re-hangs every retry → an abort-retry burn loop (same ratchet shape as NB-2). Design the abort path to distinguish "transient → retry" from "deterministic → escalate to the operator."
- **NB-2 detail:** the flat cap contradicts `reflection.md`'s verbatim *"Soft length — never a hard cap … it never cuts it,"* AND ratchets (timeout → nothing written → notes grow → next attempt longer → burns an Opus call hourly forever). Justin RULED: give Reflection its own larger budget (per-call `timeoutMsOverride`; AC0 rec 600s starting) — **but resolve it TOGETHER with (a), which may dissolve it. Do NOT raise the *global* cap** (re-weakens the wedge protection). Note: `checkReflectionConsolidation` runs *inside* the poll-loop checker block, so a hung Reflection query wedges the loop too → Reflection needs *a* cap (large), not none.

**Your call as lane owner** — if you land on the flat-cap-plus-override as simpler-and-good-enough-for-v0.2 with the watchdog as a v0.3 refinement, that's legitimate; bounce your reasoning back to Justin recommend-first. He wants it *thought about*.
