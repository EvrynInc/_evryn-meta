## 0. The frame (why this exists)

Today a forwarded email is triaged the instant it arrives (real-time, per-forward). At Mark's scale (~200 forwards/day, an "onslaught" of auto-forwarded bulk), that means ~200 separate Opus triage runs a day, each paying its own cold cache write — the dominant cost. **Clustering holds forwards in a queue and drains them in twice-daily batches:** the batch runs the pile through triage back-to-back, so the first query warms the cache and the rest ride warm (the cost win), and the gatekeeper gets **one tidy package per batch** instead of pings all day. Direct conversation with Evryn stays real-time — clustering is **forwards-only**.

**Sequencing with the other two cost changes (composition — do NOT break it):**
- **Haiku Bouncer (①, shipped, currently OFF):** a cheap pre-screen already lives at the top of `runForwardTriage` (`maybeRunHaikuScreen`). Clustering drives each queued item **through that same `runForwardTriage`**, so the screen runs *inside* the batch when it's later flipped on. Clustering must keep calling `runForwardTriage` per item; it does not re-implement triage.
- **Runtime Bookkeeping (③, not started):** rewrites `runForwardTriage`'s *internals* later. Clustering treats `runForwardTriage` as a black box ("drive an item through triage") so ③ can change its guts without touching Clustering.

**Volume reality that shapes the design:** the onslaught is bulk auto-forwards (no personal note). The handful Mark *hand-forwards with a personal note* are a different, tiny-volume animal — good customer service says don't bury those in the queue. Hence Piece B below.

## 1. The three coupled pieces + build order

Clustering is **three coupled pieces** (they share `poll.ts`, `process.ts`, `detect.ts`). Build in this order; each is testable before the next:

- **Piece A — Queue + twice-daily batch** (the core cost change). Forwards land in a `queued` state instead of triaging on arrival; a batch drains the queue.
- **Piece B — Personal-note pathway.** A forward *with* a personal note routes the note to Evryn in real time (she's never blind to a forward Mark just told her about); the triage item still queues unless she pulls it forward.
- **Piece C — Reply-aware forward detection** (a correctness hardening that Clustering makes load-bearing). A message that's Mark *replying to Evryn* is never mis-routed to triage, even if it quotes a forwarded chain.

**Operational requirements (§5) and "what NOT to build" (§6) are part of the spec — read them before you start.**

## 2. Piece A — Queue + twice-daily batch

### A1. New lifecycle status: `queued`

Add a new `emailmgr_items.status` value **`queued`** = "a forward captured, waiting for its batch — not yet triaged."

**Why a new status and not reuse `new`:** `new` is overloaded — it is *also* the crash-recovery park state (M1's lossless halt parks an in-flight item `processing → "new"`, and the poll loop's dedup-resume rule (`process.ts`, `findItemByExternalId` → `resumeParkedItem`) treats *any* re-fetched `new` item as "resume triage now"). If a fresh forward sat at `new` awaiting its batch, a cursor re-fetch (after any transient hold) would trip that resume rule and triage it immediately — breaking clustering. A distinct `queued` keeps "waiting for batch" and "recovering from a halt" from colliding. `queued` is a *status* (workflow stage), NOT a verdict — Evryn hasn't judged it yet.

**Where the new status must land (all of these — a missed one is a silent break):**
1. `src/db/items.ts` — add `"queued"` to the `ItemStatus` union.
2. **Migration** (`backups/add-queued-status.sql`, dev-first per ADR-037, RLS unchanged — this only widens a CHECK constraint): the `emailmgr_items.status` CHECK constraint must accept `queued`. Take pre/post `pg_dump`. **AC2 applies to dev + prod** — you write the migration; you do not apply it.
3. `src/triage/classify.ts` — the `supabase_upsert` tool description's status enum lists the valid statuses; add `queued` (Evryn should not normally set it, but the enum must be complete). Do NOT add a new tool.
4. **Flag LOUDLY in your output** that this touches the ADR-018 lifecycle (`ARCHITECTURE.md` emailmgr_items Status Lifecycle) — **AC owns the ARCH edit** (Justin reviews it in SCM). You do not edit `ARCHITECTURE.md`.

### A2. Ingest change: forwards queue, they don't triage

Today: `processEmail` → (registered-lane forward) → `processForward` → **creates the item (default `new`) then immediately calls `runForwardTriage`** (real-time triage).

New behavior, **gated by `CLUSTERING_MODE`** (§A4):
- **`CLUSTERING_MODE=off`** (default — ships dormant): unchanged. `processForward` triages in real time exactly as today. (This is what lets us deploy Clustering dormant and flip it on before Mark — mirrors the Bouncer's ships-off discipline.)
- **`CLUSTERING_MODE=manual` or `scheduled`:** `processForward` creates the item with status **`queued`** and **returns without calling `runForwardTriage`.** The forward is fully captured — deduped (`external_id`), original-sender reified (ADR-036), `original_subject` stashed, cursor-tracked — nothing is triaged. It waits for the batch.

Direct messages (`processDirect` / `runDirectTriage`) are **untouched in all modes** — real-time, forwards-only clustering.

### A3. The batch drain

A batch drains **every `queued` forward item**, oldest-first, and drives each through the existing triage. Concretely:

- **A new cron checker** in the poll loop (`poll.ts`), alongside `checkProactiveOutreach` / `checkMorningSweep` — call it `checkClusterDrain` (or similar). It follows the same shape: a per-cycle guard, its own try/catch with `alertCheckerFailure`, gated behind `CLUSTERING_MODE`.
- **It fires when:** `CLUSTERING_MODE=scheduled` AND the current PT hour matches a configured batch hour (twice-daily, default **~10am + ~4:30pm PT** — see §A4); OR it is triggered manually (§A5). Reuse the existing `getPacificHour` helper + a once-per-batch-window guard (mirror `shouldRunProactiveCheck` / the morning-sweep once-per-period guard).
- **It drains** via `findItemsByStatus("queued")` (already exists in `db/items.ts`), oldest-first.
- **For each queued item, drive it through the existing forward triage.** The batch works from the **stored item**, not a live Gmail fetch, so it must reconstruct the triage inputs from the item: `sender = findUserById(item.user_id)` (the gatekeeper); reconstruct `forwardInfo` by re-running `detectForward(item.subject, item.content_raw)` (the item stored the raw forwarded body as `content_raw` and the subject); build the minimal email-shape `runForwardTriage` needs (`subject`, `body`) from the item. Then call the existing `runForwardTriage(...)` on the **existing** item (do NOT create a new item). **This is the key implementation point — extract a helper (e.g. `triageQueuedItem(item)`) that reconstructs and calls `runForwardTriage`.** `runForwardTriage` already sets the item `processing`, runs the screen (Bouncer) + Opus, and transitions it terminal — so a drained item leaves `queued` on its own; the next batch won't re-pick it (status guard).
- **Sequential** (per the cost analysis: ~10 survivors/day clear in ~5–10 min; sequential keeps the cache warm and avoids rate-limit pressure from parallel multi-hundred-K-token loops). Do NOT parallelize.
- **Per-item isolation:** one item's triage failure must NOT abort the batch — per-item try/catch (mirror `checkFollowUps`'s loop). A failed item stays where `runForwardTriage`'s own error handling leaves it (`error` status) and is caught by the existing stale-checkers.
- **M1 interaction (load-bearing):** `runForwardTriage` already returns `"halted"` on a breaker trip (the in-flight item was parked by `runEvrynQuery`). In the batch loop, on a `"halted"` outcome **stop the batch** (the breaker tripped — no point continuing to spend). But **the in-flight item must land back in `queued`, not `new`** — see §A6.

### A4. Mode + cadence config

Add to `config.ts` (mirror the Bouncer's `parseHaikuScreenMode`):
- **`CLUSTERING_MODE`** = `off` | `manual` | `scheduled`, **default `off`** (unset or unrecognized → `off`, the safe no-op). `off` = current real-time triage. `manual` = queue + drain only on the operator command (§A5). `scheduled` = queue + drain at the batch hours (+ manual still works).
- **`CLUSTER_HOURS_PT`** = comma-separated PT hours, default `"10,16"` (~10am + ~4:30pm → use whole hours 10 and 16; if you want the :30, gate on hour AND a minute check — but whole-hour is fine and simpler; **confirm with AC2 if you think the half-hour matters**). Env-overridable, no code change to re-tune.

### A5. Manual trigger (operator command)

An operator Slack command **`cluster now`** (or similar) hand-fires a batch drain immediately, regardless of the hour, when `CLUSTERING_MODE` ∈ {`manual`, `scheduled`}. **Operator-gated** — only the configured `SLACK_OPERATOR_USER_ID` may trigger it (reuse the existing operator-auth gate; see Lane-B Step 62 / `isAuthorizedOperator`). This is how Justin watches the first few batches by hand before flipping to `scheduled`. The command posts back a summary (§A7).

### A6. Restart catch-up = the PRIMARY recovery (Justin, 2026-07-06 — do not get this backwards)

An interrupted batch (breaker halt, crash, deploy) must **not** wait ~6h for the next scheduled batch to recover. On process start (`index.ts` / wherever startup recovery runs), if `CLUSTERING_MODE` ∈ {`manual`, `scheduled`}, **run a catch-up drain of any `queued` items immediately** (subject to the breaker being clear — if it's tripped, the existing M1 flow governs). The scheduled cron is the **extra** failsafe, not the main one.

- **Interrupted batch = only the *remaining* items are affected.** Items already drafted are safe in the approval queue; the one in-flight at the halt was parked by `runEvrynQuery`. **That park must target `queued`, not `new`,** for a clustered item — otherwise the parked item either (a) never re-drains (nothing drains `new` except the poll-resume, which won't re-fetch a batch-drained item) or (b) trips the poll-resume rule. **Implementation:** `parkInFlightItemForHalt` (in `classify.ts`) currently parks `processing → "new"`. For a clustered drain it must park `processing → "queued"` so the catch-up/next batch re-drains it. Decide the cleanest seam (e.g. the park target depends on whether the item came from the batch vs. the real-time poll) — **flag your chosen seam in your output for AC2 review**, since it touches the M1 lossless-halt invariant (ADR-041).

### A7. The positive "batch ran" heartbeat (M1 — you build it)

Clustering decouples ingest from triage, so the **existing post-ingest Healthchecks heartbeat stays GREEN even if a batch silently fails and forwards pile up untriaged.** So a batch must emit a **positive** signal every run: log + a summary of the form **`cluster ran at HH:MM PT — drained N, screened S, escalated E, drafted K, errors X`**. Wire it into the existing safety layer (`src/safety/*`) — e.g. a cluster-specific Healthchecks affirmation (mirror `checkDailyAffirmation` / `pingHealthcheck`), or at minimum a `notifyDev` summary per batch. A batch that finds **zero** `queued` items should still emit a light "cluster ran, nothing queued" heartbeat (so silence = something's wrong, not "empty queue"). **The ARCH-doc absorption describing this heartbeat is AC's (Justin SCM-reviews it) — you build the code + flag it.**

## 3. Piece B — Personal-note pathway

**Goal (Justin, 2026-07-06):** a forward Mark hand-sends **with a personal note** must reach Evryn in real time — she should never be blind to a forward he just told her he's sending. The forwarded email still becomes a `queued` triage item; Evryn judges from the note whether to pull it forward now (urgent) or acknowledge and let it ride in the queue.

**Detection:** a forward has a personal note when `forwardInfo.forwardingNote` (already extracted by `detect.ts` `splitForwardBody` — the gatekeeper's own words before the forward separator) is non-null and non-trivial (guard against whitespace / a bare separator artifact).

**Behavior (in `CLUSTERING_MODE` ∈ {`manual`,`scheduled`}):**
1. Create the `queued` triage item as normal (Piece A).
2. **Also** route the note to Evryn as a **real-time conversational turn** — a `runEvrynQuery` on the operator/user pathway (this is Mark talking to Evryn), whose prompt carries: the personal note, and a pointer to the linked `queued` item (its `id` + the forwarded content, so she can act on it inline). She decides:
   - **urgent** → triage the queued item **now** (drive it through `runForwardTriage` / the same `triageQueuedItem` helper), moving it out of `queued`; then respond to Mark as warranted.
   - **not urgent** → she leaves it queued; she **always considers replying** to Mark (good customer service — the voice/judgment is Mira's, §8).
3. **Bare forward (no note)** → Piece A only (straight to the queue, no real-time turn).

**The exact "pull-forward" mechanism is yours to design** (e.g. hand her the item + note in one turn and let her triage inline if urgent, vs. a small "triage this queued item now" affordance). **Surface your chosen mechanism in your output for AC2.** Constraints: it must NOT double-triage (a status guard — an item triaged in the note turn is no longer `queued`, so the batch skips it); it must reuse the existing triage path (no parallel triage implementation); the note content is **untrusted** and goes in `prompt`, never `systemPrompt` (fence it like the forward body — `fenceUntrusted`).

**The voice half is Mira's.** How Evryn weighs urgent-vs-acknowledge, and the "always consider replying, guided by good customer service" beat, live in an identity file — AC2 writes Mira a brief; you build the mechanics + **flag LOUDLY any identity-file coupling** (per your CLAUDE.md — identity files are runtime; never touch them silently).

## 4. Piece C — Reply-aware forward detection

**The hole (fired twice in Phase 6; Step 85 only partially closed it):** `detectForward` (`detect.ts`) classifies a message as a forward if the subject is `Fwd:`/`Fw:` **OR** the body has a strict forward-separator line. When Mark **replies to Evryn about a triage item**, his subject is `Re: …` (doesn't trip the subject test) but his reply usually **quotes the original forwarded email, separator and all** — which trips the body test → his reply is mistaken for a *new* forward → a **phantom duplicate triage item.** His reply has a fresh Message-ID, so `external_id` dedup does NOT catch it. Under Clustering the phantom lands silently in the queue (worse than surfacing in real time).

**The fix:** a message that is clearly Mark **replying to Evryn** must never be classified as a forward, even if it quotes a forwarded chain. Concretely, `detectForward` becomes:
- **Forward** iff: `Fwd:`/`Fw:` subject (the affirmative signal) **OR** (a strict body separator **AND NOT** reply-indicated).
- **reply-indicated** =
  1. subject matches `^\s*Re:` (cheap, deterministic — the common case), **OR**
  2. the inbound is part of an existing Evryn↔sender thread (its `thread_id` matches a thread where Evryn has already conversed with this sender — a `messages`-table lookup). Include this belt-and-suspenders layer (Justin: "be extra sure"), but the `Re:` check is the primary, zero-lookup guard.

This preserves genuine no-`Fwd:`-subject forwards (some auto-forwarders) — they still trip via the strict separator when NOT a reply — while excluding replies-that-quote-a-forward. Keep it a **pure, unit-tested** predicate (mirror `hasStrictForwardSeparator`). **This also protects the slip-through case Justin raised:** even if a phantom ever reached triage, the existing `original_from_user_id` prior-history + `external_id` dedup are the deterministic second catch — but Piece C stops it at the door, cheaply.

## 5. Operational requirements (gate on these — AC2 will ask)

- **Retry/failure:** per-item try/catch in the batch loop (one item's failure never aborts the batch). The batch cron's own try/catch routes through the existing `alertCheckerFailure` rate-limited gate (mirror the other cron checkers).
- **Singleton / no double-fire:** the batch drain must fire once per window — reuse the once-per-period guard pattern (`lastRun` timestamp / PT-date guard) the proactive cron + morning sweep use. Single-replica is confirmed (1 Railway instance).
- **Idempotency:** only `queued` items are drained; a drained item leaves `queued`, so a re-run is a clean no-op. The manual command and a scheduled window landing in the same minute must not double-drain (the once-per-window guard + the status guard together cover this).
- **Shutdown / restart:** a batch interrupted at shutdown leaves its remaining items `queued`; the restart catch-up (§A6) re-drains them. Lossless.
- **M1 lossless halt preserved:** a breaker trip mid-batch parks the in-flight item to `queued` (§A6) and stops the batch; one emergency alert, no error-storm, no dropped forward. Do not regress ADR-041's invariants.
- **Operator-gating:** the manual `cluster now` command is gated to `SLACK_OPERATOR_USER_ID` (reuse `isAuthorizedOperator`).

## 6. What NOT to build

- **No separate "urgent keyword / urgent Haiku" detector.** Urgency is handled by Piece B (Evryn's judgment on the personal note), not a keyword scan. A bare forward always queues.
- **No sub-cluster / real-time path for bare forwards.** Bare forwards always queue; the twice-daily package IS the product.
- **No verdict-change (edge→gold) tracking.** Deferred (see the ARCHITECTURE.md audit note, 2026-07-07). Not this build.
- **Do NOT re-implement triage.** The batch and the note-pathway both drive items through the existing `runForwardTriage`. This is what keeps Clustering composed with the Bouncer (①) and Runtime Bookkeeping (③).
- **Do NOT edit `ARCHITECTURE.md` or identity files.** Flag both loudly instead (AC owns the ARCH edit; Mira owns the identity voice via an AC2 brief).
- **Do NOT `git add -A`, touch `main`, or deploy.** You work in your own worktree/branch; AC2 gates the merge; deploy is Justin's.
