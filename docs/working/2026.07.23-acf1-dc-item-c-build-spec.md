## 1. What item (c) is ‚Äî five changes, one build

Item (c) makes the whole thread the agent's working context (no more 30-message window), gives the agent a passive "your load is heavy" nudge with a hard safety backstop, and serializes each agent to one wake at a time ‚Äî with a **priority-ordered queue** when several wakes are pending. Plus one bundled bug-fix (L5) and one comment fix.

The five (design-of-record ¬ß4.3 + ¬ß4.6):

- **(c1) Full-thread-load** ‚Äî remove the 30-msg/8K window for the current-thread layer (layer 10).
- **(c2) L5 fix** ‚Äî layer 10 currently mislabels a non-empty-*capable* thread as "no thread" when it happens to be empty; surface the thread id unconditionally.
- **(c3) Composed-load advisory** ‚Äî generalize the current-state size advisory into a whole-load advisory that reads the wake's real per-layer sizes, names which layer is heavy, and offers a burn-down menu. Passive: no wake, no DB write, no forced action.
- **(c4) Hard safety ceiling** ‚Äî a backstop that head+tail-windows a single thread ONLY if it exceeds a large token ceiling, purely to keep the wake runnable. The advisory (c3) should make this practically unreachable.
- **(c5) Per-agent wake serialization + priority-ordered queue** ‚Äî one wake at a time per agent; when several are pending, pick by priority (Justin > mains > workers), oldest-first within a tier; defer the rest **unconsumed**.
- **(c6) Comment fix** ‚Äî `db/types.ts` header says "migrations 001‚Äì015"; it's 016 now.

## 3. (c2) L5 fix ‚Äî surface the thread id unconditionally; empty ‚âÝ absent

**Where:** `src/composer/layers.ts`, the layer-10 render branch. **Finding L5** (lane-state ledger): today it renders
`inputs.thread.length > 0 ? '## This thread (<id>)‚Ä¶' : '## This thread\n(no thread ‚Äî this wake is not thread-scoped)'`.
That branches on **whether the thread has messages**, so a wakeup / check-in / any thread that is genuinely thread-scoped but *currently empty* (non-null `wake.threadId`, zero messages yet) is told **"no thread ‚Äî this wake is not thread-scoped"** ‚Äî false to its face (the runtime locks on that thread, mirrors the reply there). The thread id also reaches the agent ONLY in this header, so the false "no thread" hides it.

**Build:** branch on **`inputs.wake.threadId` presence**, not on `inputs.thread.length`:
- `wake.threadId` present, thread has messages ‚Üí `## This thread (<threadId>)\n<full thread>` (the c1 render).
- `wake.threadId` present, thread empty ‚Üí `## This thread (<threadId>)\n(no messages in this thread yet)` ‚Äî **name the id; do NOT say "not thread-scoped."**
- `wake.threadId` null ‚Üí `## This thread\n(no thread ‚Äî this wake is not thread-scoped)` (the ONLY case that keeps today's wording).

**Test:** the three cases above, asserting the empty-but-scoped case renders the id and does NOT contain "not thread-scoped". Break it (revert to length-branching) ‚Üí confirm RED.

## 4. (c3) Composed-load advisory ‚Äî generalize `currentStateSizeAdvisory`

**Where:** `src/composer/layers.ts` (`currentStateSizeAdvisory` at ~L135) + `src/composer/index.ts` (`composeWake`, where it's computed and threaded through `WakeContext`).

**Today:** `currentStateSizeAdvisory(chars)` returns a gentle line only about current-state size; `composeWake` computes it from `currentState.content.length` BEFORE `composeLayers`, threads it via `WakeContext.currentStateAdvisory`, and `composeLayers` folds it into layer 12 (wake context). It is passive: no wake, no DB write.

**‚öÝ The compute-location wrinkle ‚Äî decided, build it this way:** the design says the advisory should read "the manifest totals." But the manifest components are built *inside* `composeLayers`, and the advisory folds into layer 12 which is also built there ‚Äî so the full totals aren't known at that point (layer 13 scratchpad composes after). **Resolution (mirrors the existing pattern): compute the advisory in `composeWake`, BEFORE `composeLayers`, from the per-layer content lengths `composeWake` already holds in hand** (`teamCommon`, `hub`, `currentState`, `spokes`, `identityContent`, `memoryContent`, the rendered `thread`, `tray`, `scratchpad`, `digestEntries`, `requiredSkills`, skill index). Sum them into a total + per-component breakdown, build the advisory line, thread it through `WakeContext` exactly as `currentStateAdvisory` is today. Do NOT put an LLM call or a second composition pass in the hot path.
- You may rename `WakeContext.currentStateAdvisory` ‚Üí `composedLoadAdvisory` (or keep the field and repurpose) ‚Äî your call; keep it a nullable passive string folded into layer 12. Keep the existing current-state-specific behavior *subsumed* by the generalized one (don't lose the "current-state is big ‚Üí suggest a standup" case).

**What the advisory says (design-of-record ¬ß4.3, the concrete menu):** above threshold, name the total (~Nk tokens) + which layers are heavy (memory ~Ak, current-state ~Bk, this thread ~Ck, spokes ~Dk) + the burn-down menu ‚Äî a **standup** (shrinks current-state), a **consolidation** (shrinks memory), a **fresh thread** (caps this one), your call, no action required. **The diagnosis must name the heavy layer so the agent picks the matching lever** (heavy memory ‚Üí consolidate; heavy current-state ‚Üí standup; heavy thread ‚Üí fresh thread).

**Thresholds (tune-able consts, comment them):** per-component AND aggregate. Suggested: memory > 20K chars ‚Üí name consolidation specifically (matches the protocol's soft trigger); aggregate composed load > ~120K tokens (~480K chars) ‚Üí name the full menu. Keep chars‚Üítokens at ~4 chars/token (the file's existing heuristic).

**Invariants to preserve:** no wake, no DB write, no forced action; it's a line the agent sees and may ignore. Cache-safe (it's in layer 12, the volatile tail ‚Äî unchanged).

**Test:** the pure advisory builder ‚Äî under-threshold ‚Üí null; over on memory ‚Üí line names consolidation; over on aggregate ‚Üí line names the menu + the heavy layer. Break a threshold comparison ‚Üí confirm RED.

**Read design-of-record ¬ß4.6 ("One wake at a time per agent") + the A2 answer in the lane brief (the ACf‚ÜîACf1 exchange) IN FULL before building this.** The A2 design is SETTLED ‚Äî do not re-derive it.

### 6a. The two-keys principle ‚Äî do NOT conflate them
There are two *separate* questions, and the whole point of the A2 fix is that they must **not** share a key:
- **Coalescing** ‚Äî "merge duplicate triggers for the same *conversation*." Stays keyed on **(agent, thread)** ‚Äî UNCHANGED. `planDispatches` already coalesces same-(agent,thread) triggers into one decision; keep that exactly.
- **Serialization** ‚Äî "one wake at a time per *agent*." Becomes keyed on **agent** only.

Today both use `lockKey(agent, threadId)` (per-(agent,thread)), which is why two wakes of one agent in different threads run in parallel. Serialization changes the *serialization* key to per-agent while leaving *coalescing* per-(agent,thread).

### 6b. The second pass in `planDispatches` (dedup.ts)
After coalescing (unchanged), add a **second pass** that, for each agent, **selects ONE decision and defers the rest**:
- **Deferred = UNCONSUMED.** A deferred decision is simply NOT returned in the plan ‚Äî its triggers stay pending (`agent_messages.processed_at` null / `wakeups.fired` false), so the next cycle re-evaluates and retries them. **A deferred-but-marked candidate is silent loss** ‚Äî never mark a deferred trigger processed/fired.
- **The pick order (priority ‚Äî folded per Justin 2026-07-23; ¬ß6c below) with an oldest-first tiebreak.**
- **The per-agent in-flight guard still applies first:** if the agent is already in-flight (per-agent inFlight, ¬ß6d), *none* of its decisions dispatch this cycle (they all stay pending).

**Rewrite the now-false comments in dedup.ts** in the SAME commit ‚Äî e.g. the header line *"Two due wakeups for one agent therefore run in parallel legally"* becomes false the moment this ships. Rewrite it to state the new truth (serialized: one per agent per cycle, rest deferred). We spent this lane fixing docs that lied about the runtime; do not add a fresh one.

### 6c. The priority ordering (the queue-order Justin asked for)
When an agent has multiple coalesced decisions competing for its one wake this cycle, pick by **priority tier of the sender**, then **oldest-first within the tier** (anti-starvation):
- **p1 ‚Äî Justin.** `from_actor === 'justin'` (case-insensitive).
- **p2 ‚Äî a main.** `from_actor` in `ACTIVE_AGENTS` (an agent name). **A self-scheduled wakeup counts as p2** (a message from yourself-to-yourself; deferring one just costs it a cycle).
- **p3 ‚Äî a worker.** Anything else non-system (Phase-4 workers; none exist today).
- **A coalesced decision's priority = the HIGHEST priority among its coalesced triggers.** (If Justin posted in a thread that also carries a main's messages, that thread-decision is p1.) You'll need to carry the sender(s) onto the candidate/decision so the pass can compute this ‚Äî a `fromActor` (and/or the min-priority) field on `CandidateTrigger`/`DispatchDecision`. Today `addressedMessageCandidate` has `from_actor` at the call site (`scheduler/index.ts` passes `m.from_actor`) but drops it; thread it through. Wakeups have no external sender ‚Üí p2.
- **Tiebreak within a tier: oldest trigger first** (earliest `created_at` / `fire_at`). This bounds starvation: a steady p1 stream can only delay a p3 item while p1s keep arriving ‚Äî at our cadence (queue drains every 15s, wakes finish in seconds) it drains as soon as the higher tier clears. Note this property in a comment; true starvation needs *continuous* p1 saturation, which won't occur at our scale ‚Äî age-escalation is a documented future lever if it ever does.

**Priority is a pure, unit-testable comparator.** Keep it a small exported function (e.g. `senderPriority(fromActor)` + a decision-priority reducer) so tests can assert the ordering without a live DB.

**‚öÝ Honesty note for the handoff (carry it, don't hide it):** priority ordering is **unit-tested, not live-proven at divergent-sender scale** ‚Äî today `ACTIVE_AGENTS = ['lucas']`, no workers exist, and real inter-agent addressed traffic arrives with item (d). So the *placement* (a principled pick vs. arbitrary order) is what we're locking in now ("build for one, structure for many"); its multi-sender behavior is exercised for real when item (d) lands. Do not report it as live-proven.

### 6d. The in-flight lock ‚Üí per-agent (scheduler/index.ts)

- `planDispatches`'s `inFlight` check must align ‚Äî it now asks "is this agent in-flight?" (per-agent), not per-(agent,thread). Update `lockKey` usage / the `inFlight.has(...)` check in the planner to per-agent. (Keep `lockKey` if it's still useful, but its meaning/signature changes ‚Äî update its doc comment.)
- **KEEP the persisted `agent_state.singleton_lock.threads` map shape** (`{threads: {<thread_id>: {wake_id, started_at}}}`) ‚Äî it now holds **at most one entry per agent**. Add a why-comment explaining it's still a map holding one entry deliberately (so the next reader doesn't "clean it up" and break recovery). `boot/recovery.ts` iterates that map to park stuck wakes on restart ‚Äî leave its shape untouched; add a one-line why-comment there too. This keeps recovery + the (absent) migration untouched ‚Äî A2 rider (b).
