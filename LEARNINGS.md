**System Actors Are FK Anchors and Senders, Never *Implicit* Subjects.** Evryn and the Operator have user records so the messages table has valid `sender_id` / `recipient_id` values when Evryn sends outbound or the Operator messages Evryn on Slack ŌĆö they're legitimately senders and recipients in those flows. What they should *never* be is the *implicit* subject of a user-scoped operation ŌĆö operations that assume "the queried user is a person Evryn might want to know more about" applied to a system actor without a deliberate decision to do so. Examples of implicit subject-ification: profile loading via `buildPersonContext` from a sender-resolution path, history loading via `getRecentMessages(systemActorId)` (which returns every conversation in the system because system actors are involved in all of them), proactive outreach enrollment by status filter that fails to exclude system roles. These all happen when code assumes "this is a regular user" without checking. The Slack-to-Evryn flow already follows this correctly: when the Operator messages Evryn, the handler stores the message with `sender_id = OPERATOR_USER_ID` (correct), but composes the prompt from a fixed Operator context string and *doesn't* call `getRecentMessages` ŌĆö system actor is sender, never *implicit* subject. The 2026-04-27 loop bug was the inbound email path violating this principle: Evryn's own polled-back outbound resolved to her own user record, and the pipeline subject-ified it (built person-context, queried history). The fix is at the gate (don't let a system actor resolve as inbound sender), not at history loading downstream. The originally-proposed "skip history loading when sender is system actor" was wrong altitude ŌĆö defense-in-depth at the wrong layer adds runtime branching for a state that shouldn't arise. **Deliberate exceptions are different.** A *deliberate* design that subject-ifies a system actor with explicit pathway-gating ŌĆö e.g., ADR-030 loading Operator's `profile_jsonb` in the Slack-Operator pathway only, because Evryn is partnering with the Operator there and partnership-knowledge is the relevant context ŌĆö is a designed exception, not a violation. The principle bars *accidental implicit* subject-ification; it does not bar deliberate designs that name what they're doing and gate where they apply. *Promote to:* evryn-backend ARCHITECTURE.md (Data Model ŌĆö System Actors section) pending Justin's approval. Principle worth stating durably.

**Silent-Diversion via Word-Match: Parsers Must Yield, Not Capture, on Ambiguity.** 2026-05-29 evening: Justin sent Evryn a substantive Slack question containing the word "approve" (in quotes, inside a sentence ŌĆö *"if I send an 'approve' on that..."*). **Verify No-Information-Loss Doc Restructures by Diffing Extracted Specifics ŌĆö Don't Trust a Read-Through.** 2026-06-05: collapsing the evryn-backend SPRINT doc (597ŌåÆ290 lines ŌĆö merging a summary table + verbose day-by-day prose into one status-prefixed record) under a hard *no information loss* constraint from Justin. The problem: at that scale, a visual read-through is unreliable ŌĆö you can't reliably eyeball whether every fact survived. **The technique that worked:** programmatically extract every *high-signal specific* (7-char commit hashes, `ADR-NNN` / `EVR-NNN` refs, file paths) from BOTH the pre-edit version (`git show <old-commit>:<file>`) and the post-edit version, then **set-diff old-against-new** so anything present in OLD but missing from NEW surfaces ŌĆö e.g. `comm -23 <(git show OLD:f | grep -oE 'ADR-[0-9]+' | sort -u) <(grep -oE 'ADR-[0-9]+' f | sort -u)`. It caught a **real drop** the read-through had missed (the M1 `#emergency-alerts` implementation spec), which I then re-added. Every remaining old-only token was then triaged as either an intentional **de-dup** (the fact lives in a home doc ŌĆö fine) or a genuine drop (re-add). **Lesson:** for any large doc restructure carrying a no-loss requirement, run an extracted-specifics set-diff as a backstop ŌĆö it catches dropped facts visual review misses, and it forces you to consciously classify every delta as de-dup vs. loss. *Promote to:* AGENT_PATTERNS.md (a "verify large doc restructures programmatically, don't eyeball them" working-pattern) and/or AC CLAUDE.md (Documentation Approach). #sweep to evaluate.

**Use GitHub's Native Branch Rename for Default-Branch Renames ŌĆö Not CLI `branch -m` + `push --delete`.** 2026-06-08 (AC3, evryn-backend `master`ŌåÆ`main`): renaming a *default* branch is not the same as renaming a feature branch. GitHub's native rename (`gh api -X POST repos/{owner}/{repo}/branches/{old}/rename -f new_name={new}`, or repo ŌåÆ Branches ŌåÆ rename in the UI) does four things atomically that the raw-CLI path (`git branch -m` + `git push origin --delete master`) does not: (1) **retargets every open PR** to the new branch (a raw remote-master delete with an open PR against it is messy/destructive), (2) **sets up redirects** so old `master` links/clones still resolve, (3) **moves the default branch** pointer, and (4) **migrates branch-protection rules**. After the server-side rename, each local clone/worktree just syncs: `git branch -m master main && git fetch origin --prune && git branch -u origin/main main && git remote set-head origin -a`. **Pre-rename recon that mattered:** confirm no open PRs target the branch, no CI keys off the branch name (`.github/workflows`), deploy isn't branch-bound (Railway `railway up` ships working-dir contents ŌåÆ branch-agnostic ŌåÆ rename is deploy-neutral), and no branch protection to migrate. **Cross-instance gate:** in a shared repo, the *open-PR / in-flight-merge* state is exactly what a single instance's recon cannot see ŌĆö hold for the orchestrator (AC0) to confirm all merges landed before executing, rather than self-authorizing off a clean-looking branch tip. *Promote to:* AC CLAUDE.md (Worktree & Branch Discipline ŌĆö a "default-branch rename = native rename" bullet) and/or a short ops how-to. #sweep to evaluate. Full record: `docs/sessions/historical/2026.06/2026-06-08-ac3-master-to-main-recon.md`.
